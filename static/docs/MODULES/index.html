    Module & API Reference¶

    A per-module reference for the browser-side compute engine in app/src/lib/compute/* and the JSON data loaders in app/src/lib/data/*. Each section lists the module's responsibility, its key exported signatures with input/output shapes, the numerical thresholds and unit conventions that matter, and where it is called.

    This page is a navigable reference, not a tutorial. For the architecture overview, the per-bloc file contract, and the estimation pipeline see:

    • Architecture for AI — end-to-end data flow and the 8-file bloc contract
    • Adding a country and Re-estimating models
    • Fiscal block methodology — the budget-identity recursion in detail
    • Data contracts & JSON schemas — the 8 per-bloc JSON files, including the spec.json prior / transformation / isFinancial tags this engine reads
    • Pipeline — the MATLAB GVAR-EM estimator and Python data builders that produce the artifacts the modules below consume

    The two estimation-side facts that the browser engine inherits and must be read alongside the module reference:

    • Per-variable prior tag. Each spec.json variable carries prior ∈ {"RW","WN"}. RW (random walk) is the default own-first-lag prior mean of 1; WN (white-noise / stationary) sets the own-first-lag prior mean to 0 so the series mean-reverts. This is a property of the MATLAB estimator (bvarGLPmf.m, §Fiscal stationary prior), not the TS engine, but it changes the shape of the baselines the engine loads — the fiscal GDP-ratios (FREV_GDP, FPEXP_GDP, SFA) now revert to their in-sample mean instead of drifting at the long horizon. See DATA_SCHEMA §3 for the field and PIPELINE §5 for how it is set.
    • isFinancial and COVID masking. A variable with isFinancial:true is kept observed through the 2020Q2–Q4 COVID-as-missing window (only macro columns + the two macro stars are masked). The global commons GLB_USEQ (US equity) and GLB_OIL (oil) are isFinancial:true everywhere, so they condition the imputation rather than being imputed.

    All matrices in this engine are flat row-major Float64Arrays wrapped as { data, rows, cols } (see matrix.ts). All BVAR data is in companion space: 100·log levels for log variables, the raw value/rate for lin variables. A quarter-on-quarter annualized growth rate is 4·Δ(level) for a log variable.

    The compute modules are pure functions over the loaded JSON. The two sections below document the estimation-side mechanisms (MATLAB / pipeline/*) that determine the content of the JSON the engine consumes; they belong here because a reader debugging a baseline shape needs them, and because the prior / isFinancial tags are read by gvarSolve.ts and the loaders.

    Fiscal stationary prior¶

    Where it lives. BVAR/MFBVAR/GLPreplicationWeb/bvarGLPmf.m (the GLP/Minnesota prior builder), threaded through BVAR/MFBVAR/cgl_bvarV5.m (the COVID-as-missing data-augmentation Gibbs), driven by the two estimators pipeline/export_bvar_gvar24_loop.m (GVAR) and pipeline/export_bvar_euro_fsap.m (euro-area financial blocks).

    The problem it fixes. Under the default random-walk own-lag prior (mean 1) the fiscal GDP-ratios FREV_GDP, FPEXP_GDP, SFA inherit a near-unit-root and drift at the long horizon — e.g. US revenue stuck near the last-observed ~19% of GDP rather than reverting toward its in-sample ~17.4% mean. A GDP-ratio is economically stationary, so a random-walk prior is the wrong shrinkage target.

    The mechanism. GLP shrinks each equation's own-first-lag coefficient toward a per-variable prior mean held on the diagonal of the prior-mean matrix b. bvarGLPmf.m builds it from a vector diagb, and the pos argument zeroes the selected entries:

    % BVAR/MFBVAR/GLPreplicationWeb/bvarGLPmf.m  (lines 193–195)
diagb        = ones(n,1);
diagb(pos)   = 0;   % own-first-lag prior mean = 0 for the WN (stationary) variables in pos
b(2:n+1,:)   = diag(diagb);

    pos is the 3rd positional argument of cgl_bvarV5 (named stationary in its signature):

    % BVAR/MFBVAR/cgl_bvarV5.m
function [Xcond,storeP,BETA,SU,X_interp0,Dforecast] = cgl_bvarV5(x,p,stationary,nDraws,nBurn,MCMCconst,hz,varargin)
pos = stationary;                       % line 19
...
r = bvarGLPmf(xx(1:lastFull,:), p, ..., 'pos', pos);   % line 54

    How the estimators set pos. Each reads the per-variable prior string from the spec and selects the WN columns of the kept domestic block (stars are appended after the domestic columns and are always RW):

    % pipeline/export_bvar_gvar24_loop.m
statCols = find(Spec.Prior(keep)=="WN")';   % domestic WN columns; [] for the 44 non-fiscal blocs
...
[Xcond,~,B,S,~,Dfc] = cgl_bvarV5(x, lags, Dc.statCols, nDraws, nBurn, MCMCconst, H, ...);

    % pipeline/export_bvar_euro_fsap.m
Prior    = priorsfrom(J.variables);         % per-variable "RW"/"WN" string array, RW default
statCols = find(Prior=="WN")';
[~, storeP, BETA, SU, ~, Dforecast] = cgl_bvarV5(x, lags, statCols, nDraws, nBurn, MCMCconst, H, ARGS{:});

    priorsfrom is deliberately robust to MATLAB's jsondecode dropping a prior field that is present on only some variable objects (a heterogeneous struct array collapses the field); an absent field ⇒ all-RW (the historical behaviour).

    Which variables are WN. Only the three fiscal GDP-ratios, in the six fiscal blocs (US, EA, JP, UK, CA, CN). EFFI (the effective rate, isFinancial:true) stays RW — a rate level is not a mean-reverting ratio. The shipped US/spec.json confirms exactly three WN tags:

    Variable transformation prior financial
    FREV_GDP (revenue/GDP) lin WN macro
    FPEXP_GDP (primary exp/GDP) lin WN macro
    SFA (stock-flow adj/GDP) lin WN macro
    EFFI (effective rate, % p.a.) lin RW financial

    Effect and scope. The fiscal ratios now mean-revert to their in-sample means at the long end (US revenue → ~17.4% instead of ~19%). The change is byte-identical (a no-op) for the 44 non-fiscal blocs, whose statCols is empty. Committed in 6f0161a (prior change), 22250be / e38673a (full 33-bloc refit + port). The browser engine reads the resulting mean-reverting baselines unchanged — there is no TS-side code for the prior.

    Coherence fixes (companion to the refit)¶

    Shipped alongside the fiscal-prior refit (22250be / e38673a); they change the content of the loaded JSON, not the engine:

    • GLB_USEQ and GLB_OIL are isFinancial:true everywhere. Both are now kept observed through the COVID-as-missing window (2020Q2–Q4) instead of being imputed as macro columns, so the US-equity and oil paths condition the imputation of the masked macro block. Verified in US/spec.json (GLB_USEQ → log·RW·fin, GLB_OIL → log·RW·fin).
    • Canada IP_CA rebuilt as a log-LEVEL. Chained from the discontinued FRED level index + growth to 2025Q4, so IP_CA carries transformation:"log" (a level the engine differences to a growth rate) rather than a pre-differenced series. Verified in CA/spec.json.
    • Per-economy variable table. A transformation · prior · financial table was added to the user manual; all 1048 variables across the 50 shipped specs were audited coherent.

    ⚠️ The illustrative spec.json example in DATA_SCHEMA §3 predates this refit (it shows FREV_GDP/SFA as RW and GLB_OIL as isFinancial:false). The shipped US/spec.json is the authority: FREV_GDP/FPEXP_GDP/SFA are WN, and GLB_OIL/GLB_USEQ are isFinancial:true.

    matrix.ts¶

    Responsibility. Core dense linear algebra over flat Float64Arrays, row-major. No external dependency. Every other compute module imports from here.

    Type¶

    interface Matrix { data: Float64Array; rows: number; cols: number }

    Key exports¶

    Function Signature Notes
    mat (rows, cols, values?: number[]) => Matrix row-major; values optional
    eye / zeros (n) => Matrix / (rows, cols) => Matrix identity / zero
    get / set (a, i, j) => number / (a, i, j, v) => void element access
    multiply (a, b) => Matrix skips zero entries of a (sparse-friendly)
    multiplyVec (a: Matrix, v: Float64Array) => Float64Array matrix·vector
    transpose, add, subtract, scale, symmetrize shape-preserving add/subtract throw on dimension mismatch
    submatrix / setSubmatrix block read / write
    solve (A: Matrix, b: Float64Array) => Float64Array LU with partial pivoting; throws solve: singular matrix when the pivot magnitude < 1e-15
    solveMatrix (A, B: Matrix) => Matrix solves A·X = B column by column
    inverse (A) => Matrix solveMatrix(A, eye)
    cholesky (A) => Matrix lower-triangular L, A = L·Lᵀ; throws when a diagonal ≤ 1e-14 (not positive definite)
    toeplitzLower (irf: number[], horizons) => Matrix lower-triangular impulse-response convolution matrix for B&M
    vecAdd / vecSub / vecScale Float64Array helpers

    Numerical thresholds. LU singularity pivot floor 1e-15; Cholesky positive-definite floor 1e-14. These are the two failure modes consumers must guard (the smoother and the simulation draw catch a failed Cholesky and fall back to a diagonal sqrt).

    kalman.ts¶

    Responsibility. Companion-form state-space construction and the Durbin–Koopman (DK) filter + smoother for conditional forecasting of a single BVAR bloc. Ported from MATLAB conforekf.m / VARcf_DKcks.m / runKF_DK.m. This is the affine engine the GVAR solver probes.

    State-space form (state a_t = [y_t; y_{t-1}; …; y_{t-p+1}], dimension n·p):

    a_t = T·a_{t-1} + c2 + u_t,   u_t ~ N(0, Q)
y_t = Z·a_t + c1 + e_t,        e_t ~ N(0, R)

    Core types¶

    interface BVARParams { betahat: Matrix; sigmahat: Matrix; n: number; lags: number }
// betahat is (1 + n·lags, n): row 0 = constant, rows 1..n·lags = lag coefficients
// sigmahat is (n, n) innovation covariance

interface LinObs { h: Float64Array; value: number }  // linear-combination pin: h'·y_t = value

    Key exports¶

    Function Signature (abbreviated) Output
    buildStateSpace (params: BVARParams) => StateSpace companion T, Z, Q, R, c1, c2; R = 1e-12·I (tiny measurement noise)
    initState (data: Matrix, lags) => { a0, P0 } last p rows stacked; P0 = 1e-7·I
    kalmanFilter (ss, Y, a0, P0, linObs?) => { filteredA, filteredP, predictedA, predictedP, innovations, ZF } forward pass; Y is (n, T_fwd), NaN = missing
    kalmanSmoother (ss, predictedA, predictedP, filteredA, innovations, ZF, Y, linObs?) => Matrix DK backward pass (n·p, T_fwd)
    conditionalForecast see below the production single-bloc forecast
    affineBlocKernels see below the 18× GVAR perf path
    dkSimulationDraw (params, hist, condPaths) => { smoothed, forward } one DK simulation-smoother draw (band loop)
    bvarForecastDraw (params, hist, H, addInnovation=false) => Matrix iterated VAR forecast; addInnovation toggles parameter-only vs predictive bands

    conditionalForecast¶

    function conditionalForecast(
  params: BVARParams,
  historicalData: Matrix,                 // (T_hist, n), companion space
  condPaths: Matrix,                      // (H, n), NaN = free (to be forecast)
  linConstraints?: { t: number; h: Float64Array; value: number }[]
): { mean: Matrix; variance: Matrix[] }   // mean (H, n); variance is H × (n, n)

    Given history and a partially-observed future path, returns the smoothed forecast for all variables. Per-variable pins live in condPaths (set a cell, leave others NaN); the optional linConstraints pin a linear combination h'·y_t = value at quarter t (used for the fiscal primary-balance constraint revenue − primaryExp = target). The forecast is affine in the pinned values for a fixed pin mask — the property the GVAR solver exploits.

    Convention invariant. An all-NaN condPaths reproduces baseline_levels.mean to ~1e-11. This is the canary that the engine and the shipped data agree.

    ⚠️ The returned variance is the FILTERED state covariance, not the DK-smoothed one (it is submatrix(filteredP[t], …), kalman.ts:405). The MEAN is smoothed; the variance is not — so the conditional band is overstated for quarters that precede a future pin (a later observation would tighten them). Known issue #4.

    Called from useConditionalForecast.ts (single-bloc path), ChartGrid.tsx, gvarSolve.ts (baseline + every probe), and runGvarBandSolve.ts.

    affineBlocKernels¶

    function affineBlocKernels(
  params: BVARParams,
  historicalData: Matrix,
  condPaths: Matrix,    // (H, n) probe BASELINE: pin mask + baseline values
  cells: number[]       // variable columns to bump (one kernel set per column)
): { Sc: Matrix; kernels: Matrix[][] }
// Sc = baseline smoothed mean (H, n)
// kernels[ci][tp] = (H, n) smoothed-mean response to condPaths[tp, cells[ci]] += 1

    The exact perf path for the GVAR coupling build. Because conditionalForecast is affine in the pinned values for a fixed mask, the filter/smoother factors (PPred_t, gain PZF_t, smoother L_t, ZF_t) are value-independent. affineBlocKernels computes them once (one full filter+smoother pass on condPaths), then evaluates each unit-bump kernel as a cheap delta forward/backward recursion (pure matvecs). The arithmetic is byte-equal to running conditionalForecast once per (cell, tp) probe, at a large constant-factor win (~10–20× on the n=41, lags=3, H=20 endogenous US bloc). Used inside gvarSolve.ts's blocKernels.

    Numerical thresholds (kalman.ts)¶

    Constant Value Site Purpose
    Measurement noise R 1e-12·I buildStateSpace (set(R,i,i,1e-12)) near-exact observation; keeps the filter conditioned
    Initial state cov P0 1e-7·I initState / dkSimulationDraw tight diffuse prior on the seeded last-p rows
    Cholesky innovation floor max(Σ_ii, 1e-12) dkSimulationDraw / bvarForecastDraw diagonal-sqrt fallback when the full cholesky(Σ) is not positive definite

    These compose with matrix.ts's solve (singular pivot < 1e-15) and cholesky (PD diagonal ≤ 1e-14): a failed cholesky(Σ) in a simulation draw is caught and replaced by the diagonal square root above, so a marginally non-PD posterior Σ degrades gracefully rather than throwing.

    Worked example — single-bloc conditional forecast¶

    Pin US revenue to a fixed 18% of GDP over the first four quarters and read the smoothed forecast of every variable (the rest free):

    import { conditionalForecast } from '@/lib/compute/kalman'
import { mat } from '@/lib/compute/matrix'

// params: BVARParams parsed from companion_mean.json (n, lags, betahat, sigmahat)
// historical: (T_hist, n) Matrix in companion space (100·log levels / raw lin)
const H = 20
const n = params.n
const condPaths = mat(H, n)                 // mat() zero-fills; "free" cells must be NaN, so:
condPaths.data.fill(NaN)
const FREV = idx.revenue                    // column of FREV_GDP in this bloc
for (let t = 0; t < 4; t++) condPaths.data[t * n + FREV] = 18.0   // pin 18% of GDP, q0..q3

const { mean, variance } = conditionalForecast(params, historical, condPaths)
// mean: (H, n) smoothed forecast; mean.data[t*n + col] is variable `col` at horizon t
// variance: H entries, each an (n, n) Matrix (filtered/smoothed state cov at that horizon)

    The all-NaN canary that proves the engine and the shipped data agree:

    const free = mat(H, n); free.data.fill(NaN)
const { mean } = conditionalForecast(params, historical, free)
// mean ≈ baseline_levels.mean to ~1e-11, column by column

    Worked example — linear-combination pin (fiscal primary balance)¶

    The fiscal "required primary balance" path is conditioned as revenue − primaryExp = pb_t, a linear combination across two columns at each quarter — exactly the linConstraints channel:

    const h = new Float64Array(n)               // selector h'·y_t
h[idx.revenue] = 1; h[idx.primaryExp] = -1  // revenue − primaryExp
const linConstraints = pbPath.map((value, t) => ({ t, h, value }))   // value = required pb at q t

const { mean } = conditionalForecast(params, historical, free /* all-NaN */, linConstraints)

    This is how useConditionalForecast.ts / fiscalAccounting.ts's requiredPrimaryBalance output is fed back into the BVAR (caller iterates 1–2 passes; see fiscalAccounting.ts below).

    gvarSolve.ts¶

    Responsibility. The cross-country GVAR scenario solver. Each bloc is an independent single-country BVAR whose baseline is computed once; cross-country coupling enters only through the scenario delta, which propagates via three trade-weighted "RoW" star variables (plus, when the US is endogenous, the US global commons). Reuses kalman.ts unchanged.

    The delta system is a linear fixed point x = M·x + d (x = stacked star pin-deltas, optionally augmented with US-global deltas). It is solved two ways that agree to ~1e-6:

    1. Gauss-Seidel sweeps (solveGVAR) — sums the Neumann series Σ Mᵏ d.
    2. Direct closed form (solveGVARLinear / solveGVARClosedForm) — (I−M)⁻¹d or the inverse-free Neumann fixed point, with M cached across scenarios that share a pin structure.

    Star variables and link convention¶

    Each non-US (and endogenous-US) spec carries three block:'row' stars, all lin in the receiver:

    Star Meaning Linked partner series
    ROWDEM (DEM) trade-weighted foreign GDP growth, % q/q ann partner GDP (log)
    ROWINF (INF) trade-weighted foreign inflation, % q/q ann partner deflator/CPI (log)
    ROWFIN (FIN) trade-weighted foreign short rate, % partner policy rate (lin)

    gdp/price deltas are 4·Δ(level-gap) growth operators (t=0 anchored to the shared in-sample last level, whose delta is 0); rate deltas are the level gap. Rate-less blocs (CH, NO) drop out of FIN; hyperinflation blocs (AR) drop out of INF and FIN. (NOMINAL_GDP_BLOCS is empty since the 2026-06 CN-only re-estimate — CN now ships real GDP, so its DEM reads gdp directly rather than gdp − price.)

    Topology configuration¶

    function configureGVARTopology(specs: Partial<Record<CountryCode, SpecJSON>>): CountryCode[]

    (Re)configures the module from loaded spec.jsons + weights_24bloc.json. A country is an active RoW bloc iff its spec carries block:'row' stars and it has a weights row; the US is the closed anchor unless its own spec carries stars (then US_ENDOGENOUS = true). Mutates these live module exports in place (importers hold references):

    • ROW_COUNTRIES — every active bloc except US.
    • COUPLED_COUNTRIES — the blocs in the fixed point (ROW_COUNTRIES, US prepended when endogenous).
    • US_ENDOGENOUS — boolean.
    • IDX: Record<CountryCode, BlocIdx> — per-bloc column indices (gdp, price, rate, n, row, dem, glob), derived by deriveBlocIdx.
    • W — trade weights, renormalized per active panel so each row sums to 1 (zero diagonal).

    deriveBlocIdx(code, spec) mirrors the estimator's column resolution: gdp = GDP_<CC> else GDPH; price = HICP_<CC> else CPI_<CC> else JGDP; rate = RATE_<CC> else POLRATE else YIELD10_<CC> (optional → -1). resetGVARTopologyToDefault() restores the shipped 6-bloc set (US, EA, JP, UK, CA, CN) and runs at module load.

    Solvers¶

    Function Strategy Returns
    solveGVAR(models, scenarioPins, opts?) Gauss-Seidel; tol default 1e-4, maxSweeps default 12. US frozen unless endogenous. GVARResult (scenLvl, delta, sweepCount, residuals, rhoEmpirical)
    solveViewedScenLvl(models, pins, viewed, opts?) single-bloc shortcut: a US-only view (frozen US) needs one forecast, not the full sweep Matrix
    solveGVARLinear(models, pins, opts?) direct (I−M)⁻¹d or Smith/doubling; accepts cached coupling GVARLinearResult (adds M, d, rhoSpectral, method)
    solveGVARClosedForm(models, pins, modelSig?) production path: cached M, inverse-free Neumann; routes to the endogenous augmented form when US_ENDOGENOUS GVARResult
    solveGVARDirect(models, pins, modelSig) first-order one-hop preview (x = d), pure cached-kernel matvecs, sub-50 ms; falls back to the full solve when the kernel cache is cold GVARResult

    ScenarioPins = Partial<Record<CountryCode, Matrix>>; each pin is (H, n), NaN = free.

    M build and coupling cache¶

    M depends on the models and the conditioning structure (which cells are pinned) but not on the pinned values, so it is cached and reused across the dominant drag-a-pin UI flow. The relevant exports:

    • buildGVARCoupling / buildGVARCouplingEndo — assemble M, (I−M)⁻¹, ρ(M) (frozen / endogenous).
    • buildGVARCouplingM / buildGVARCouplingMEndo — assemble M and ρ without the inverse, for the export-time precompute that inverts (I−M) in MATLAB BLAS (precompute_gvar_coupling.mjs shells out to invert_coupling.m; invert_coupling.py is a NumPy fallback). The in-JS O(dim³) inverse was measured at ~46 min at the frozen-anchor dim≈1380, and is larger at the endogenous dim=2060, so it is never built live.

    This is the shape of app/public/data/bvar/gvar/coupling_default.json(.gz), produced by app/scripts/precompute_gvar_coupling.mjs and consumed by runGvarSolve.ts.

    US_ENDOGENOUS augmented state¶

    When the US is a coupled bloc it both receives partner spillovers (its own ROW*_US stars) and drives every RoW bloc's global commons (oil / US 10Y / US BAA / US equity), which RoW blocs pin to own-base + US-global-delta. The compact state is therefore augmented:

    x = [ star pin-deltas s_{c,v,t} for every coupled bloc c (US included) ]
    ++ [ US global-column deltas gΔ_{p,t} ]

    with three coupling families assembled in buildLinearSystemEndo: (1) star ← star, (2) US-global ← US-star (definition row), (3) RoW-star ← US-global. solveGVARClosedFormEndo solves x = (I−M)⁻¹d (cached inverse) or the inverse-free Neumann fixed point (maxIt = 2000, tolerance 1e-10), then reconstructs each bloc with one final conditionalForecast.

    ⚠️ No ρ(M) < 1 divergence guard on the live solve (known issue #3). The Neumann iteration (gvarSolve.ts:1296) stops at maxIt/tolerance but never checks ρ(M) < 1. A refit producing ρ ≥ 1 would diverge silently and return a confident-wrong forecast. ρ is shipped in the artifact (currently 0.924) but not read as a guard. See Known issues #3.

    Spectral radius by panel size¶

    spectralRadius(M) power-iterates M² (deterministic seed, 400 iters) and takes the sqrt (M is block-off-diagonal, so raw power iteration oscillates). Representative ρ(M):

    Panel ρ(M)
    6-bloc ~0.59
    24-bloc, US frozen ~0.726
    33-bloc, US endogenous (dim≈2060) 0.924041 (live coupling_default.json.gz; moves with each refit, stays < 1)

    fiscalAccounting.ts¶

    Responsibility. Derives the public-finance reporting variables (primary/overall balance, deficit, interest, debt-to-GDP, gross financing needs, snowball) from the BVAR-estimated fiscal drivers via the IMF FPP / CBO budget identity. Debt is derived, never estimated. See fiscal block methodology.

    Units (the main bug source)¶

    Quantity Unit
    revenue, primaryExp, sfa, amortization, debt-service-to-GDP % of GDP, annual ratio
    effRate % per annum (effective rate on the debt stock)
    gNomQ % per quarter (nominal GDP growth)
    d0, debt debt-to-GDP, %

    The recursion (per quarter, lagged debt d₋₁) — LIVE / displayed path¶

    This is computeFiscalBlock (fiscalAccounting.ts:163), the only fiscal recursion wired to the displayed debt-to-GDP charts and bands (called from ChartGrid.tsx and runGvarBandSolve.ts). It is a passive accounting recursion — no debt-feedback term:

    pb_t      = rev_t − primexp_t                             // primary balance (NO Bohn term)
i_eff_t   = effRate_t                                     // effective rate (NO risk premium)
int_t     = i_eff_t · d₋₁ / 100                           // interest / debt service, % GDP
deficit_t = int_t − pb_t
d_t       = d₋₁·(1 + i_eff_t/400)/(1 + gNomQ_t/100) − pb_t/4 + sfa_t/4
gfn_t     = deficit_t + amortization_t

    All four drivers (rev, primexp, effRate, sfa) are BVAR-forecast; debt/GDP is a derived accounting identity, never a BVAR variable. sfa_t in the deployed roll is the structural constant sfaDefault (not the per-quarter forecast SFA column — that gets dragged to extremes by the long-run anchors). There is no debt→primary-balance reaction and no debt→yield risk premium in anything the user sees.

    ⚠️ The γ/β (Bohn FRF + risk-premium) levers are DORMANT / test-only. A separate function runFiscalRecursion (fiscalAccounting.ts:66) adds the optional terms pb_t += γ·(d₋₁ − d*) (Bohn, default 0.03) and i_eff_t += β·max(0, d₋₁ − d*) (risk premium, default 0), gated by feedbackOn. runFiscalRecursion has ZERO production callers — it is invoked only by __tests__/compute/fiscalAccounting.test.ts and fiscal-backcast.test.ts. No UI control sets γ/β/feedbackOn. They are documented and validated but not in the live path; a possible future endogenous-debt-feedback extension, not current behaviour. See fiscal-block-methodology.md.

    ✅ CN inflation double-count — RESOLVED (2026-06). computeFiscalBlock forms gNomQ as Δ(level[idx.gdp]) + Δ(level[idx.deflator]). The CN block previously shipped a nominal GDP_CN, so the price term was double-counted; the surgical CN-only re-estimate replaced it with real GDP (idx.gdpIsNominal = false), so CN's roll-forward now matches US/EA/JP/UK/CA. CN-only graft; the other 32 blocs are byte-identical.

    Key exports¶

    Function Input → output
    runFiscalRecursion(input: FiscalRecursionInput) driver arrays → FiscalRecursionResult. If dsvc (estimated debt-service-to-GDP) is supplied it is the interest and the effective rate is backed out; else the effective-rate form is used.
    computeFiscalBlock(input: FiscalBlockInput) reads drivers off a level-space (H × n) forecast via FiscalBlockIndices (gdp, deflator, revenue, primaryExp, effRate, optional debtService, optional sfa); rolls the same identity forward → FiscalRecursionResult. gQ = q/q nominal growth = Δ(real 100·log) + Δ(deflator 100·log).
    requiredPrimaryBalance(input: DebtInversionInput) inverts the recursion: the annual primary balance per quarter required to make debt follow targetDebt (IMF DSA "required primary balance"). pb = 4·[(d₋₁ + int/4)/(1+gQ) − d_t] + sfa. The caller conditions the BVAR on this path via the revenue − primExp linear constraint, iterating 1–2 passes.

    FiscalRecursionResult carries primaryBalance, effectiveRate, interest, overallBalance, deficit, debt, dDebtGdp (= debt[t] − debt[t-1], derived), gfn, snowball — each a length-H array.

    Driver provenance — the stationary prior matters here¶

    The three estimated drivers computeFiscalBlock reads off the level path — revenue (FREV_GDP), primaryExp (FPEXP_GDP) and sfa (SFA) — are exactly the variables now carried with the stationary WN prior (§Fiscal stationary prior). So the level path this module differences and rolls forward now mean-reverts at the long end rather than drifting; the derived debt-to-GDP trajectory inherits that. effRate (EFFI) stays RW. The recursion code is unchanged — only the shape of the forecast it consumes changed.

    The two parametrisations (debt service vs effective rate)¶

    Both runFiscalRecursion and computeFiscalBlock branch on whether an estimated debt-service-to-GDP column is supplied:

    Supplied Interest intₜ (% of GDP) Effective rate (% p.a.)
    dsvc / idx.debtService set = dsvcₜ directly (the BVAR forecast) backed out: intₜ·100/d₋₁
    not set (fallback) = effRateₜ · d₋₁ / 100 = effRateₜ (the estimated EFFI)

    The quarterly roll-forward iQ = effRate/400 = intₜ/(4·d₋₁) is identical either way (this keeps the effRate-only callers byte-identical and fixes a ~100× over-statement of interest on the DSVC path).

    Worked example — derive the fiscal block from a GVAR scenario level path¶

    import { computeFiscalBlock } from '@/lib/compute/fiscalAccounting'

// prevLevelRow: the last historical level row (for the t=0 growth diff)

  levelPaths, prevLevelRow,
  idx: {                                  // FiscalBlockIndices — column positions in the level path
    gdp: usIdx.gdp, deflator: usIdx.price,
    revenue: cRev, primaryExp: cPexp, effRate: cEffi,
    sfa: cSfa,                            // ESTIMATED SFA column (WN-prior) → per-quarter, with bands

})
// result.debt[t]    : derived debt-to-GDP (%), budget-identity roll-forward
// result.deficit[t] : interest − primary balance (% of GDP, annual)
// result.snowball[t]: ((1+iQ)/(1+gQ) − 1)·d₋₁  (the interest-growth differential, pp)
//   where gQ = q/q nominal growth = Δ(real 100·log) + Δ(deflator 100·log), over 100

    This is the exact call runGvarBandSolve.ts makes per draw (line ~163, req.fiscal.sfa.slice(0,Hf)) to band the US debt path, and the single-bloc useConditionalForecast.ts / ChartGrid.tsx make for the central path.

  targetDebt,          // desired debt/GDP path, % (length H) — e.g. a flat 100% anchor

    taylor-rule.ts¶

    Responsibility. Helpers for a rule-based mode: an implied policy-rate path from a 3-parameter Taylor rule, plus translation to the Macroeconomic Model Database (MMB) 33-coefficient rule format. No component currently imports this module (no callers in src/, no test) — it is a standalone, self-contained API kept for the rule-based path.

    barnichon-mesters.ts¶

    Responsibility. Optimal monetary policy via the Barnichon & Mesters (2023 AER) sufficient- statistics approach: given DSGE IRFs to a policy shock, find the shock sequence minimizing the central bank's loss, as a regularized least-squares QP min (H·ε − b)ᵀ W (H·ε − b), ε* = (HᵀWH)⁻¹ HᵀWb.

    Key exports¶

    Function Input → output
    optimizePolicy(irfs: DSGEIRFs, targetInflation, targetOutput, lambdaPi, lambdaY) irfs = { inflation, outputgap, interest } (number[] IRFs); targets are Float64Array(H) deviations. → BMOptimizationResult (optimalShocks, policyPath, impliedInflation, impliedOutput, loss). Builds Toeplitz convolution matrices via matrix.toeplitzLower; normal equations regularized with Tikhonov α = 1e-6 on the diagonal.
    computeCounterfactualDeltas(irfs: Record<string, number[]>, rateDeviation: Float64Array) back out the shock sequence from a desired rate deviation (forward substitution on the lower-triangular interest IRF), then convolve every IRF → Record<string, Float64Array> of per-variable deltas. Reads irfs['interest'] or irfs['interest_']; throws if neither is present or if the impact effect |interestIRF[0]| < 1e-6.

    optimizePolicy is called from components/controls/ControlPanel.tsx. computeCounterfactualDeltas is currently only exercised by barnichon-mesters.test.ts.

    longRunTilt.ts¶

    Responsibility. After-estimation entropic tilting toward a "Prior for the Long Run" (PLR) restriction (Giannone–Lenza–Primiceri 2019). It does not re-estimate: it takes the existing posterior draws and computes per-draw importance weights w_i ∝ exp(γ'·g_i) so the weighted mean of each long-run moment hits a target, then the band loop applies them as weighted quantiles. Ported from pipeline/_verify/entropic_tilt.m.

    The long-run moment¶

    Default mode: 'forecast' — g = h'·E[y_{T+H}], the H-step mean forecast (default horizon = 40 ≈ 10y). The alternative mode: 'steadyState' (g = h'·(I−ΣB_l)⁻¹c) is opt-in and only valid for genuinely stationary combinations: on the I(1) 100·log-level real VARs (I−ΣB_l) is near-singular and the steady state diverges (verified −21103 vs an H≈40 forecast of +35), so forecast is the default.

    Key exports¶

    Function Input → output
    computeLongRunMoments(draws, historicalData, restrictions, opts?) per draw, per restriction → gByDraw[r] = Float64Array(nDraws)
    solveEntropicTilt(gByDraw, restrictions, opts?) Newton dual with backtracking line search → LongRunTiltResult (weights, ess, tiltedMean, priorMean, gamma, converged, maxResidual). maxIter default 200, gradient tol default 1e-10.
    longRunTilt(...) convenience wrapper: moments + tilt
    computeLongRunTilt(draws, historicalData, anchors: AnchorSpec[], opts?) the function ChartGrid consumes: display-unit variable anchors → LongRunTiltDiagResult (weights, ess, nDraws, tiltedMean, priorMean, maxResidual, converged). Empty anchors → uniform weights / full ESS.
    computeAnchorMoments(draws, historicalData, anchors) one mean-forecast pass per draw; log anchor → 4·Δ annualized growth at H, lin anchor → level at H
    weightedQuantile(values, weights, q) weighted quantile (drops non-finite, renormalizes)
    weightedBands(drawPaths, weights, H, n, qLo=0.16, qHi=0.84) per-draw paths → { q16, q84 }, each (H × n) 
    interface AnchorSpec { varIdx: number; isLog: boolean; target: number; horizon?: number }

    PosteriorDraws is the flat draws layout beta[draw*k*nv + j*k + i], beta_shape = [k, nv, nDraws] — the same layout loadCompanionDraws produces. ESS = 1/Σwᵢ² ∈ [1, nDraws]; a joint multi-constraint tilt can degenerate (ESS → 1) at low draw counts, so computeLongRunTilt std-scales each anchor by its own sample standard deviation for conditioning.

    ⚠️ Non-finite long-run moments are NOT filtered before the Newton dual (known issue #2). solveEntropicTilt builds Gc straight from gByDraw and the softmax exp(Σ Gc·γ) will NaN if any draw's moment is non-finite (an unstable draw whose long-horizon forecast diverged), silently emptying the anchor band. The scaling pre-pass skips non-finite values, but the matrix fed to the dual does not. See Known issues #2.

    useGvarForecast.ts — drag-aware two-tier GVAR solve hook¶

    Responsibility. The React hook that turns the active bloc's own-pins into a propagated GVAR forecast, keeping the heavy solve off the live drag path. It owns a module Worker (main-thread fallback), a stale-response reqId guard, and two timers (a live debounce + a separate drag-release macrotask). The load-bearing perf fact (verified in the code, useGvarForecast.ts:279): on the live 50-bloc US-endogenous panel every mean solve — direct OR full — is ~4571 ms warm / ~13327 ms cold (the US-pinned structure misses the shipped empty-pinStructure coupling artifact and falls to the dim-2060 Neumann fixed point + one Kalman smoother per coupled bloc).

    The drag boundary (useGvarForecast.ts:287):

    • While a drag is in progress (isGvarDragging()): ZERO solves fire — neither value-only nor structure-change. The hook marks pendingReleaseSolveRef = true and bails. The dragged bloc's own line still follows the cursor live because ChartGrid's effective falls back to the single-country condForecast (a ~1–5 ms DK smoother) while gvarFc is frozen at the last solve.
    • On pointer-up (onGvarDragEnd): exactly one direct solve is dispatched, deferred to a ~100 ms macrotask (coalesces back-to-back gestures + lands after the final store write flushes into dataRef). direct is used so the ~95 MB coupling artifact is stripped (withoutCoupling) from the worker post — solveGVARDirect rebuilds/caches M from the per-bloc kernel cache and never consumes the inverse.
    • Non-drag edits (numeric/programmatic pin entry, Reset, scenario switch) dispatch live: full solves immediately (explicit user action — solveFullToken bumped via Compute / "Solve full GVAR"), direct previews after a 120 ms debounce.

    So the full cross-country fixed point (mode:'full', hydrated (I−M)⁻¹) fires only on the explicit Compute / "Solve full GVAR" token bump; everything else is the direct one-hop preview. Pinned by gvar-drag-scheduling.test.ts.

    Render memoization in ChartGrid (reconcileCharts + chartDeepEqual + React.memo)¶

    Responsibility (Phase-2 render optimization, issue #68). A single conditioning edit changes scenarios, re-running the monolithic charts useMemo → every chart object gets a fresh identity → absent memoization, Plotly would repaint all ~35 panels. Two levers (in ChartGrid.tsx) make one edit repaint ≈1 panel:

    Mechanism Where What it does
    chartDeepEqual(a, b) ChartGrid.tsx:173 tight structural equality over the chart objects (plain objects / arrays / primitives — no Maps/Sets/Dates/functions appear in a chart object)
    reconcileCharts(next, prev, onRebuilt?) ChartGrid.tsx:205 reuses the PRIOR render's object reference for any panel whose freshly-built content is deep-equal (keyed by chartVar.bvarIdx, so a panel maps to its prior self even if list order/length shifts on a country switch); returns the prior array when every element kept its reference (stable outer identity, no [charts] effect churn)
    MemoDraggableChart = memo(DraggableChart) ChartGrid.tsx:2523 so a stable chart reference + the grid's useCallback'd handler props mean an unchanged panel does not re-render / repaint

    The math/builder is untouched: reconcileCharts only swaps a NEW object for an EQUAL old one, so the rendered data is byte-identical; it just preserves reference identity where the content did not move. Two test-only instrumentation counters (chartRebuildCounter, draggableChartRenderCounter, both exported, ~zero production cost) let chartgrid-render-memo.test.ts assert that one conditioning edit rebuilds ≈1 panel and re-renders only the touched panel(s), not all ~35.

    Empirical evaluation (companion repo — in progress)¶

    A separate repository — em-bayesian-gvar-eval (sibling to this one) — turns the deployed engine + the MATLAB estimator into a publishable evaluation. It does not ship in the app; it is the empirical core of the paper. The full design is final in em-bayesian-gvar-eval/docs/EMPIRICAL_PROTOCOL.md; results are preliminary / forthcoming.

    It races two proposed models against six benchmarks (model codes are the registry in eval/tables.py / eval/score.py):

    Code Model Role
    emgvar_endo EM-Bayesian GVAR, US endogenous proposed (headline)
    emgvar_frozen EM-Bayesian GVAR, frozen US anchor proposed
    rw driftless Random Walk (+ sample-mean "climatology" anchor) universal reference
    arp AR(p), p by AIC over {1..4}, iterated and direct strong univariate
    sbvar single-country Bayesian VAR (stars dropped, same prior) isolates coupling
    cgvar_a textbook OLS Pesaran–Schuermann–Weiner / Dees GVAR (no COVID-as-missing) literature GVAR
    cgvar_b single-pass Bayesian GVAR (MAXIT=1, COVID-as-missing) isolates EM completion
    dfm Bayesian dynamic factor model non-GVAR global benchmark

    Nesting map (stated in the paper): sbvar ⊂ cgvar_b ⊂ emgvar_*; RW/AR nested in every VAR.

    Design (final). Recursive expanding window, origins 2010Q1–2024Q4 (drop the 2020 COVID origins from the headline); rolling as-of-t0 trade weights (leak-free — built from IMTS bilateral trade with endPeriod ≤ year(t0); the shipped 2021–2023 weights_full.json is forbidden in the OOS sweep, as it would inject future trade structure); targets {real GDP growth, inflation, short rate} for US/EA/JP/UK/CA/CN; horizons h ∈ {1, 2, 4, 8} (each scored only where a ≤2025Q4 realization exists). COVID-as-missing reapplies in-sample at every origin where 2020Q2–Q4 ≤ t0.

    Scoring. CRPS is the primary density score (ensemble estimator, robust at the modest draw counts); a parametric Gaussian/skew-t log-score corroborates; PIT / coverage for calibration. Significance: Giacomini–White is the headline test (valid under nesting and the recursive scheme), Clark–West for nested point/MSE comparisons, Diebold–Mariano (HAC, HLN-corrected) for non-nested pairs only; multiplicity handled with Romano–Wolf + the Model Confidence Set per (variable, horizon).

    Part 2 — spillovers. Tier A (ships now, zero new estimator code): the conditional-scenario endogenous-vs-frozen US contrast, driven through the shipped gvarSolve.ts (x = Mx + d) — its driver is em-bayesian-gvar-eval/eval/tier_a_scenario.mjs, specified to be exercised from the main repo via a vitest harness (app/src/__tests__/eval/tierA_spillover.test.ts, planned). Tier B (new gvar_girf module, constructible from the exported betahat/Σ + coupling M): GIRF / generalized FEVD / Diebold–Yilmaz index. The protocol is explicit that gvarSolve.ts computes conditional-mean scenario deltas, not GIRFs/FEVD — the only spillover strings in this repo are UI labels.

    Headline result (first full pass — results/tables.{txt,json}, 56 origins, 9 models). AR(p)/DFM have the lowest point CRPS, but the 90% Model Confidence Set is the publishable signal: emgvar_endo is in the inflation MCS at all horizons and the short-rate MCS at h=1,2,4 (indistinguishable from the best on the nominal block), out of the growth MCS at h=1 & h=8; emgvar_endo ⪰ emgvar_frozen across the MCS; and the EM-Bayesian treatment decisively beats the textbook OLS classical GVAR (cgvar_a is removed from most sets). The unconditional point leader for inflation h=1 is the univariate AR(p) (relRMSE 0.641); emgvar_endo (relRMSE 0.799) is the best among the EM/GVAR family, not the overall best. Honest caveats: the draw-based densities are over-confident (PIT cov90 ~0.73–0.83 vs 0.90), and the parametric log-score column is numerically broken (CRPS + MCS are the robust primaries). Tier-A (results/tierA/us_rate_100bp.json) shows non-trivial endogenous-only cross-country feedback the frozen anchor omits, but the magnitudes are large/sign-flipping and it is a multi-period conditional projection, NOT a causal IRF — needs a validation pass before being headlined. The design is final; see em-bayesian-gvar-eval/docs/EMPIRICAL_PROTOCOL.md.

    Known issues & limitations (flagged-for-fix, NOT yet fixed)¶

    Documented honestly so a reader does not mistake a known bias for a verified result. Each is flagged-but-unfixed at the current deployment; cite this section, do not silently rely on the affected output.

    1. CN fiscal debt — inflation double-count (fiscalAccounting.ts). RESOLVED (2026-06). The CN debt-snowball roll-forward computes nominal growth as gQ = Δ(level[idx.gdp]) + Δ(level[idx.deflator]). CN previously shipped a nominal GDP_CN, so the price term was double-counted and the CN debt-to-GDP path was biased. The surgical CN-only re-estimate replaced GDP_CN with real GDP (spec.json var 0 = "CN Real GDP (quarterly, chain-linked SA volume index, base 2011Q1=100)", fiscal_meta.json idx.gdpIsNominal = false), so the snowball now reads the correct real-GDP + deflator nominal growth, matching US/EA/JP/UK/CA. The fix re-estimated CN only; the other 32 GVAR blocs are byte-identical to the prior working model.

    2. Entropic long-run tilt does not filter non-finite moments (longRunTilt.ts). solveEntropicTilt builds the centered/scaled constraint matrix Gc.push(col) directly from the per-draw moments gByDraw (longRunTilt.ts:~254) and never drops non-finite draws. The Newton dual then evaluates z[i] = Σ Gc[r][i]·γ_r and exp(z[i] − zmax) (:284–:289): a single non-finite Gc[r][i] (from an unstable draw whose long-horizon forecast diverged) makes z[i]=NaN, exp(NaN)=NaN, which poisons wsum and silently empties the anchor band (all weights NaN). The scaling pre-pass does skip non-finite values when computing each anchor's mean/std (:529–:532), but the matrix fed to the dual is not filtered. Fix: drop the non-finite draws (and renormalize) before the Newton iteration, as weightedQuantile already does at the band step.

    3. No ρ(M) < 1 divergence guard on the live closed-form solve (gvarSolve.ts). The endogenous Neumann fixed point x ← d + M·x (gvarSolve.ts:1296) iterates to maxIt = 2000 or a relative tolerance 1e-10·(1+‖x‖) — but there is no check that ρ(M) < 1. If a refit produced a M with ρ ≥ 1 (a non-stationary coupling), the iteration would diverge (or stop at maxIt far from a fixed point) and the solver would return a confident-wrong forecast with no warning. ρ is computed at export time and shipped in the artifact (and is currently 0.924), but the live solve does not read it as a guard. Fix: assert coupling.rhoSpectral < 1 (or estimate ρ and bail/flag) before trusting the iterate.

    4. Conditional band uses the FILTERED, not DK-smoothed, covariance (kalman.ts). conditionalForecast returns the smoothed MEAN but extracts the per-horizon variance from the filtered state covariance: variances.push(submatrix(filteredP[t], 0, 0, n, n)) (kalman.ts:405–:409, the comment says "Extract forecast variances from filtered covariances"). The DK smoother would tighten the covariance for quarters that precede a future pin (a later observation reduces earlier-quarter uncertainty); using the filtered covariance does not apply that reduction, so the conditional band is overstated for the quarters before a future pin. The mean is correct; only the band is conservative.

    5. The porter does not regenerate fiscal_meta.json — STALE-POSITIONAL-INDEX (port_gvar_exports.py). fiscal_meta.json's six idx values are positional column offsets, but pipeline/port_gvar_exports.py ports the file verbatim. Any variable-order change (the 33-bloc GVAR-EM refit inserted variables ahead of the fiscal block in every spec except US) silently re-points them — EA's effRate once landed on the equity index (~col 537) and debt/GDP exploded. Mitigation (mandatory after any re-port): run node pipeline/fix_fiscal_meta_indices.mjs, which re-resolves all six indices BY VARIABLE ID from the deployed spec.json (idempotent, preserves d0/sfaDefault). fiscal-meta-indices.test.ts is the guard that blocks a deploy with stale indices. See DATA_SCHEMA §9 and ARCHITECTURE §7.

    Build & test¶

    From app/:

    npx tsc --noEmit     # type check
npm test             # FAST suite (vitest run minus the heavy solver --exclude globs)
npm run test:heavy   # HEAVY solver/perf canaries (1 fork) — BOTH halves gate the deploy
npm run build        # static export

    Compute-engine tests live in app/src/__tests__/compute/. The suite is split — npm test is vitest run minus seven heavy --exclude patterns; npm run test:heavy runs exactly those serialised to one fork; npm run test:all sweeps everything (see BUILD_AND_CI.md). The deploy-blocking canaries (all in the fast suite) are scenario-scale-convention, gvar-convergence (Gauss-Seidel ≡ closed-form), gvar-coupling-artifact (the shipped .gz matches a live build at the current modelSig), and fiscal-meta-indices (the STALE-POSITIONAL-INDEX guard above).
