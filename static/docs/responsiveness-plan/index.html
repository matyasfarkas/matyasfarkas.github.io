    2. Two tiers¶

    QUICK WINS — low-effort, low-risk, do now (independent of delta-c)¶

    These can land immediately, in parallel with the delta-c slider work, with no merge contention:

    1. #1 Pre-gzip companion_draws.json — the single biggest byte-for-effort win on the Compute path. Reuses the proven coupling-artifact .gz + DecompressionStream pattern. Also cuts the deploy ~310 MB.
    2. #2 Prune scenario_priors/ — 559 MB of pure dead deploy weight, zero runtime references. One deploy-pipeline change; removes the dominant 1 GB-Pages-limit risk.
    3. #3 Custom scatter-only Plotly bundle — app emits only type:'scatter' (verified: 0 scattergl, 0 other trace types). Drops ~2–2.5 MB of download+parse with zero visual change.
    4. #8 Idle-prefetch US draws — warms _drawsCache while the user reads default charts; first Compute becomes near-instant. (Land after #1 so it prefetches the .gz.)
    5. #9 Parallelize loadBVARData optional fetches — fold the 3 serial baseline_levels / historical_levels / fiscal_meta awaits into the existing Promise.all (use allSettled for 404 tolerance). −2 serial round-trips per switch.
    6. #15 Throttle the 132-fetch loadGVARModels burst — concurrency pool of ~6-8 and/or a shipped manifest to stop probing 33 blocs with 404s; start the coupling .gz download in parallel.

    4 (no-unmount-on-switch) is also independent of delta-c but is rated M/med (touches¶

    page.tsx render gating + a loading-overlay), so it sits just above the structural tier rather than in the trivially-safe quick-wins list. Strongly recommended as the next thing after the byte wins.

    STRUCTURAL — higher effort/risk, schedule deliberately¶

    • S1 Band on affine fast path — the remaining compute floor. The mean solve already has a closed-form affine path (solveGVARDirect/affineBlocKernels, ~10-20× faster, byte-equal); the band per-draw loop pointedly does NOT use it. Partner kernels + coupling M are draw-invariant → build once outside the loop; per draw rebuild only the viewed bloc's kernels. EA/JP band ÷4–÷10. Overlaps gvarSolve.ts (delta-c) — sequence after.
    • S2/S3 — second smoother-pass removal + Kalman constant-factor wins. S3 multiplies through every solve/band/kernel call, so it compounds with S1. Both touch gvarSolve.ts/kalman.ts; S3's matrix.ts/kalman.ts work is independent of the delta-c files.
    • S4 Virtualized/staggered grid mount — LazyChart (the index×100ms staggered mounter) is defined but wired into 0 grids (verified). Reviving it (or an IntersectionObserver gate) breaks the ~40-panel synchronous newPlot storm into frames. Pairs naturally with #4.
    • S5 Per-panel memoized trace build — the structural ceiling under the per-render redraw fixes. rawCharts rebuilds all ~35 panels' trace arrays on any scenarios change even though reconcileCharts then repaints only 1. Split the builder so only the touched variable's panel rebuilds. Heavy; overlaps ChartGrid.tsx.

    4. Single highest-leverage change for perceived responsiveness¶

    #4 — Stop unmounting ChartGrid on country switch (page.tsx:20 setBvarData(null) + page.tsx:60-62 gate).

    Rationale: country switch is the most-felt "frozen" action. Today it tears down all ~35 react-plotly panels and cold-restarts them — ~35 serialized Plotly.newPlot + axis/WebGL inits on the main thread, layered on top of the data fetch. Keeping the grid mounted under a loading overlay (the uncertaintyComputing overlay pattern already exists) makes panels diff-update via Plotly.react instead of remounting, so the switch becomes fetch-bound only. It is independent of the delta-c files and pairs naturally with S4.

    Byte caveat: for raw download/deploy leverage the top pick is #1 (companion gzip, −21 MB US per Compute) + #2 (scenario_priors prune, −559 MB) — both independent, both quick. #4 is the top pick for interaction-latency feel; #1+#2 are the top picks for load/deploy. Land all three first.
