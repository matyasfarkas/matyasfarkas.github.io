    Data Refresh 2026Q1 — Rollout Runbook (no re-estimation)¶

    Status: US pilot LANDED on improve-live. Full rollout is HELD for green-light (this doc).

    This is the operator runbook for extending the rest of the panel to 2026 Q1 with the same no-re-estimation refresh that shipped for US. It complements the design in docs/data-refresh-2026q1-feasibility.md (read that first for the why); this file is the how to roll out and the what to check before merge.

    The mechanism is fixed: app/scripts/refresh-quarter.mjs extends the OBSERVED panel by one quarter and lets the forecast RE-ANCHOR from the FIXED companion. No estimation runs. cgl_bvarV5.m is never touched. companion_mean / companion_draws(.gz) / spec / transformations / coupling / weights stay frozen. Per bloc the script rewrites only: historical.json, historical_levels.json, baseline.json, baseline_levels.json, and (fiscal blocs only) fiscal_meta.json d0 — plus the single per-bloc date in app/src/lib/dates.ts.

    0. Hard prerequisite — the source registry (this is what gates the rollout)¶

    refresh-quarter.mjs only knows how to refresh a bloc that has a SOURCES[<CC>] entry (live source IDs + transform + window, honouring the R6 ID traps). Today only US is mapped. Every other bloc is fail-closed: running the script against it exits 4 ("NO SOURCE MAP — refusing to touch it") and writes nothing. CN / IN are fail-closed by design (exit 3, structural laggards — see R5).

    So the rollout is NOT a single command yet. It is: author + gate the registry for each target bloc, one at a time, then run the refresh. The "one command" below is the per-bloc command; the rollout is that command run down the target list once each registry is in.

    Target set (advanced economies + FRED-sourced EMs)¶

    EA  (+ euro members)   UK   JP   CA   BR   MX   ZA

    HELD at 2025 Q4 (never fabricated, R5): CN, IN — real source data still ends ~2025 Q2; they keep the 2025-10-01 override already in dates.ts. Do not attempt to advance them.

    Israel (IL) stays at its existing 2024-10-01 override unless/until its quarterly source advances — out of scope for the 2026Q1 rollout.

    1. The one command (per bloc)¶

    Gate-then-write (the rollout command), run from app/:

    FRED_API_KEY=8ce324fcfb4663bfef63d3d44fbdbc73 \
  npx vite-node scripts/refresh-quarter.mjs --bloc <CC> --asof 2026Q1

    Useful variants (use these while authoring a new registry):

    # gate ONLY — proves transform/window/source fidelity, never writes:
… scripts/refresh-quarter.mjs --bloc <CC> --asof 2026Q1 --gate-only
# gate + project + report, still no write (eyeball the seam/bands first):
… scripts/refresh-quarter.mjs --bloc <CC> --asof 2026Q1 --dry-run
# tighten the gate against a full-precision DataRaw_*_glob.json panel:
… --gate-tol 1e-9     # default is 5e-3 vs the 6-sig-fig shipped historical_levels

    Exit codes (what a non-zero means)¶

    code meaning action
    0 gate passed, wrote (or, with --gate-only/--dry-run, validated) proceed to verify
    1 GATE FAILED — transform/window/source fidelity not proven, or fatal fix the registry; nothing was written
    2 bad args (missing --bloc/--asof) fix invocation
    3 bloc is a HELD laggard (CN/IN) expected — leave held
    4 no source map for this bloc author SOURCES[<CC>] first, then re-run
    5 new quarter not fully published yet for this bloc wait for the data; do not partial-write
    6 fiscal: debt/GDP series missing at as-of fix the fiscal source id

    The gate (R1) is the binding safety check. Before appending the new quarter it re-derives the existing last shipped row (2025 Q4) from the same live sources the same way and diffs it against the shipped panel. STABLE columns must be byte-exact (max abs diff 0); revision-prone columns (debt/IP/ cumulative) are allowed a documented revision tolerance. On any STABLE-column breach it writes nothing. A passing gate is the precondition for the write — there is no override.

    2. Rollout sequence¶

    For each bloc in the target set, in order, do NOT batch — one bloc fully verified before the next:

    1. Author SOURCES[<CC>] in refresh-quarter.mjs (live FRED/Eurostat IDs + transform + window). Honour the R6 traps: discontinued OECD-on-FRED CPI mirrors; FREV/FPEXP are internal names, not FRED IDs; stale Eurostat HICP → use CP0000EZ19M086NEST for EA HICP. EA members get the shared-EA exogenous rate/corp handling already baked into the model spec (do not re-source those endogenously).
    2. Gate with --gate-only until it passes (max abs diff 0 on STABLE cols). If it can't pass, the registry is wrong — fix the IDs/transform/window, never loosen the tolerance to force it.
    3. Dry-run (--dry-run) and eyeball the reported seam + bands (see §3 R8 note — a continuous level with a noisy annualized q/q step is fine; a true level discontinuity is not).
    4. Write (no flag) once gate+dry-run are clean.
    5. Set the date in dates.ts: when the whole target panel has advanced, the default is already 2026-01-01; a bloc that advances to 2026Q1 needs no override (it inherits the default). Only add/keep an override for a bloc that does NOT advance (CN/IN held; IL held).
    6. Verify (§3) for that bloc, then move on.

    When all target blocs are in: re-run the GVAR star aggregates if the build owner wants the ROW*_X star columns recomputed from the refreshed cross-section (the US pilot held them last for a single-bloc refresh). For a panel-wide 2026Q1 advance this should be revisited so the star aggregates reflect the new quarter rather than held-last.

    3. Verify checklist (per bloc, before merge)¶

    Run all of these green before considering a bloc done. (The US pilot passed every item.)

    • [ ] tsc clean: cd app && npx tsc --noEmit → exit 0.
    • [ ] cgl_bvarV5.m 0-diff: git diff --stat BVAR/MFBVAR/cgl_bvarV5.m empty. Production estimator untouched.
    • [ ] Frozen artifacts unchanged: git status --porcelain app/public/data/bvar/<CC>/ shows ONLY historical.json, historical_levels.json, baseline.json, baseline_levels.json, and (fiscal) fiscal_meta.json modified. companion_mean / companion_draws(.gz) / spec / transformations MUST NOT appear.
    • [ ] fiscal_meta = d0 only (fiscal blocs): git diff …/fiscal_meta.json changes only d0; sfaDefault and idx frozen.
    • [ ] History extends one quarter: historical*.json row count +1, last row dated 2026 Q1.
    • [ ] R1 gate PASS: the run's gate report shows STABLE max abs diff 0 (revision-prone within tol).
    • [ ] Bands sane: baseline_levels.json mean+bands all finite, q05 ≤ q16 ≤ q84 ≤ q95 everywhere.
    • [ ] Re-anchor (split-brain healed): the engine's re-projected baseline (from the refreshed history) equals baseline_levels.json mean to ~0. The grey fan and the engine sit on the SAME vintage.
    • [ ] δc = 0 == baseline (R7): an all-unconditional solve reproduces the baseline (no spurious shift).
    • [ ] Dates plot (R… ragged edge): lastHistOf('<CC>') = 2026-01-01; x=−1 is 2026Q1, x=0 is the first forecast (2026Q2); held blocs still date at 2025Q4. No off-by-one on the x-axis.
    • [ ] Seam continuity (R1, the GATE visual): the level across the 2025Q4→2026Q1 join is continuous. A large annualized q/q number on a genuinely volatile quarter (e.g. WTI) is annualization, not a discontinuity — confirm against the underlying level before flagging.
    • [ ] No CN/IN fabrication (R5): CN/IN untouched, still held at 2025Q4.

    4. Guardrails (do not violate)¶

    • improve-live ONLY. No deploy, no push to main, no production release as part of the rollout.
    • No re-estimation. Reuse kalman.ts / bvarForecastDraw verbatim; never touch the companion.
    • Never git add -A. Stage only the touched bloc's data dir + the script + dates.ts + this doc.
    • Fail-closed is a feature. If a bloc exits 4/5/6, the data isn't ready or the registry is wrong — do not force a write. Honour CN/IN as held.
    • Never loosen the gate to make it pass. A failing gate means the source/transform/window is wrong.

    5. Files¶

    • app/scripts/refresh-quarter.mjs — the refresher (gate + re-project + bands writer).
    • app/src/lib/dates.ts — the single per-bloc last-historical-date constant (default + overrides).
    • app/public/data/bvar/<CC>/{historical,historical_levels,baseline,baseline_levels,fiscal_meta}.json — the regenerated per-bloc display/state files.
    • docs/data-refresh-2026q1-feasibility.md — the design and the R1–R8 risk register.
