    Monetary-Financial Model -- User Guide¶

    1. Overview¶

    The Monetary-Financial Model (a global projection and scenario tool) produces conditional macroeconomic forecasts built on a Bayesian Vector Autoregression (BVAR) baseline and lets you explore "what-if" policy counterfactuals using DSGE impulse-response functions from the Macroeconomic Model Database (MMB). You select a country, condition the BVAR forecast on paths you choose for any subset of variables, and optionally overlay a DSGE-based policy counterfactual to see how alternative interest-rate paths would propagate through the economy.

    2. Getting Started¶

    When the application loads it shows a top bar, a left-hand control panel, a chart grid occupying the main area, and a scenario bar pinned to the bottom of the screen.

    Choosing a country¶

    In the top bar, use the Country dropdown to select one of roughly 50 economies — 6 core majors (US, Euro Area, UK, Japan, Canada, China), 34 OECD members, and 10 non-OECD emerging markets, grouped in the dropdown. Switching the country loads a new BVAR specification; conditioning, scenarios, and uncertainty bands are reset for that country.

    The United States carries a fiscal block (see The US Fiscal Block below). 33 of the economies are linked into a Global VAR (GVAR) — the US plus 32 other blocs (the Euro Area as one aggregate + 31 Rest-of-World blocs) — so conditioning one of them propagates to all the others (see Global VAR Cross-Country Linkage below). The US is a fully endogenous coupled bloc: it drives the rest of the world and now also receives spillback feedback from it. The remaining economies (including the 17 individual euro-area members) are standalone single-country models: useful on their own, but not part of the cross-country spillover system.

    Reading the layout¶

    • Top bar -- country selector, uncertainty controls, full-session Export button.
    • Control panel (left sidebar) -- scenario conditioning sliders, policy counterfactual settings, per-scenario Save/Load/Reset.
    • Chart grid (center) -- one chart per variable, arranged in a 3-column grid.
    • Scenario bar (bottom) -- tabs for each scenario, Add Scenario button, format-export stubs.

    3. Conditional Forecasting¶

    Conditional forecasting answers the question: "If variable X follows a path I specify, what does the BVAR imply for all other variables?"

    Activating conditioning on a variable¶

    Each chart has a small "condition" button next to its title. Click it to add that variable to the conditioning set. The button changes to "CONDITIONED" and the chart border highlights in blue. Click again to remove conditioning.

    Right-click to plant a pin at the cursor. You can also right-click directly on a forecast chart to condition it: the right-click both turns conditioning on for that variable (if it was off) and plants the conditioned point exactly at the cursor — the value you clicked becomes the pinned target for that quarter. (Left-clicking the "condition" button toggles conditioning on without moving any point.) This is the fastest way to drop a single pin at a specific value.

    Setting the conditioned path¶

    Once a variable is conditioned you can set its path in three ways:

    1. Drag directly on the chart. The forecast portion of the chart (quarters 0-19) shows draggable markers. Click and drag any marker vertically to set the value for that quarter. Dragging is continuous -- hold the mouse button and sweep across quarters to draw an entire path quickly. The marker (the conditioned dot) snaps to your click instantly — it tracks the value you set without waiting for the forecast to recompute (see The optimistic pin below).

    2. Right-click to plant a pin at the cursor (above) — drops a single conditioned point exactly where you click.

    3. Use the sliders in the control panel. Each conditioned variable appears in the Scenario section of the left sidebar with 20 quarter-level sliders (labeled NC, t+1, t+2, ... , t+19). Each slider ranges from -3 to +3 (percentage-point deviation from baseline). The numeric value is shown to the right of each slider.

    The optimistic pin¶

    When you click, drag, or right-click to set a conditioned point, the dot snaps to your chosen value immediately — it is read straight from the target you just set, not from the model solve. The smoothed forecast line and the uncertainty bands then catch up a moment later, once the conditional forecast finishes computing, and reconcile around the pinned point. So the pin always sits exactly where you put it, even while the rest of the curve is still updating. (On a GVAR economy the cross-country solve is heavier, so the line/band reconciliation can take a beat longer — the dot still snaps instantly.)

    Levels vs. deviations¶

    A toggle in the control panel ("Show deviations" / "Show levels") switches the numeric display between:

    • Levels -- the absolute value (baseline + deviation).
    • Deviations -- the signed deviation from baseline (e.g., +0.5).

    The underlying data is always stored as deviations; this toggle only affects how values are displayed in the sidebar and exported.

    Per-quarter toggles¶

    Each quarter label (NC, t+1, ...) is a clickable button. Click it to disable conditioning for that single quarter -- the label grays out and the slider becomes inactive. The BVAR will treat that quarter as unconditioned, allowing it to be determined freely by the model dynamics. Click again to re-enable. The "All" button above the sliders toggles all 20 quarters on or off at once.

    Removing a conditioned variable¶

    Click "remove" next to the variable name in the sidebar, or click the "CONDITIONED" button on the chart.

    Resetting everything¶

    The Reset link at the top of the Scenario section clears all conditioning paths, rate adjustments, and conditioned variables back to baseline.

    4. Reading the Charts¶

    The chart grid is arranged in a 3-column grid, top to bottom, in this order: the macro triad (row 1), the estimated fiscal drivers (row 2, where present), the derived fiscal block (row 3, where present), then the remaining domestic variables, and finally the cross-country blocks (Rest-of-World and Global commons) for GVAR economies.

    Row 1 -- macro triad¶

    Chart Variable Draggable
    Inflation Headline CPI/HICP/GDP deflator Yes (when conditioned)
    Policy Rate Short-term interest rate Yes (when conditioned, or in counterfactual mode)
    GDP Growth Real GDP growth Yes (when conditioned)

    For euro-area members (DE, FR, IT, …) the first row also carries the ECB Policy Rate, the EA Long Yield (Bund), and the Sovereign Spread (vs Bund). The ECB rate is common -- it reads the same across the EA aggregate and every member, and conditioning it anywhere is an area-wide shock. Conditioning a member's sovereign spread is a country-specific stress that spills to the rest of the area by GDP weight. A US or EA scenario reaches the members when you press Solve full GVAR. See Euro-area Spillover & Harmonization for the full picture.

    Row 2 -- estimated fiscal drivers (US and the GVAR fiscal blocs)¶

    Revenue (% of GDP), Primary Expenditure (% of GDP), and the Effective Rate on debt (% p.a.) -- the three estimated drivers of the fiscal accounting. Shown only for economies that carry a fiscal block.

    Row 3 -- Fiscal block (derived by accounting identity)¶

    Primary Balance, Δ Debt-to-GDP, and Debt-to-GDP -- derived by rolling the budget identity forward from the row-2 drivers (debt is not estimated directly). This block now sits immediately under the fiscal drivers so the estimated inputs and their accounting outputs read together. Primary Balance and Debt-to-GDP are conditionable on a trajectory (right-click to pin, drag to set a target); Δ Debt-to-GDP is read-only. See The US Fiscal Block below.

    Remaining domestic variables¶

    Gov. Spending, Core Inflation, Unemployment, Consumption, Investment, and any country-specific series -- all conditionable via the right-click button but not pre-wired for dragging.

    Levels vs. growth display¶

    Most series are shown as q/q annualized growth (for log variables) or as a level (for rates and ratios). A handful of index/price series are shown as price levels instead of growth -- including the oil price ($/bbl), the US equity index, exchange-rate indices and house prices.

    Trace colors and styles¶

    • Gray dotted line -- BVAR unconditional baseline (no conditioning applied).
    • Gray shaded band -- Baseline 16th-84th percentile credible interval (always shown).
    • Blue solid line -- Active scenario (the conditional forecast, i.e., baseline + your conditioning). The first scenario uses blue (#0067B9); additional scenarios use orange-red, green, and purple.
    • Red/orange dashed line -- Policy counterfactual (only appears when counterfactual mode is on and a rate adjustment is applied). Shown with diamond markers on the interest-rate chart.
    • Light colored band around scenario line -- Scenario-specific uncertainty (only shown after clicking Compute; see Section 6).

    Light-shaded history region¶

    A faint blue-gray rectangle marks the last 3 quarters of actual historical data to the left of quarter 0. Most economies' panels now end in 2026 Q1 (the observed data was refreshed one quarter past the 2025 Q4 estimation vintage, with no re-estimation), so the first forecast — or "nowcast" — quarter is 2026 Q2. A few blocs have a ragged edge where genuine source data lags and is never fabricated forward: China and India end 2025 Q4 and Israel ends 2024 Q4, so each of those begins forecasting from the quarter right after its own last observation. The x-axis is dated by calendar quarter on hover, with year ticks.

    Growth rate display¶

    Above the chart grid, a toggle lets you switch growth-rate variables between q/q ann. (quarter-on-quarter annualized) and y/y (year-on-year).

    Variable definition¶

    Below each chart, a small label shows the BVAR variable ID, name, and transformation (e.g., "JGDP -- Inflation (q/q ann. growth)").

    5. Policy Counterfactual¶

    The counterfactual layer answers: "If the central bank had set rates differently from what the BVAR scenario implies, what would have happened to the rest of the economy?"

    Enabling counterfactual mode¶

    In the control panel, under Policy Counterfactual, click "Policy counterfactual". The button turns orange to indicate the mode is active. Click "Exit counterfactual mode" to return to pure BVAR conditioning.

    Choosing a DSGE model¶

    When counterfactual mode is active, a DSGE Model section appears. It shows the currently selected model (default: US_SW07). Click "Browse..." to open a searchable catalog of models from the Macroeconomic Model Database, organized by category. You can type to filter by model name, author, or reference. Select a model to load its impulse-response functions.

    Choosing a policy rule¶

    Below the model selector, a Policy rule dropdown lists all available rules (e.g., Taylor, GR, CEE, SW). Rules that have pre-computed IRFs for the selected model are enabled; others are shown as "(no IRFs)" and are disabled.

    Rule-based vs. Optimal¶

    Two small buttons switch between:

    • Rule-based -- You directly parameterize a Taylor-type rule with three sliders:
    • Smoothing (rho): 0 to 0.95 (default 0.8). Degree of interest-rate inertia.
    • Inflation coefficient (phi_pi): 0 to 4 (default 1.5). Response to the inflation gap.
    • Output gap coefficient (phi_y): 0 to 3 (default 0.5). Response to the output gap.

    The rule formula is displayed: r = rho * r(-1) + (1 - rho) * [r + phi_pi * (pi - pi) + phi_y * y].

    • Optimal -- Barnichon and Mesters (2023) optimal policy. A single slider trades off between the two mandates: slide left to weight price stability (inflation at 2%), slide right to weight full employment.

    Macro response modes¶

    The Macro response dropdown controls how the counterfactual interest-rate path feeds back into other variables:

    Mode Behavior
    Hold conditioned paths fixed Your conditioned variables stay exactly where you set them. Only unconditioned variables respond to the rate change. Use this to ask: "given my macro scenario, what would a different rate path imply for unconditioned variables?"
    IRF-consistent (scenario + DSGE deltas) The tool computes the interest-rate shock sequence needed to match your rate adjustment, applies the DSGE model's impulse-response functions to translate those shocks into effects on all variables, then re-conditions the BVAR to produce an internally consistent path. This is the default and most complete mode. If no IRFs are available for the chosen model/rule pair, it falls back to BVAR-only re-conditioning (a warning is displayed).
    Let all variables adjust All variables -- including ones you previously conditioned -- are free to respond to the altered rate path. Your conditioning is released.

    Rate adjustment sliders¶

    Below the macro-response dropdown, 12 quarter-level sliders (Q1-Q12) let you specify the interest-rate deviation from the scenario path, in percentage points, ranging from -2 to +2 in 0.05 pp increments. You can also drag the interest-rate line directly on the Policy Rate chart when counterfactual mode is active -- the chart shows diamond markers for the counterfactual rate.

    Fiscal adjustment sliders¶

    Below the monetary rate adjustment, a second set of 12 sliders controls autonomous government spending shocks, in percentage-point deviations from the scenario path (-3 to +3 in 0.1 pp steps). Fiscal counterfactuals use the fiscal_ impulse response functions from the selected DSGE model.

    Not all DSGE models include fiscal shocks. 60 of the 132 available models have fiscal_ IRFs (including US_SW07, EA_SW03, EA_QUEST3, US_CMR14noFA). If the selected model lacks fiscal IRFs, the fiscal sliders will have no effect on the counterfactual.

    Monetary and fiscal adjustments can be combined: the tool computes separate shock sequences for each instrument, then sums the DSGE-implied deltas on inflation, output, and the interest rate before re-conditioning the BVAR.

    6. Uncertainty Bands¶

    Modes¶

    The top bar offers three uncertainty settings:

    • None -- No uncertainty computation. Only the baseline credible interval is shown (gray band).
    • Quick -- Draws 50 posterior samples for a fast approximation of the scenario-conditional uncertainty.
    • Full -- Draws 200 posterior samples for a more precise estimate.

    Computing¶

    Select Quick or Full, then click the Compute button. A loading overlay ("Computing scenario uncertainty (50/200 draws)...") appears while the tool runs the Durbin-Koopman simulation smoother across posterior draws. When finished, colored uncertainty bands appear around the active scenario line on every chart.

    When bands reset¶

    Scenario uncertainty bands are cleared whenever you switch the active scenario, change the country, or change the uncertainty mode. You must click Compute again to regenerate them. Modifying conditioning paths does not automatically recompute bands.

    7. Multiple Scenarios¶

    Adding a scenario¶

    Click "+ Add Scenario" in the bottom scenario bar. Up to 4 scenarios can exist simultaneously. Each is assigned a distinct color (blue, orange-red, green, purple).

    Switching between scenarios¶

    Click a scenario tab in the bottom bar to make it active. The control panel updates to show that scenario's conditioning paths, model, rule, and counterfactual settings. Only the active scenario is fully computed with the DK smoother; inactive scenarios show approximate paths.

    Removing a scenario¶

    Click the "x" on a scenario tab. At least one scenario must remain.

    Visibility¶

    Each scenario's line is plotted on the charts as long as it is visible. All visible scenarios appear simultaneously so you can compare conditioning assumptions side by side.

    8. Export and Import¶

    Per-scenario Save (Control Panel)¶

    Click Save in the Scenario section header. This exports the active scenario as an Excel file named:

    scenario_{COUNTRY}_{ScenarioName}_{timestamp}.xlsx

    For example: scenario_US_Scenario_1_2026-05-30T14-22-05.xlsx.

    Per-scenario Load (Control Panel)¶

    Click Load and select a file. Accepts:

    • .xlsx / .xls -- Restores all settings, conditioning paths, active quarters, rate adjustments, and (if saved) the country. If the file was saved for a different country, the tool switches countries first.
    • .json -- Legacy format. If the JSON contains a scenarios array (full-session export), the first scenario is restored.

    Full-session Export (Top Bar)¶

    The Export button in the top-right corner of the top bar exports all scenarios at once into a single Excel file:

    calibration_{COUNTRY}_{timestamp}.xlsx

    Each scenario becomes a separate sheet.

    9. Excel File Format¶

    Per-scenario export (Save button) -- 5 sheets¶

    Sheet Contents
    Settings Key-value pairs: scenario name, country, model, rule, mode (rule-based/optimal), all policy-rule parameters (rho, phiPi, phiY, lambdaPi, lambdaY, piStar, yStar), showCounterfactual flag, counterfactualMode, input format (levels/deviations), export timestamp.
    Conditioning One column per conditioned variable, 20 rows (NC, t+1 ... t+19). Values are in whatever format was active at export time (levels or deviations). Inactive quarters are left blank.
    Active Quarters Same layout as Conditioning but with 1/0 flags indicating which quarters are active for each variable.
    Rate Adjustment 20 rows of rate adjustment values in percentage points.
    Variable Map Maps column names to BVAR variable IDs and indices so the file can be re-imported even if the BVAR specification changes. Columns: Column (name), Variable Name, Variable ID, BVAR Index.

    Data export (bottom bar) -- 4 formats¶

    Button Format Contents
    CSV .csv All variables: Baseline, Scenario, Deviation columns per variable. Includes 3 quarters of history.
    Excel .xlsx One sheet per variable with Q16, Baseline (Median), Q84, Scenario, Deviation columns side by side. Plus a Summary sheet.
    PNG .zip All charts as high-resolution PNG images (1200x500) bundled in a single ZIP file. Each chart includes the scenario name in the title.
    PDF .pdf All charts combined in a single multi-page PDF (one chart per page, landscape orientation).

    Full-session export (Export button) -- 1 sheet per scenario¶

    Each sheet contains a header block (country, timestamp), a settings block (same key-value pairs), a conditioning table, and a rate-adjustment table, all in a single vertically stacked layout.

    10. Scenario Cookbook and Examples¶

    Scenario Cookbook (instant-load recipes)¶

    The Cookbook ▾ button in the scenario header (next to Save / Load) opens an instant-load library of named, documented conditioning recipes — it replaces the old download-only Example link. Pick the bloc (US, EA, or CN), open the cookbook, expand a recipe to read its rationale, and load its conditioning into the active scenario with one click. No download or re-import needed.

    Every recipe is built around a deliberate two-stage pedagogy:

    • Load naive — condition the head variable(s) alone and let the model fill in the rest. The result is the model's own read, which is usually demand-led and often the economically wrong story — the point is to expose it.
    • Load + restriction — add a few short-run pins that restrict the joint response into the meaningful shock (e.g. force output down while prices rise → a supply shock). The contrast between the two stages — often a sign flip — is the entire lesson.

    The nine recipes are: (1) credit crunch / spread shock, (2) energy supply shock, (3) fiscal loosening → twin deficits, (4) monetary behind-the-curve / risk-off, (5) AI productivity boost, (6) supply-chain collapse, (7) deglobalization, (8) inflation de-anchoring, (9) fiscal default risk. Loading a recipe replaces the active scenario's conditioning (like an Excel import); any recipe variable not present in the current bloc is skipped and reported in a short note. Recipes write the active bloc's pins only — press Solve full GVAR afterwards for the cross-country answer. See the full per-recipe write-up (rationale, naive vs restricted pins, country-specific notes) in the Scenario Cookbook methodology page.

    Downloadable example scenario¶

    The app also still ships a built-in example scenario (example_scenario_inflation_surge.xlsx) for the Load workflow, available from public/data/example_scenario_inflation_surge.xlsx.

    What it contains¶

    The scenario models a severe inflation surge in the US -- inflation hitting 10% q/q annualized -- with an aggressive monetary policy tightening response based on the Smets-Wouters (2007) estimated rule:

    Conditioning (Inflation, JGDP): - +7.7 pp deviation in the nowcast quarter (NC), pushing inflation from ~2.3% to ~10% - Concave decay over 16 quarters (4 years) back to baseline - Quarters t+16 through t+19 are unconditioned (the BVAR fills them freely)

    Policy counterfactual (Rate Adjustment): - +300 bp rate adjustment in quarters NC through t+3 - +200 bp in quarters t+4 through t+7 - +100 bp in quarters t+8 through t+11 - 0 thereafter (policy reverts to the scenario path)

    Settings: - Model: US_SW07 (Smets-Wouters 2007) - Rule: SW (Smets-Wouters estimated rule, rho=0.81, phi_pi=2.04, phi_y=0.08) - Counterfactual mode: IRF-consistent

    How to use it¶

    1. Click Load in the Scenario section of the control panel
    2. Select the downloaded example_scenario_inflation_surge.xlsx
    3. The tool will restore all conditioning paths, rate adjustments, and model/rule settings
    4. The charts will immediately show the conditioned scenario (blue line diverging from baseline)
    5. Enable the Policy Counterfactual toggle to see the tightening response (red dashed line)

    Creating your own scenario files¶

    You can create scenario Excel files manually or programmatically. The minimum required structure is:

    • Settings sheet: at least Country and Name rows
    • Conditioning sheet: Quarter column + one column per conditioned variable, with deviation values

    The Active Quarters and Variable Map sheets are optional but improve import reliability. If omitted, the importer infers active quarters from non-empty cells and matches variable names against the BVAR specification.

    11. Keyboard and Interaction Tips¶

    • Drag on charts: Click and hold on the forecast region of any conditioned variable's chart to set values by dragging. The cursor changes to a crosshair over draggable areas. A small badge reading "drag to set path" (or "drag to adjust policy" for the rate chart) confirms the chart is interactive.

    • Sweeping paths: Hold the mouse button down and move horizontally across quarters to draw an entire path in one gesture rather than clicking quarter by quarter.

    • Quick conditioning workflow: Click "condition" on the chart, then immediately drag to set the path. This is faster than using the sidebar sliders for exploratory work.

    • Right-click to drop a pin at the cursor: Right-clicking on a forecast chart conditions that variable (if needed) and plants the conditioned point exactly where you clicked — a one-gesture way to set a single target value. The dot snaps to the cursor instantly.

    • Fine-tuning with sliders: After dragging a rough path on the chart, switch to the sidebar sliders for precise numeric adjustments (0.1 pp resolution).

    • Per-quarter on/off: Use the quarter labels in the sidebar as toggles to "punch holes" in your conditioning -- for example, condition quarters 1-4 and 8-12 but leave 5-7 unconditioned to let the model interpolate freely.

    • Compare scenarios quickly: Add a second scenario, set different conditioning, and both lines appear on every chart simultaneously.

    • Hover for values: Hover over any point on any trace to see its numeric value in a tooltip (formatted to 2 decimal places).

    • Save early, save often: The Save button is instant and timestamped. Export your scenario before making large changes so you can Load it back if needed.

    • Growth rate toggle: Switch between q/q annualized and y/y above the chart grid depending on which convention your analysis requires.

    12. The US Fiscal Block¶

    When the country is the United States, a Fiscal block (derived) group appears below the standard charts. It implements public-finance projection the way the CBO and IMF financial-programming frameworks do: the BVAR estimates the fiscal drivers and the reporting variables are derived from them by an accounting identity, never freely estimated.

    Estimated drivers (conditionable). Two fiscal driver charts behave like any other variable -- you can condition them by dragging or with the sidebar sliders:

    • Government revenue -- general-government current receipts as a ratio to GDP.
    • Primary expenditure -- government primary (non-interest) spending as a ratio to GDP.

    Derived reporting variables (not draggable). These are computed from the drivers plus the macro forecast by the budget-identity recursion, and are labelled as derived:

    • Primary balance -- revenue minus primary expenditure (positive = surplus).
    • Debt-to-GDP -- accumulated from the change in the debt ratio, seeded at the latest observed level. The path responds to the primary balance and to the snowball term (the gap between the effective interest rate and nominal GDP growth).

    Because the reporting variables are derived, conditioning a driver flows straight through to them: impose a tighter revenue path and the primary balance improves and the debt-to-GDP path bends down accordingly. The effective interest rate is backed out of the budget identity so the stock and the flows always stay mutually consistent; for that reason the effective-rate, interest (debt-service), and overall-deficit charts are intentionally not shown (under the change-in-debt/GDP parametrization they collapse to a residual close to nominal growth and are not independently informative).

    13. Global VAR Cross-Country Linkage¶

    33 economies are linked into a Global VAR (GVAR) layer — the United States plus 32 other blocs (the Euro Area as a single aggregate plus 31 Rest-of-World blocs: Japan, the UK, Canada, China, and a wide set of OECD and emerging-market economies). Together they cover roughly 90% of global demand. Each bloc carries a Rest-of-World chart group of three trade-weighted "star" variables -- foreign demand, foreign inflation, and foreign policy/financial conditions -- that summarize its trading partners, plus a shared set of global commons (oil and US financial conditions) common to every bloc. Unlike earlier versions where the US was a closed anchor, the US is now a fully endogenous bloc: it carries its own Rest-of-World stars, so it both drives the rest of the world and receives spillback feedback from it.

    What this means in practice. Condition a path on any GVAR economy (for example a US policy-rate or GDP path), then switch to another linked economy. Its forecast already embeds the spillover: the tool solves the whole set of countries as a single cross-country system, so the shock flows from one bloc to the others through their Rest-of-World blocks (and the shared oil / US-financial-conditions commons), with the responses feeding back on each other until the system settles.

    Global commons (oil and US financial conditions). Below the Rest-of-World block each GVAR bloc shows a Global commons chart group: the oil price (WTI, plotted as a price level in $/bbl), the US 10-year Treasury yield, the US BAA corporate yield, and the US equity index. These four are US-determined: the United States carries them as its own variables, and every other bloc receives the same path. You set them on the United States screen — condition the oil price (or any of them) there, and the change is carried over identically to every other economy, where the chart is read-only and labelled carried over from the US path. Each bloc's domestic variables then respond to the common shock through that bloc's own estimated relationships. Economically this captures oil and US financial conditions as genuinely global drivers: a US-set oil-price scenario moves activity, inflation and rates in every linked economy at once.

    Why it stays fast, and where the "Solve full GVAR" button comes in. At 33 blocs — with the US now endogenous — the full cross-country solve is large (it takes a few seconds), so the tool keeps the drag itself instant and defers the heavy solve to an explicit button:

    • While you drag, nothing heavy runs. Dragging or planting a pin updates the chart from a cheap local preview — the conditioned dot snaps to your value immediately (see The optimistic pin) and the drag stays smooth at the screen's frame rate. No cross-country solve fires during the drag.
    • On release, one fast preview solve fires. When you let go of the mouse, the tool runs a single cheap "direct" preview — the first-order, one-hop cross-country channels — so the lines settle to a reasonable propagated path.
    • Click "Solve full GVAR" for the complete answer. A "Solve full GVAR" button (and the top-bar Compute) runs the complete cross-country fixed point — every feedback loop, including the new Rest-of-World → US spillback — plus the uncertainty bands. This is the heavy step, so it is gated behind the button rather than fired on every drag. A small pill shows whether you are viewing the fast DIRECT preview or the full solve, and a "computing…" badge appears while the full solve runs.

    The UI also shows a small read-out (sweeps, the spectral radius, and the solve time in milliseconds).

    Uncertainty in GVAR mode. By default the scenario uncertainty band is the model's own (unconditional) forecast density centred on the propagated (spillover-adjusted) scenario path — it keeps the full forecast width and sits around the displayed line rather than on the baseline. If you turn on Experimental features (the toggle in the lower-left corner), the band instead re-solves the whole cross-country system for every posterior draw — the conditional cloud, which pinches tightly at the points you have pinned and is slower to compute. The default is recommended for most uses; the experimental mode is there if you specifically want the conditional pinching behaviour.

    The United States is now a fully endogenous member of the system, not a closed anchor: it carries its own Rest-of-World block (three trade-weighted stars built from a US trade-weights row), so it drives the rest of the system and is driven back by it. Conditioning another economy therefore feeds back onto the US forecast through the full "Solve full GVAR" solution (the fast direct preview shows only the one-hop channels).

    The two cross-border spillover switches¶

    On a GVAR-linked bloc the sidebar carries a Cross-border spillovers section with two independent switches, both off by default. They act through different mechanisms and control different things — keep them straight:

    • Scenario spillovers (GVAR solve) — store flag gvarScenarioSpillover. Controls whether a conditioning scenario (the pins you drag) on one economy propagates to its trade partners through the live coupled GVAR solve (the fixed point x = M·x + d). Off (default): the conditioned forecast stays local — trade partners are frozen — which keeps dragging fast; press Solve full GVAR when you want the complete cross-country answer. On: the cross-country preview engages on every pin edit.
    • Long-run target spillovers (GVAR constant shifts) — store flag spilloverEnabled. Controls whether the long-run-target shifts (the δc drift setter of Section 14) propagate cross-country through the precomputed (I − M)⁻¹ coupling (no live solve). Off (default): a long-run target moves only its own economy (the viewed bloc's own target always applies locally). On: every active bloc's long-run target is injected into the GVAR and propagated to the rest of the world. In both modes a target left at the model estimate (δc = 0) is an exact no-op.

    In short: the first switch propagates scenario pins via the solve; the second propagates long-run constants via the precomputed inverse. Both persist across a country switch and reset to off with Reset all.

    Note (naming). These are the precise mechanism names. The in-app captions currently read "Long-run target spillovers (GVAR constants)" for the scenario/solve switch and "Long-run trend spillovers" for the long-run constant switch; a caption rename to match the names above is flagged as a follow-up. The mechanisms behave exactly as described here.

    14. Long-Run Targets (per-variable drift sliders)¶

    In the control panel, the Long-run targets section lets you set a variable's long-run (≈H-quarter, the chart horizon) forecast to a target value in the chart's display units. Every forecastable variable has its own slider with a ±0.1 number input; drag the slider or type a target. The change is instant and exact — there is no Compute, no re-estimation, and no draw reweighting. Each slider defaults to the model's own current terminal forecast, and its arrows step ±0.1 from that estimate, so an unmoved slider (or one snapped back to the estimate) is an exact no-op (it changes nothing).

    What it does, in one sentence. It shifts the model's drift (the companion constant) for the chosen variable's own equation by exactly the amount needed for that variable's terminal forecast to land on your target, and lets the estimated model dynamics drag every other variable along coherently. Mechanically the deviation is ΔY = S_h[:, j]·δc_j, where S_h is the model's own partial companion-power sum (computed once in the browser from the shipped companion) and δc_j is the single drift shift that hits your target. Setting several targets superposes linearly.

    This is the tool for imposing a structural long-run view that per-quarter conditioning cannot easily express — anchoring long-run inflation to 2%, or trend real growth to a target — without re-estimating. Because the shift is added identically to the central path and to every confidence band (q05/q16/q50/q84/q95), the baseline and scenario fans translate rigidly: the band widths are unchanged and the bands never cross. The targets persist until you remove them; the panel badge reads N targets · exact because the terminal forecast hits the target by construction (the achieved value equals the target to numerical precision).

    This drift setter replaces the earlier entropic-tilt long-run mechanism, which reweighted the posterior draws toward the target. The tilt was correct in principle but degenerated in practice — as a target was pushed away from the draw cloud the effective sample size collapsed and the pinned band rested on a handful of draws. The drift setter is deterministic and exact at any target, so it never degenerates, and it is instant (no draw loop to reweight).

    Read the labels. - This sets the long-run drift, not persistence — it moves where the path glides to, not how fast it gets there. - The cross-variable response (other variables moving when you set one target) is a real model side effect, not a bug — it is the model's own estimated dynamics responding to the shifted drift. - The real-growth target owns the debt direction (it drives the (i−g) snowball in the fiscal block). It is labelled distinctly. It is not a structural debt fix — it changes the growth assumption that the debt path inherits.

    Near-singular variables are disabled. For a few variables the model's terminal sensitivity to a drift shift is too small to invert reliably (a tiny δc would imply an enormous, ill-posed move). Those sliders are disabled (greyed out, with a "near-singular response" tooltip) and cannot be targeted. This is the |sensitivity| < 0.3 guard; it is expected behaviour, not a fault — pick a better-behaved variable (for example target growth rather than a near-unit-root level).

    Availability and the GVAR caveat. Long-run targets apply to every model — single-country and GVAR alike, and are computed entirely from each model's own shipped companion. On a GVAR bloc the target is marginal: it shifts that bloc's own long-run forecast, but the trade partners and the US-determined global commons are not re-solved, so the realised joint long-run value may differ from the target. The control panel shows this amber caveat explicitly when a GVAR bloc is selected.

    Model-based vs. with-judgement (per variable). Every variable is, by default, model-based: its slider sits at the model's own terminal estimate (the est reference shown next to it), so its forecast is exactly what the BVAR/GVAR produces with no user input. The moment you move a slider — or apply the consensus preset below — that variable becomes with-judgement: its target is highlighted, a ↺ reset-to-estimate control appears next to it, and the panel footer badge reads N targets · exact. The distinction is per-country and per-variable: within a single chart grid some series can be left purely model-based while others carry your judgement, and switching country resets every variable back to model-based (targets reference the old model's variables, so they are cleared on a country switch). Pressing ↺ on a row, or reset all to estimate, returns the chosen variable(s) to model-based with no residue (the default is the model estimate, so a cleared slider is byte-exact).

    Consensus anchors (optional preset, OFF by default). When a bloc has a set of publicly-defensible long-run values, a Consensus anchors panel appears at the top of the section with an Apply consensus anchors button. It is opt-in and off by default — the raw model estimate stays the default view until you click it. The preset loads public-knowledge defaults for the bloc's well-conditioned variables: the central bank's inflation target (or band mid-point), potential growth (IMF-WEO / OECD), a neutral policy rate (r* + π*), and structural/NAIRU unemployment. It deliberately omits any variable whose slider is near-singular/perverse (for example the BE/CA/CZ policy-rate sliders), so the one-click preset never ships an unreliable thumb. Once applied, the button shows ✓ Consensus applied; Clear (or reset all to estimate) reverts every variable to model-based. These values are a transparent starting point for discussion, not an IMF forecast — they are the same anchors documented in docs/pre-ship-smoke-test.md §2 and shipped in app/public/data/default_anchors.json.

    Note (policy-rate row). The policy rate is an ordinary variable of the model, so it carries its own long-run-target slider like every other series — set its terminal value directly. There is no separate Taylor-rule rate overlay; the rate's long run is governed solely by its own slider.
