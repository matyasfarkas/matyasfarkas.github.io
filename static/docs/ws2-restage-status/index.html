    Workstream-2 Re-stage Status — Genuine-Only Re-source of CN/KR/ID/TH/IN + RU-Drop Plan¶

    Date 2026-06-25. STAGING + PLANNING ONLY — no production changes. This document captures the state after the genuine-only re-source of the 5 contaminated EM blocs the user chose to KEEP (CN, KR, ID, TH, IN) and the plan to DROP RU. It is the bridge between the read-only feasibility (docs/workstream2-feasibility.md) / audit (docs/synthetic-data-audit.md) and an eventual re-estimation + ship.

    Hard constraints honoured (NO-SYNTHETIC is ABSOLUTE): every series was re-derived from its GENUINE upstream source; any quarter that could not be genuinely sourced is left honestly null and flagged needs-human-check — NO WEO-fill, held-last, or interpolation anywhere. All staged output is under pipeline/ws2_staging/<C>/ (gitignored, line 48 of .gitignore). app/public/data is untouched, no MATLAB was run, no coupling was regenerated, nothing was committed.

    1. RU-DROP CHECKLIST (enumerated; DO NOT APPLY)¶

    The user chose to DROP RU (genuine macro core stuck at 2024-Q1; no FRED activity mirror; reaching current needs Rosstat/CBR/FCS connectors the pipeline lacks — see feasibility §RU). Removing RU from the live system is the following precise, exhaustive set of edits. Nothing here has been applied.

    1.2 Trade-weight files (the operative bloc-list + the star renormalisation)¶

    RU is BOTH a bloc (origin row) and a partner (column) in the weight matrices. It must be removed as a bloc entry, as an origin key, and as a partner key under EVERY other origin, then the remaining rows must be renormalised (each origin's partner weights must re-sum to 1 over the surviving partners) — otherwise the GVAR stars are mis-weighted.

    1. app/public/data/bvar/gvar/weights_24bloc.json (THE LIVE FILE — gvarSolve.ts imports it):
    2. remove "RU" from the blocs array (currently 33 entries → 32).
    3. remove "RU" from bloc_codes ("RU": "RUS").
    4. remove the "RU" origin key from weights (RU's own partner row).
    5. remove "RU" as a partner key from all 32 other origins' weight dicts (verified: RU appears as a partner under 32 origins).

    6. RENORMALISE each of the 32 surviving origins' partner weights to sum to 1 after RU's column is dropped. (Functionally equivalent to what buildstars does at estimation time when a partner is absent — but the shipped weights file must be made consistent for the in-browser solver.)

    7. app/public/data/bvar/gvar/weights_full.json (full 33-bloc reference; not the live import but shipped + used by tests/topology checks): same 5 sub-edits + renormalise. RU appears 35× (blocs array, bloc_codes, origin key, 32 partner cols).

    8. app/public/data/bvar/gvar/weights_49.json (50-bloc superset; RU 50×): same removal + renormalise IF this file is ever promoted to live; otherwise drop RU for consistency.

    9. Back-ups (optional, for hygiene — not load-bearing): app/public/data/bvar/gvar/weights_24bloc.json.bak24, app/public/data/bvar/gvar/weights_full_50bloc_backup.json — leave as historical backups OR strip RU to avoid a future accidental restore re-introducing RU. Recommend: leave untouched, document.

    2. RE-ESTIMATION READINESS — per bloc (CN/KR/ID/TH/IN)¶

    The production estimator cgl_bvarV5.m handles ragged ends NATIVELY: lastFull = find(~isnan(sum(x,2)),1,'last') (line 39) → bvarGLPmf fits GLP coefficients only on xx(1:lastFull,:); remNaNs_prev method 6 + the DK simulation smoother impute trailing NaNs (same mechanism as the COVID mask). So a genuine-then-NaN ragged tail is estimated exactly as single-country to the last fully-observed row.

    CRITICAL READINESS CAVEAT (load-bearing). lastFull is the last row with NO NaN in ANY column. A staged panel where the derived columns (the 3 ROW* stars, SFA) are entirely null has lastFull =(first all-null row) ≈ panel start → the GLP fit would collapse. Therefore none of the 5 staged panels is directly feed-ready as-is: the derived columns must be populated first. Two of them are populated only by the GVAR-EM estimator itself:

    • ROWDEM/ROWINF/ROWFIN_<C> (all 5 blocs): trade-weighted partner stars. NOT standalone-sourceable; export_bvar_gvar24_loop.m buildstars computes them over the genuine partner panels DURING the EM loop. They are SUPPOSED to be empty in the staged per-bloc panel — the loop fills them. So the staged panel feeds the loop (which builds stars), not cgl_bvarV5 directly.
    • SFA (CN; analogous fiscal residual logic): a recomputed debt-recursion residual (fiscalAccounting.ts computeFiscalBlock), produced app-side from the genuine fiscal inputs + observed debt/GDP. Not a pulled column. CN's genuine quarterly gross-debt input (BIS FRED QCNGANXDCA to 2025-Q4) is recorded in provenance_CN.json for that recomputation.

    Net readiness model: the staged panels are the GENUINE DOMESTIC + GLOBAL columns ready to feed the GVAR-EM loop (export_bvar_gvar24_loop.m), which then (i) builds the stars and (ii) calls cgl_bvarV5 per bloc on the star-augmented panel. They are NOT a drop-in for a bare cgl_bvarV5 call (the stars would be missing). This matches the existing pipeline contract — build_gvar_2025q4.py produces DataRaw_<C>_glob.json WITHOUT stars and the loop adds them.
