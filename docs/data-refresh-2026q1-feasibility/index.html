    2026Q1 Data-Refresh Feasibility (no re-estimation)¶

    Date of check: 2026-06-22 · Branch: improve-live · Mode: READ-ONLY availability probe (no data/code edited, nothing committed).

    Goal recap¶

    Extend the OBSERVED panel by one quarter (2025Q4 → 2026Q1) and let the forecast re-anchor from the new last-observed state. The model companion (companion_mean / companion_draws) stays fixed — same post-estimation philosophy as the δc long-run sliders. This note only answers: is 2026Q1 published yet, source-by-source, economy-by-economy?

    What the current panel looks like¶

    • Estimated/shipped BVAR panels (app/public/data/bvar/<C>/historical.json) end at 2025Q4 (n_periods=40, US). Dates aren't stored in the shipped matrices; anchor comes from build_gvar_2025q4.py (common 2025Q4 endpoint).
    • Committed raw JSONs (pipeline/country_rawdata/DataRaw_<CC>.json) are already ragged: CA & JP at 2026Q1, EA/UK/BR/MX/ZA at 2025Q4, CN & IN only at 2025Q2 (these EMs lag and are back-filled to 2025Q4 via WEO-annual + Denton disaggregation in build_gvar_2025q4.py — i.e. already partly synthetic at quarterly freq).

    Sources & access (verified live)¶

    • FRED — key 8ce324fcfb4663bfef63d3d44fbdbc73 (hardcoded in pipeline/build_country_data.py:19, build_fiscal_us_gg.py, build_credit_*.py, etc.). All US series + EA HICP + several EM real-GDP series resolve here.
    • Eurostat SDMX 2.1 (build_oecd_eurostat.py): namq_10_gdp real GDP, prc_hicp_midx HICP, etc. — used for EU/EFTA OECD members' real GDP.
    • Note: EA-aggregate HICP is sourced from FRED (CP0000EZ19M086NEST), not the Eurostat HICP endpoint (build_ea_eurostat.py:27).

    Caveats found¶

    1. Eurostat HICP endpoint is stale in the direct API (ends 2025-12) — but it's a non-issue: EA HICP is pulled from FRED (CP0000EZ19M086NEST, current to 2026-05). For euro-member CPI inputs, confirm each member build's price source isn't the stale Eurostat HICP path.
    2. Discontinued OECD-on-FRED CPI mirrors: GBRCPIALLMINMEI, CPALTT01CAM659N both end 2025-03 (discontinued) — these are NOT the pipeline's UK/CA price sources, but a refresh script must avoid them.
    3. CN / IN are the structural laggards: real source data ends 2025Q2; the panel only reaches 2025Q4 via WEO-annual disaggregation. A genuine 2026Q1 observed point is not available today; refreshing them would mean another round of back-fill, not real data.
    4. FREV / FPEXP are internal variable names, not FRED IDs — the real FRED fiscal IDs are W066…/W068…/A180…/GFDEGDQ188S (all ✅).
    5. Rates/CPI are monthly and already run into 2026-04/05; quarterly aggregation for 2026Q1 (Jan–Mar) is complete.

    Bottom line — realistic 'as-of' for a refresh TODAY¶

    A clean, no-re-estimation refresh to 2026Q1 is feasible for the advanced economies — US (incl. all fiscal), the euro area + EU/EFTA members (real GDP from Eurostat, HICP from FRED), UK, JP, CA, and the FRED-sourced EMs (BR, MX, ZA). For these, 2026Q1 macro is published and the forecast can simply re-anchor from the new last-observed state.

    The binding constraint is the EM laggards (CN, IN) whose real source data still ends ~2025Q2 and which only reach 2025Q4 through WEO-annual disaggregation — no genuine 2026Q1 observation exists for them today.

    Recommended as-of quarter: 2026Q1 for the AE + FRED-EM blocs (the bulk of the system), while CN/IN remain at their back-filled 2025Q4 (or are extended again by the same WEO mechanism, flagged as synthetic). If a single common balanced endpoint is required across all blocs, the honest common observed endpoint is still 2025Q4; 2026Q1 is achievable as a deliberately ragged-edge refresh with CN/IN flagged.

    Part 2 — Refresh MECHANICS & Automation (no re-estimation)¶

    Complements Part 1 (data availability). This part answers: given the data, exactly HOW does a no-re-estimation refresh work — what re-projects vs what's a static artifact, which JSONs must be regenerated, whether δc/conditioning/plotting follow, and a one-command automation sketch.

    (1) How the displayed BASELINE is produced — split brain (precomputed and re-projected)¶

    There are two distinct baseline objects, and the refresh story depends on which:

    A. The SCENARIO / DEVIATION engine RE-PROJECTS in-browser (no JSON dependence). makeCountryModel (gvarSolve.ts:344-363) computes baseLvl at load time via conditionalForecast(params, histMat, allNaN) — the DK smoother projecting the fixed companion (companion_mean.betahat/sigmahat) forward from historical_levels.data. It does not read baseline_levels.json. The single-country path is identical: useConditionalForecast.ts:199-203 re-runs the unconditional smoother and explicitly uses "SMOOTHER's unconditional as reference (not stored baseline)" so deviation=0 ⇒ delta=0. ⇒ Every conditioning result, GVAR cross-country solve, δc overlay, and scenario band already re-anchors from historical_levels + companion_mean the instant the history changes — no MATLAB, no JSON regen.

    B. The displayed BASELINE LINE + always-on BANDS are PRECOMPUTED MATLAB artifacts. The grey baseline central line and its fan are read directly from static JSON: q/q view from baseline.json .mean/.q16/.q84 (ChartGrid.tsx:1396,1491-1492); level view from baseline_levels.json (ChartGrid.tsx:1399-1402,1539-1540). These come from export_bvar_gvar24_loop.m:218,233-234: meanLvl = VARcf_DKcks([Xc;nan(H,n)],lags,Gam,sm,0) (deterministic DK mean of the fixed posterior-mean companion — byte-identical to the browser's conditionalForecast(allNaN), memory invariant "~1e-11"); bands = quantile(R.Dfc,…) where R.Dfc is the predictive density cgl_bvarV5 produced during estimation from the old last-observed state.

    Settled: the engine re-projects (JSON-independent); the displayed baseline line+band is a precomputed artifact. A refresh that re-anchors only the engine but leaves baseline*.json stale shows the scenario on the new vintage while the grey baseline fan stays on the old anchor — visibly inconsistent. So baseline*.json MUST be regenerated.

    (2) Minimal refresh — exact regen list (per bloc)¶

    File Refresh? How / why
    historical_levels.json REGEN (append 1 row) observed 100·log levels; engine state anchor
    historical.json REGEN (append 1 row) q/q-ann display history = gh(levels) (4·Δlog)
    baseline.json REGEN (mean+bands) mean = re-project fixed companion (deterministic); bands = re-project companion_draws β,Σ from the new state
    baseline_levels.json REGEN (mean+bands) same, in level space
    fiscal_meta.json REGEN d0 only (US/EA/JP/UK/CA/CN) d0 = last-observed debt level (US=122.568, 100·log) = debt-recursion seed; sfaDefault/idx unchanged
    companion_mean.json UNCHANGED estimated model — frozen
    companion_draws.json(.gz) UNCHANGED posterior β,Σ — frozen (re-used to re-project bands)
    spec.json, transformations.json UNCHANGED frozen
    bvar/gvar/coupling_default.json.gz, weights_*.json UNCHANGED (I−M)⁻¹ depends on betahats+trade weights, not data vintage → no 19-min rebuild, no 100MB issue
    DSGE catalog/IRFs UNCHANGED independent of BVAR vintage

    App recomputes for itself (no regen): all conditioning/deviation paths, the GVAR solve, scenario bands (runGvarBandSolve.ts re-runs each companion_draws draw through the solver from the refreshed history), and the δc drift cache.

    Bands — the one real "need MATLAB?" question → NO. Today's baseline bands come from cgl_bvarV5's R.Dfc. But companion_draws.json ships the full per-draw (β,Σ), and the browser already re-projects bands from them via conditionalForecast per draw (runGvarBandSolve.ts:64-70, drawCompanion). So the baseline band can be regenerated in pure JS/Python by looping the shipped draws through the same DK projector from the new state. (The app's band is parameter-only by documented design — runGvarBandSolve.ts:11-14 — so reproducing it parameter-only is the intended, consistent choice; R.Dfc additionally carried DK simulation shocks, which the live app does not display anyway.)

    (3) δc overlay + conditioning on the refreshed baseline — YES, both still work¶

    • δc drift: the drift cache is built from betahat/n/lags (fixed companion) and re-anchors off baseline.mean's terminal (ChartGrid.tsx:1062-1087). Post-baseline by construction — feed it the refreshed baseline.json and it re-anchors automatically. No code change.
    • Conditioning: the engine reference is the recomputed unconditional smoother (§1A), so pins/scenarios re-anchor with zero changes; bands re-run draws from the new history. Fiscal conditioning seeds the debt recursion from fiscal_meta.d0 — hence d0 is in the regen list.

    (4) Plotting / dates — ONE hardcoded constant blocks the auto-extend¶

    X-axis is numeric offsets (x=−1 last hist, x=0 first forecast) mapped to calendar quarters by lastHistOf(country) (dates.ts). Forecast-start is not derived from historical.json length — it hangs off one constant:

    dates.ts:38   export const DEFAULT_LAST_HIST_DATE = '2025-10-01'   // 2025 Q4 — common panel end

    Every dated label (offsetLabel/offsetYQ → ChartGrid.tsx:2328,2338,2345, ScenarioBar.tsx, CSV export ChartGrid.tsx:24) flows from this. Left at 2025-10-01, an appended 2026Q1 obs would be mis-dated (new point labelled as a forecast quarter; whole fan slides one quarter into the past). Fix: bump DEFAULT_LAST_HIST_DATE → '2026-01-01'; advance the per-bloc LAST_HIST_DATE overrides only for blocs that actually got a new quarter (CN/IN, if held at 2025Q4, get an override). Horizon stays 20 (H = min(horizon, baseline.mean.length)) — not auto-extended, so the forecast END recedes one quarter (2031Q1), which is the desired re-anchor. No other hardcoded dates/horizons found in app/src (grep 2025Q4/2025-12/2025-10/forecastStart/lastObserved). Stale comments to fix (not load-bearing): ScenarioBar.tsx:10, ChartGrid.tsx:2327, dates.ts:31-37.

    (5) Automation — one command, NO MATLAB¶

    All four projected artifacts reduce to append + DK-project the shipped companion, a computation the codebase already implements in pure TS (kalman.ts:conditionalForecast, validated to 1e-11 vs MATLAB). A Node refresher reuses the production engine verbatim; MATLAB is needed only to re-estimate β,Σ (out of scope).

    refresh-quarter.mjs — for each bloc C in weights_full.json:
 1. PULL 2026Q1 for C from the SAME sources build_*.py use (FRED hardcoded key, Eurostat keys,
    WEO/raw JSON); reuse each build's series map, fetch only the latest obs.
 2. APPEND → rebuild the 100·log level row exactly as the build script does (same transforms,
    COVID-as-missing, window policy); push onto historical_levels.data; historical.json = gh(levels).
 3. RE-PROJECT (pure JS, FIXED companion — import kalman.ts/conditionalForecast):
      params  = companion_mean.json  (UNCHANGED)
      histMat = refreshed historical_levels
      baseline_levels.mean = conditionalForecast(params, histMat, allNaN).mean   # = VARcf_DKcks(…,0)
      baseline.mean        = gh(baseline_levels.mean)
      bands: for each draw (β_d,Σ_d) in companion_draws:
               Lfc_d = conditionalForecast({β_d,Σ_d}, histMat, allNaN).mean
             baseline_levels.q16/84/05/95 = quantile_d(level);  baseline.q16/… = quantile_d(gh(Lfc_d))
 4. FISCAL blocs: fiscal_meta.d0 = refreshed last-observed debt level.
 5. WRITE historical.json, historical_levels.json, baseline.json, baseline_levels.json, (fiscal_meta).
 6. ONE-TIME: dates.ts DEFAULT_LAST_HIST_DATE → '2026-01-01' (+ per-bloc overrides for laggards).
    • Cost: band re-projection = nDraws × conditionalForecast per bloc (US: ~500 × one 20q smoother over 41×41) — exactly what runGvarBandSolve does in the worker; offline it's seconds/bloc. Companion + coupling untouched → no (I−M)⁻¹ rebuild, no GitHub 100MB issue.
    • MATLAB-free: YES for a pure data refresh; required ONLY for re-estimation.
    • Validate before shipping: match each build_*.py source map + the COVID-as-missing / WEO-extension / window rules so the appended row reproduces the harmonised panel byte-for-byte (the panel was built to a common endpoint with held-last/truncate — Part 1; the refresher must apply the same window policy, and treat CN/IN per Part 1's ragged-edge recommendation).

    Bottom line (mechanics)¶

    • Baseline source: split — engine re-projects from companion_mean+historical_levels (JSON-independent); the displayed line+band is precomputed baseline*.json.
    • Regen exactly: historical.json, historical_levels.json, baseline.json, baseline_levels.json (+ fiscal_meta.d0 for fiscal blocs). UNCHANGED: companion_mean/draws, spec, transformations, coupling, weights.
    • δc + conditioning: unaffected (post-baseline, auto re-anchor).
    • Plotting: one constant dates.ts:DEFAULT_LAST_HIST_DATE 2025-10-01→2026-01-01; rest is offset-based. Horizon stays 20 (forecast end recedes one quarter — desired).
    • Automation: single Node script (append → DK-project the shipped companion for mean AND per-draw bands → write 4 JSONs → bump the date constant) refreshes the whole panel without MATLAB.

    Part 3 — Synthesis: verdict, risks, and the build plan¶

    Consolidates Parts 1–2 into the six deliverables. Parts 1–2 stand as the evidence; this part is the decision-ready answer plus the actual implementation plan for later review on improve-live.

    (1) FEASIBILITY VERDICT¶

    YES — a no-re-estimation refresh to 2026Q1 is feasible today, for the bulk of the system, and it is exactly the post-estimation "re-anchor the fixed model from a new last-observed state" operation the δc sliders already embody. The companion (companion_mean/companion_draws) never changes; only the observed panel extends by one quarter and the forecast re-projects forward from it.

    Realistic as-of quarter: - 2026Q1 for the advanced economies + FRED-sourced EMs — US (incl. all fiscal), EA + EU/EFTA members (real GDP Eurostat, HICP FRED), UK, JP, CA, BR, MX, ZA. 2026Q1 macro is published; monthly rates/CPI already run to 2026-04/05 so the quarterly aggregate is complete. - 2025Q4 (held / WEO-synthetic) for CN and IN — their real source data still ends ~2025Q2 and only reaches 2025Q4 via WEO-annual disaggregation; no genuine 2026Q1 observation exists today. - ⇒ Deliberately ragged-edge refresh: ship 2026Q1 for the AE+FRED-EM blocs, hold CN/IN at back-filled 2025Q4 (flagged synthetic). The honest common balanced endpoint across all blocs remains 2025Q4; 2026Q1 is a ragged edge, which the panel already tolerates (raw JSONs are ragged today: CA/JP at 2026Q1, EA/UK at 2025Q4, CN/IN at 2025Q2).

    (2) AUTOMATION DESIGN (one command, MATLAB-free) — recap¶

    refresh-quarter.mjs: pull → append → re-project the FIXED companion → regen JSONs → bump one date constant. Mean = conditionalForecast(companion_mean, refreshedHist, allNaN) (the production TS DK smoother, validated 1e-11 vs VARcf_DKcks(…,0)); bands = the same projector looped over the shipped per-draw companion_draws (β,Σ) — identical to what runGvarBandSolve already does in the worker. Companion + coupling untouched ⇒ no (I−M)⁻¹ rebuild, no MATLAB, seconds/bloc. Full sketch in Part 2 §5.

    (3) JSON REGEN LIST + who re-projects¶

    Object Regen by script? App auto-reprojects?
    historical_levels.json, historical.json YES (append 1 row) — (these ARE the new state)
    baseline.json, baseline_levels.json (mean + bands) YES engine ignores them, but the displayed grey line+fan reads them → must regen
    fiscal_meta.d0 (US/EA/JP/UK/CA/CN) YES (debt-recursion seed only) —
    companion_mean/draws, spec, transformations, coupling_default, weights_*, DSGE NO — frozen —
    Conditioning / deviation / GVAR solve / scenario bands / δc cache NO YES — re-anchors automatically from historical_levels+companion_mean at load

    The split-brain rule (Part 2 §1): the engine re-projects itself JSON-independently the instant history changes; the static baseline line+band does NOT. So regen baseline*.json or the grey fan sits on the stale anchor while scenarios move on the new vintage.

    (4) PLOTTING CHANGES¶

    • One load-bearing constant: dates.ts:38 DEFAULT_LAST_HIST_DATE = '2025-10-01' → '2026-01-01'. Everything dated flows from it via numeric offsets + lastHistOf(); left stale, the appended 2026Q1 point is mis-labelled and the whole fan slides one quarter into the past.
    • Per-bloc overrides in LAST_HIST_DATE only for blocs that did NOT advance (add a 2025-10-01 override for CN/IN so they stay correctly dated at 2025Q4).
    • Horizon unchanged (H = min(horizon, baseline.mean.length), =20): forecast END recedes one quarter (→2031Q1) — the intended re-anchor.
    • No other hardcoded 2025Q4/2025-12/forecastStart in app/src. Stale comments to tidy (cosmetic): ScenarioBar.tsx:10, ChartGrid.tsx:2327, dates.ts:31-37.

    (5) RISKS + WHAT TO VERIFY¶

    # Risk Mitigation / verification
    R1 Panel-build mismatch — appended row not built byte-identically to the harmonised panel (transforms, COVID-as-missing, WEO-extension, window/truncate policy) ⇒ a level discontinuity at the seam. Replay each build_*.py source map + transform exactly; diff the re-derived 2025Q4 row against the shipped historical_levels last row → must match to ~1e-9 before trusting the appended 2026Q1 row.
    R2 Stale-anchor inconsistency — regen the engine but not baseline*.json ⇒ grey fan on old anchor, scenarios on new. Always regen all four JSONs together; visual check that grey baseline central line == unconditional smoother at deviation 0.
    R3 Date misalignment — forget the DEFAULT_LAST_HIST_DATE bump ⇒ fan slides one quarter. After bump, eyeball the x-axis: last hist tick = 2026Q1, first forecast = 2026Q2 (AE blocs), CN/IN still 2025Q4.
    R4 Band reproduction drift — JS per-draw projection ≠ original cgl_bvarV5 R.Dfc band. Expected & acceptable: the live app's band is parameter-only by design (runGvarBandSolve.ts:11-14); regen the same parameter-only way → consistent with what users already see. Sanity: new bands ⊇ central path, widen with horizon, no NaN.
    R5 CN/IN ragged edge — accidentally extending them with fake 2026Q1. Explicitly exclude CN/IN from the 2026Q1 pull; hold at 2025Q4; flag as synthetic in UI/notes.
    R6 FRED ID traps — discontinued OECD-on-FRED CPI mirrors (GBRCPIALLMINMEI, CPALTT01CAM659N end 2025-03), FREV/FPEXP are internal names not FRED IDs, stale Eurostat HICP endpoint (ends 2025-12). Use only the build scripts' actual series IDs; pull EA/member HICP from FRED, never the Eurostat HICP path.
    R7 δc / conditioning regression — sliders or pins anchor to the wrong terminal after refresh. Post-refresh smoke test: δc=0 ⇒ overlay == baseline; a single pin re-solves; fiscal pin uses refreshed d0 (debt recursion continuous).
    R8 Vintage-drift sanity — a refreshed bloc looks crazy (units, sign, 4·log scale). Per-bloc: 2026Q1 q/q-ann within plausible range vs 2025Q4; level continuous; debt %GDP continuous; spot-check 2–3 blocs by hand against the source print.

    (6) BUILD PLAN (later, on improve-live, for user review)¶

    Read-only today; this is the implementation sequence to propose:

    • Phase 0 — Scaffold & lock sources. Inventory each bloc's exact source map from build_*.py (series IDs, transforms, COVID/WEO/window rules). Define the bloc set: AE+FRED-EM → 2026Q1; CN/IN → hold.
    • Phase 1 — Replay/validate the LAST row (gate). For each bloc, re-derive the existing 2025Q4 row from live sources and diff against shipped historical_levels (~1e-9). Gate: must match before any append — proves the refresher reproduces the panel's transform/window policy.
    • Phase 2 — Append 2026Q1 + re-project (core). Pull 2026Q1, append the 100·log row, run conditionalForecast(companion_mean,…) for the mean and loop companion_draws for bands; write historical*.json, baseline*.json, and fiscal_meta.d0. Reuse kalman.ts/gh verbatim (no re-implementation of the projector).
    • Phase 3 — Plotting. Bump DEFAULT_LAST_HIST_DATE → '2026-01-01'; add CN/IN per-bloc overrides; tidy the stale comments.
    • Phase 4 — Verify (the gate to ship). Run R1–R8 checks: seam continuity, grey-fan vs smoother, axis dates, band sanity, δc=0==baseline, single-pin + fiscal-pin re-solve, 2–3 hand spot-checks. Green-gate + diff review before any commit.
    • Phase 5 — Ship as ragged edge. Commit the regen'd JSONs + the one date change; CN/IN flagged. Document the as-of (2026Q1 ragged; common balanced 2025Q4). Re-runnable each quarter (the script is the quarterly-refresh tool, not a one-off).

    One-line cost: nDraws × conditionalForecast per bloc (seconds), companion/coupling frozen — the whole refresh is minutes, MATLAB-free.
