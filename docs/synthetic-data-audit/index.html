    4. ESTIMATION-PANEL IMPLICATION (the decision that matters)¶

    The synthetic data WAS in the 2025Q4 estimation panel. The model saw it.

    build_gvar_2025q4.py (verified header) WEO-extends short domestic series quarter-by-quarter "so the GVAR re-estimation has a balanced panel" and truncates anything past 2025Q4. So the KR (7q), RU (8q), TH (4q), ID (3q), IN (2q) fabricated tails, the CN nominal-GDP-proxy backfill, and every held-last column entered the estimated companion (companion_mean.json, companion_draws.json). The posterior VAR coefficients, GVAR coupling, and predictive bands were fit on a panel containing synthetic observations — for KR/RU at the final 7–8 quarters, the most forecast-relevant edge.

    refresh-quarter.mjs keeps the companion FROZEN; the 2026Q1 refresh only re-anchors the display (re-projects the fixed companion on refreshed history), no re-estimation.

    Therefore display-truncation does NOT suffice for the deep-synthesis blocs. Truncating the chart edge hides the synthetic points but leaves a model estimated on them. Two tiers:

    • Tier 1 (re-estimate OR re-source) — synthetic baked into the estimated companion: CN, KR, RU, TH, ID, IN. A genuinely "no synthetic data" publish implies a real-only re-estimation on a ragged panel (or re-source of genuine 2026Q1 where it exists — KR/TH/RU GDP have live alternatives).
    • Tier 2 (re-date / truncate display only) — values genuine, dating/refresh artifact: all off-by-one & +1Q-relabel blocs, plus the US held-stars/SFA.

    historical_levels.json is a 40-row matrix, no embedded dates; dates.ts implies 2016Q2..2026Q1 (idx0..idx39). Back-transforming the shipped 100·log levels and comparing to genuine source rows:

    • shipped idx39 ("2026Q1") == genuine source 2025Q4, and shipped idx38 ("2025Q4") == genuine 2025Q3, for 8/8 macro variables in LT, LU, LV, NL, NO, MX, NZ (and MY — see MY note). Example matches:
    • LT GDP: shipped idx39 → 13087.86 ; live Eurostat namq_10_gdp CLV15 B1GQ LT 2025-Q4 = 13087.8 (2025-Q3 = 12845.0 = shipped idx38; live 2026-Q1 = 13051.5 is NOT shipped).
    • NO GDP: shipped idx39 → 106531.9 ; raw panel DataRaw_NO.json 2025-Q4 = 106532.9.
    • MX GDP: shipped idx39 → 6.4204e12 ; raw panel 2025-Q4 = 6.42047e12.
    • NZ GDP: shipped idx39 → 70579.9 ; swarm source oecd_swarm/DataRaw_NZ_dom.json last row (2025-Q4) = 70577.7.
    • Global columns (GLB_OIL/GLB_USLR/GLB_USBAA/GLB_USEQ) are ALSO shifted by one quarter — confirmed against a LIVE FRED pull (key 8ce324...):
    • WTISPLC q-avg: 2025Q3 65.737, 2025Q4 59.640, 2026Q1 71.977. Shipped GLB_OIL idx38→65.74, idx39→59.64 (= genuine 2025Q4); the genuine 2026Q1 71.977 is NOT shipped.

    • BAA: 2025Q4 5.83, 2026Q1 5.91 → shipped GLB_USBAA idx39 = 5.83 (2025Q4).
    • Source panels confirm 2025Q4 end: pipeline/country_rawdata/DataRaw_<CC>.json ends 2025-Q4 for LT/LU/LV/NL/NO/MX/NZ; DataRaw_<CC>_glob.json ends 2025-Q4 for all; swarm *_dom.json ends 2025-Q4. refresh-quarter.mjs only has a US source registry and is fail-closed for every other bloc (lines 189-191, 460-464) — so none of these 8 were genuinely refreshed to 2026Q1.

    Internal consistency¶

    Because macro, global, and derived columns are ALL shifted by the same one quarter, the panel is internally coherent (no cross-column date seam). The defect is purely the dates.ts label.

    Recommended remedy (human/code action — do NOT drop real data)¶

    The genuine data should be KEPT and RE-DATED to 2025Q4 (set a LAST_HIST_DATE override of 2025-10-01 for each of LT, LU, LV, MX, MY, NL, NO, NZ), OR the panel should be genuinely refreshed to 2026Q1 by adding a source registry for these blocs (the genuine 2026Q1 source data now exists). Until one of those is done, the publish should stay HALTED: the chart presents 2025Q4 data as 2026Q1. Because the values themselves are genuine, the right per-variable action is needs-human-check (re-date), NOT truncate/drop — truncating would discard a real observation. truncate-tail-to-genuine is recommended ONLY for genuinely-null/earlier-ending tails (LENDR_MX).

    MY note (only bloc whose raw panel actually reaches 2026Q1)¶

    DataRaw_MY.json genuinely ends 2026-Q1 (the macro source has it). BUT the shipped MY historical_levels still uses the 2025Q4-labeled-2026Q1 row — verified: shipped MY idx39 matches the raw 2025Q4 values (CONS 267533 = raw 2025Q4 267535, not raw 2026Q1 269083; CPI 135.23 = raw 2025Q4, not raw 2026Q1 136.03; NEER, INV, GOV, EXP, IMP all match 2025Q4). So MY is one quarter STALE relative to its own available genuine 2026Q1 data, AND mislabeled. Same remedy: re-date to 2025Q4 or rebuild to the genuine 2026Q1 the source already publishes.

    What is NOT flagged¶

    • COVID mask (2020Q2–2021Q1 NaN): MISSING data (handled by the COVID-as-missing Gibbs), NOT synthetic — not flagged per the directive.
    • POLRATE held tails (e.g. POLRATE_LT/LU/LV/NL idx39==idx38, MY LR/RATE/POLRATE/HP): these are EXACT carry-forwards but they reflect a policy rate / index that genuinely did not change between the two quarters (the rate was unchanged), so they are not fabrication — they are still the 2025Q4-labeled -2026Q1 mislabel like every other column.

    Verifiability caveats¶

    All macro and global columns were re-pulled and matched against LIVE sources (Eurostat SDMX, FRED API) or the in-repo genuine source/swarm panels. Fiscal genuine end re-pulled live for LT (gov_10q_ggnfa) and confirmed 2025Q4. The remaining per-member fiscal/credit/lending columns were not each re-pulled live against their primary API within scope; they sit on the SAME panel and inherit the SAME 2025Q4 end + one-quarter mislabel, but where a column's primary source was not independently re-pulled it is marked needs-human-check rather than assumed clean.
