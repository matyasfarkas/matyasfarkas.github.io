    Runbook: adding a country / updating a country's data and model¶

    This is a human, step-by-step guide for the data-and-estimation side of the tool. It covers three jobs:

    • A. Update an existing country (refresh its data, re-estimate, redeploy). Worked example: the US.
    • B. Add a brand-new country from scratch (standalone single-country model, optional GVAR link).
    • C. Add / refresh a 4-variable fiscal block (US, EA, JP, UK, CA, CN today).

    The frontend is generic: it reads n (the number of variables) and every index from the model JSON, so once a country's JSON files are in app/public/data/bvar/<CC>/ and the country is registered in the single registry app/src/lib/data/countries.ts, the charts, conditioning, uncertainty, and the long-run anchor (LRP / entropic tilt) all work automatically — no extra data files are needed for any of them. The only hard part is the data and the estimation. (For the code-side architecture, see ARCHITECTURE_FOR_AI.md.) The deployed tool is live at the /bvar-scenario path of the academic website, hosted on GitHub Pages (the matyasfarkas/starter-hugo-academic Hugo Academic site).

    Long-run anchor (LRP / entropic tilt). Once a model ships, the user can set a long-run target and tilt the forecast toward it: the Compute button is enabled whenever an enabled anchor exists (auto-selecting the Quick mode), and the tilt moves the unconditional central path (the weighted mean from reduceCloud), not just the fan. The tilt moment is the H-horizon forecast, not the steady state. For a GVAR bloc the anchor is marginal (per-bloc), with a disclaimer shown in the UI. This is entirely frontend — adding a country needs no LRP-specific files.

    Current topology (so the numbers below have context). The coupled GVAR has 33 blocs, and the US is now a FULLY ENDOGENOUS coupled bloc, not the frozen anchor. Its spec.json carries the three ROW*_US stars (ROWDEM_US/ROWINF_US/ROWFIN_US, block:'row') built from a US trade-weights row, so it both drives Rest-of-World and receives RoW→US feedback (n=41: 30 domestic + 4 global + 4 fiscal + 3 stars). The set is: the US + euro area as one aggregate bloc (EA) + 31 Rest-of-World blocs (JP UK CA CN CH KR MX AU TR SE PL NO DK IN BR RU ID SA AR ZA TH MY CL CO CR CZ HU IL IS NZ TW). With the US endogenous the coupling is the augmented (stars ++ US-global) closed-form operator of dimension 2060, with spectral radius rho(M) = 0.924 < 1 (stable; verified off the shipped coupling_default.json.gz header — rhoSpectral = 0.924041). The engine detects endogeneity from the data: US_ENDOGENOUS = !!usIdx.row && !!WEIGHTS24.weights.US in gvarSolve.ts, so COUPLED_COUNTRIES = ['US', ...rows]. The 17 euro members (DE FR IT ES NL AT BE FI GR IE LU PT SI SK EE LT LV) are standalone single-country models (domestic + 4 global commons + trade-weighted foreign stars, estimated in separation via MAXIT=1) — they are not in the coupled GVAR and have no weights row. Reason: the fully-disaggregated euro GVAR was non-convergent (rho>1); aggregating the euro area restores stationarity, which is why EA is one bloc and the members are standalone. (50 country directories ship in all: 33 coupled GVAR blocs + 17 standalone euro members.)

    Registry / selector. The country dropdown is driven by app/src/lib/data/countries.ts (one entry per economy, { code, name, group }). Adding a country = add an entry there + drop its data files; no dropdown edit is needed. The selector groups entries into "GVAR" / "Other countries" optgroups (driven by isGvarCountry from gvarLoader in controls/TopBar.tsx; plain names, no "- GVAR" suffix). group is only a coarse tag (core/oecd/em); GVAR membership is data-driven — a bloc is GVAR-linked iff its spec.json carries block:'row' vars AND it has a row in app/public/data/bvar/gvar/weights_24bloc.json (the filename is historical — the file now holds the 33-bloc weights, including the US row). The legacy app/public/data/meta/countries.json is display metadata only.

    What a country needs (the contract)¶

    The app loads these files from app/public/data/bvar/<CC>/ (<CC> = 2-letter code, e.g. US, EA):

    File Required What it is
    spec.json yes { n, variables:[{id,name,transformation,isFinancial,prior,block}] } — block ∈ domestic/row/global/fiscal; row+global are the GVAR foreign/commons blocs, fiscal tags the 4 fiscal-accounting vars; a standalone non-fiscal bloc is all domestic
    companion_mean.json yes estimated VAR coefficients { betahat, sigmahat, n, lags }
    baseline.json yes unconditional forecast in growth/display space (mean + q16/q84/q05/q95)
    baseline_levels.json yes unconditional forecast in level space
    historical.json / historical_levels.json yes last ~40 quarters (growth / level)
    transformations.json yes ['log'\|'lin', ...] per variable
    companion_draws.json optional posterior draws → enables Full uncertainty bands
    fiscal_meta.json 6 fiscal countries fiscal-block indices {gdp,deflator,revenue,primaryExp,effRate,sfa} + d0 + sfaDefault. Present for US/EA/JP/UK/CA/CN

    transformation: 'log' = the series is stored as 100·log(level) and shown as 4·Δ (q/q annualized growth) — use it for real GDP, prices, indices. 'lin' = passed through as-is — use it for rates, ratios, unemployment. isFinancial just groups the variable under the "financial" header.

    Fiscal block (the 6 fiscal countries) — now 4 variables. US/EA/JP/UK/CA/CN carry four estimated fiscal vars (block:'fiscal', all lin):

    1. FREV_GDP — revenue / GDP × 100 (% of GDP, 4-quarter trailing ratio)
    2. FPEXP_GDP — primary expenditure (= total exp − interest) / GDP × 100 (% of GDP)
    3. EFFI — effective interest rate = net interest / lagged gross debt × 100 × 4 (% per annum). This reverts the earlier debt-service swap (DSVC) back to the rate — older builds estimated DSVC; the current builds estimate EFFI.
    4. SFA — stock-flow adjustment, a new estimated lin BVAR variable = the EXACT one-step residual that closes the app's debt recursion (see below).

    fiscal_meta.json idx is {gdp, deflator, revenue, primaryExp, effRate, sfa} — there is no debtService key any more. In fiscalAccounting.ts computeFiscalBlock reads idx.effRate as the rate driver, and — because idx.sfa is set — reads the per-quarter SFA forecast off the level path (levelPaths[t][idx.sfa]), so the debt/GDP projection now carries the deficit–debt discrepancy forward with its own dynamics and bands instead of being defaulted to 0. (fiscalAccounting.ts still contains a dormant idx.debtService branch for backward compat; the shipped metas leave it unset.)

    The debt recursion (per quarter, computeFiscalBlock): 

    pb_t  = FREV_GDP_t − FPEXP_GDP_t                                  (% of GDP, annual)
gNomQ = 100·Δlog(realGDP) + 100·Δlog(deflator)                   (% per quarter, off the level path)
d_t   = d_{t-1}·(1 + EFFI_t/400)/(1 + gNomQ_t/100) − pb_t/4 + sfa_t/4     (debt/GDP, %)
    Debt-to-GDP is DERIVED (never an estimated variable); d0/sfaDefault seed and fall-back it.

    SFA = the EXACT app residual (data-build only — see §C). For every non-COVID quarter the build rearranges that same recursion onto the observed gross-debt/GDP path: 

    snowball_t = ((1 + EFFI_t/400)/(1 + gNomQ_t/100) − 1)·d_{t-1}
SFA_t      = 4·(d_obs_t − d_{t-1} − snowball_t + pb_t/4)          (% of GDP, annual)
    so rolling the recursion forward with this SFA reproduces the observed debt path to machine precision (closure checked < 1e-6pp). The COVID quarters are set to NaN (NOT interpolated — an interpolated value cannot equal the residual) so the COVID-as-missing Gibbs imputes them with the rest of the block. SFA captures the deficit–debt-discrepancy / gross-debt-perimeter seam (intragovernmental holdings, federal-flows-vs-gross-debt). Deployed sfaDefault (≈ non-COVID SFA mean, % GDP): US 1.7518, EA 0.325, JP 2.233, UK 3.1487, CA 3.191, CN 4.017 (verified in each bloc's shipped fiscal_meta.json). The SFA estimation-sample length (number of non-COVID quarters with an SFA observation) is US 41, EA 21, JP 21, UK 23, CA 23, CN 18 — these are SFA-column sample lengths, not the bloc's spec variable count n (which is US 41, EA 24, JP 23, UK 25, CA 25, CN 20). Do not conflate the two.

    A checkable walkthrough lives in docs/fiscal_accounting_walkthrough_US_UK.xlsx (built by pipeline/build_fiscal_walkthrough_xlsx.py) alongside the source docs/Fiscal_Accounting_US_UK_NSA.xlsx.

    The estimation tooling lives in pipeline/:

    • build_country_data.py — pulls FRED-based countries (EA/UK/JP/CA …) → raw CSV/.mat.
    • build_oecd_eurostat.py / build_oecd_batch.py — pull the OECD (EU/EFTA + others) economies.
    • build_us_auto.py — hands-off US builder (pulls everything from FRED + builds the fiscal drivers).
    • build_fiscal_<cc>_4var.py (cc ∈ us/ea/jp/uk/ca/cn) — the 4-var fiscal-block builder: writes {FREV_GDP, FPEXP_GDP, EFFI, SFA} into the bloc's country_rawdata/DataRaw_<CC>_glob.json (the exact-residual SFA + COVID-NaN recipe) and rewrites fiscal_meta/<CC>.json (see §C). Data build only — no estimation, no deploy.
    • export_bvar_gvar24_loop.m — the GVAR-EM iterative estimator (despite the 24 in the name it produces the 33-bloc coupled set with the US ENDOGENOUS and the euro area as one EA aggregate). Set env US_ENDOGENOUS=1 (requires a US weights row). It runs a parfor Jacobi sweep over the blocs (MAXIT iterations to the Monte-Carlo noise floor, TOL=0.03), each bloc estimated by COVID-as-missing Gibbs (cgl_bvarV5). MAXIT=1 = single-country (separated) mode for standalone blocs / euro members.
    • export_bvar_us_covid.m / export_bvar_oecd_covid.m / export_bvar_country.m — single-country exporters (US-only, OECD, generic).
    • country_specs.ts (variable lists for spec.json) / app/src/lib/data/countries.ts (the country registry / dropdown).

    API key: the FRED pullers read FRED_API_KEY from the environment. Always pass your own: FRED_API_KEY=<your_key> python3 pipeline/.... Do not commit a key (app/.env.local is the place for local secrets and is git-ignored). Get a free key at https://fred.stlouisfed.org/docs/api/api_key.html.

    Network: the FRED/IMF pulls need outbound HTTPS. If you run inside a sandbox that blocks the network, disable it for the pull step.

    A. Update an existing country (worked example: the US)¶

    The US is fully automated end-to-end — this is also the test case for this runbook. Because the US is now an endogenous GVAR bloc, a US data refresh is a GVAR re-estimation (§B6 path), not a single-country export. The single-country export_bvar_us_covid.m still exists for an isolated US-only model, but the deployed US is produced by the GVAR-EM loop.

    A1. Rebuild the US dataset (macro + fiscal)¶

    cd "<repo root>"
FRED_API_KEY=<your_key> python3 pipeline/build_us_auto.py    # macro + fiscal drivers, COVID handled
python3 pipeline/build_fiscal_us_4var.py                     # SFA → exact app residual, COVID-NaN

    build_us_auto.py pulls every US series on the auto-detected quarterly grid (currently to 2025Q4) and assembles the raw matrix. build_fiscal_us_4var.py then overwrites only the SFA column in pipeline/country_rawdata/DataRaw_US_glob.json with the exact one-step residual of the app recursion (reading FREV_GDP/FPEXP_GDP/EFFI + the real-GDP & deflator level columns the app reads, plus the observed debt/GDP from pipeline/fiscal_raw/us_grid.json), NaN-ing the COVID window (2020Q2–2021Q4), runs the < 1e-6pp closure check, and rewrites pipeline/fiscal_meta/US.json (idx{gdp,deflator,revenue,primaryExp,effRate,sfa}, d0, sfaDefault = non-COVID SFA mean).

    A2. Re-estimate the coupled GVAR (the US is endogenous)¶

    # In MATLAB, with the repo's BVAR/ on path (the .m adds it):
US_ENDOGENOUS=1 matlab -batch "run('pipeline/export_bvar_gvar24_loop.m')"

    This estimates all 33 blocs (adding/refreshing the US changes everyone's trade-weighted RoW, so the whole set is re-estimated), with the US folded into the convergence loop with its ROW*_US stars. It saves the workspace to /tmp/gvar24_workspace.mat and writes the full JSON contract per bloc to /tmp/gvar24_export/<CC>/. Confirm the printed per-iteration max star change falls below TOL (convergence) and the printed GDP1q central forecasts are sane.

    For an isolated US-only refresh (no GVAR), export_bvar_us_covid.m (cgl_bvarV5, sur=0, empirical-Bayes lambda/mu, 1000 draws / 500 burn-in) still writes a standalone US export to /tmp/us_export/. But the deployed US is the endogenous GVAR bloc, so prefer the loop above.

    A3. Sanity-check, then port over the live data¶

    Inspect /tmp/gvar24_export/ first (never overwrite live data blind). Then port every bloc (6-sigfig compaction is the deployed convention) and refresh the fiscal meta:

    cd "<repo root>"
python3 pipeline/port_gvar_exports.py                 # all blocs → app/public/data/bvar/<CC> (6 sig-figs)
cp pipeline/fiscal_meta/US.json app/public/data/bvar/US/fiscal_meta.json

    A4. Regenerate the coupling artifact¶

    Any change to a deployed bloc invalidates the precomputed coupling (it is keyed by modelSig):

    cd app
npx vite-node scripts/precompute_gvar_coupling.mjs    # → public/data/bvar/gvar/coupling_default.json.gz

    A5. Run the green gate¶

    cd app
npx tsc --noEmit
npx vitest run                         # all must pass; watch the GVAR + scaling canaries
BASE_PATH=/bvar-scenario npm run build

    If n changed, the canary test scenario-scale-convention.test.ts may carry a fixture pinned to the old layout — regenerate it (the pipeline has _verify/gt.m, which resolves columns by variable id, so it is layout-robust) and re-run. Everything else is index-agnostic.

    A6. Deploy¶

    cd "<repo root>"
git add app/public/data/bvar
git commit -m "Update US + coupled GVAR (data through <quarter>)"
git push bvarscenario main              # host auto-builds on push

    The push to bvarscenario (matyasfarkas/BVARScenarioTool) triggers a GitHub Actions run that builds the static app (BASE_PATH=/bvar-scenario) and copies app/out/* into the matyasfarkas/starter-hugo-academic website repo's static/bvar-scenario/. That repo is a Hugo Academic site published to GitHub Pages, which rebuilds and serves it at the /bvar-scenario path. (The Action itself has no Hugo step.) Hugo is pinned to 0.122.0 in the website repo (0.123+ removed getCSV and broke the wowchemy/academic theme — do not bump it).

    Done = the gate is green and the push succeeded. Open the live site (the /bvar-scenario path of the academic site on GitHub Pages) and confirm the US baseline, a quick conditioning scenario, and a "Solve full GVAR" run look sane.

    B. Add a brand-new country¶

    Use an existing FRED country (EA/UK/JP/CA) as the template.

    B1. Choose the variables and find FRED IDs¶

    Decide the variable block (a typical small open economy uses ~12–16: real GDP, consumption, investment, government spending, exports, imports, industrial production, unemployment, CPI/HICP, a short rate, a 10Y yield, the nominal effective exchange rate). For each, find the FRED series ID and tag it log (levels/indices) or lin (rates/ratios). Prefer quarterly, seasonally adjusted, long-history series. If a country's data is not cleanly on FRED (e.g. China), source it from IMF IFS/WEO or OECD and assemble the same raw matrix by hand — but use real observed data, never synthetic placeholders.

    B2. Add the country to the data builder¶

    Add a block to COUNTRIES in pipeline/build_country_data.py (mirror the EA entry):

    "XX": {
  "name": "Country Name",
  "start": "YYYY-01-01",
  "domestic": [
    ("FRED_ID", "GDP_XX", "XX Real GDP", "log", False),
    ... one tuple per variable: (fred_id, internal_name, display_name, 'log'|'lin', is_financial)
  ],
},

    Then run FRED_API_KEY=<key> python3 pipeline/build_country_data.py to pull the raw data.

    B3. Reflect the spec + register the country¶

    Add the matching variable list to pipeline/country_specs.ts (used to emit spec.json) and add a one-line entry to the registry app/src/lib/data/countries.ts:

    { code: 'XX', name: 'Country Name', group: 'oecd' },   // group: 'core' | 'oecd' | 'em'

    That single entry is what puts the country in the dropdown (in the "Other countries" optgroup until it becomes a GVAR bloc) — the engine reads the real n from companion_mean.json, so no size metadata is needed here. (The legacy app/public/data/meta/countries.json is display-only and not required.)

    B4. Estimate + export¶

    Estimate with the COVID-as-missing Gibbs sampler (cgl_bvarV5) and emit the full JSON contract (§"What a country needs"). For a standalone single-country bloc the cleanest template is pipeline/export_bvar_oecd_covid.m / export_bvar_country.m; for a GVAR bloc use export_bvar_gvar24_loop.m (§B6). The exporter produces companion_mean, baseline (growth) and baseline_levels, historical and historical_levels, transformations, and companion_draws. Standard config: the 2020 COVID quarters are NaN'd (window self-calibrated to the 2020Q2 GDP trough, searched within 2019Q4–2021Q1) and DK-imputed; lambda/mu are empirical-Bayes (MH on the completed-data marginal likelihood); 'sur',0,'noc',1,'Ndraws',1000,'Ndrawsdiscard',500.

    If the estimator returns a singular/NaN SIGMA, the data is the cause: a NaN in a discontinued series, or near-collinear national-accounts identities. Drop the offending leading-NaN rows and/or fix a multicollinear variable. With COVID imputed, the empirical-Bayes lambda optimum is well-behaved (estimated ~0.24–0.56), so a fixed lamfix is no longer needed; if you do pin one, keep noc=1 or sur=1 free or lambda degenerates to NaN.

    B5. Place files, regenerate coupling (if GVAR), gate, deploy¶

    cp /tmp/<XX>_export/*.json app/public/data/bvar/XX/      # or: python3 pipeline/port_gvar_exports.py
cd app
# (if XX is a GVAR bloc, regenerate the artifact — see B6.4)
# npx vite-node scripts/precompute_gvar_coupling.mjs
npx tsc --noEmit && npx vitest run && BASE_PATH=/bvar-scenario npm run build
cd .. && git add app/public/data/bvar/XX app/src/lib/data/countries.ts pipeline/*.py pipeline/country_specs.ts
git commit -m "Add XX country model" && git push bvarscenario main

    B6. (Optional) GVAR integration¶

    To link the new country into the 33-bloc coupled GVAR (a Rest-of-World bloc rather than a standalone single-country model). Caveat: keep the euro area aggregated — adding the euro members as their own coupled blocs makes the system non-stationary, which is why the 17 euro members ship as standalone models (no weights row) and EA is one bloc. Verify the spectral radius stays < 1 after re-estimation (it is 0.924 for the current 33-bloc / dim-2060 endogenous-US set).

    1. Add the bloc to the trade-weights file. Add XX (with its ISO-3 code) to the bloc list in app/public/data/bvar/gvar/weights_24bloc.json (regenerated by pipeline/build_trade_weights_full.py / pipeline/build_trade_weights.py from IMF IMTS; the US own-reporter row is built by pipeline/build_us_weights_row.py). Every bloc's row is renormalized over the active set, so adding a country changes everyone's trade weights.
    2. Give the bloc its foreign + global variables. Its spec.json must carry the 3 Rest-of-World stars (ROWDEM_XX/ROWINF_XX/ROWFIN_XX, block:'row') and the 4 shared global commons (GLB_OIL/GLB_USLR/GLB_USBAA/GLB_USEQ, block:'global'). The estimator export_bvar_gvar24_loop.m builds the stars from partners' completed series and iterates the foreign components to a cross-country fixed point — re-estimating every bloc (adding a country changes everyone's trade-weighted RoW). The raw inputs are the pipeline/country_rawdata/DataRaw_<XX>_glob.json files (domestic + the 4 globals; the stars are appended at export time, not stored in the glob). Run with US_ENDOGENOUS=1 so the US stays a coupled bloc.
    3. No frontend code edits are needed for topology. The GVAR engine self-configures from the deployed spec.json files + weights_24bloc.json (configureGVARTopology/deriveBlocIdx/US_ENDOGENOUS in gvarSolve.ts): a bloc becomes active automatically once it ships block:'row' vars and has a weights row. Do not hand-edit IDX/W/COUPLED_COUNTRIES.
    4. Regenerate the precomputed coupling. After re-estimating + porting, rebuild the coupling artifact: 
      cd app && npx vite-node scripts/precompute_gvar_coupling.mjs
      It builds the augmented-state coupling M in JS (DK-smoother probes), inverts (I − M) via a MATLAB BLAS batch (app/scripts/invert_coupling.m), and writes the gzipped artifact app/public/data/bvar/gvar/coupling_default.json.gz directly ({modelSig, pinStructure:'', dim, rhoSpectral, M, invImM}, 6-sig-fig). The plain .json is removed/skipped because at this bloc count it would exceed GitHub's 100 MB per-file limit (gzip brings the current dim-2060 artifact to ~30 MB); the loader decompresses it in-browser via the native DecompressionStream (falling back to a plain .json only if the .gz is absent). The browser solves x = (I − M)⁻¹d from this artifact; without it the in-browser inverse (~19 min at 24 blocs; far worse at the augmented endogenous dim) is infeasible. The artifact is keyed to the data by modelSig, so it must be regenerated whenever any deployed bloc changes.
    5. Add tests. Extend the GVAR canaries in app/src/__tests__/compute/ (e.g. gvar-closed-form-routing.test.ts, gvar-coupling-artifact.test.ts, gvar-derived-topology.test.ts, and the endogenous-US / closed-form stress tests) to cover the new bloc.

    Two-tier solve UX, drag off the heavy path (FYI, no per-country work). On the live ~50-bloc US-endogenous panel every mean solve is multi-second (~4.5 s warm / ~13 s cold), so no GVAR solve fires during a drag: the drag path is a cheap conditioning preview (the conditioned dot snaps to the cursor instantly via conditionedDotY.ts), exactly one cheap DIRECT first-order solve (solveGVARDirect, affine per-probe kernels, numerically exact) fires on pointer-up, and the complete fixed point + uncertainty bands run only on the "Solve full GVAR" button / top-bar Compute. A "computing…" badge and a direct/full mode pill are shown; the ≈95 MB coupling artifact is stripped from direct-mode worker posts; and per-chart React.memo means one conditioning edit repaints one panel, not all ~35. New blocs inherit this automatically.

    C. Add / refresh a 4-variable fiscal block¶

    The 6 fiscal countries (US/EA/JP/UK/CA/CN) carry {FREV_GDP, FPEXP_GDP, EFFI, SFA} (block:'fiscal', all lin). The dedicated builders are pipeline/build_fiscal_<cc>_4var.py. They are data-build only (no estimation, no deploy, idempotent).

    C1. What the builder does¶

    Each build_fiscal_<cc>_4var.py:

    1. Computes the four columns from the country's raw fiscal source (e.g. pipeline/fiscal_raw/<cc>_grid.json
    2. the macro glob), with the same scaling as the legacy build_fiscal_drivers.py columns: FREV_GDP = 100·rev4/ngdp4, FPEXP_GDP = 100·(exp−interest)4/ngdp4, EFFI = 100·4·interest_t/gross_debt_{t-1} (% p.a.).
    3. Computes SFA = the exact one-step residual that closes the app recursion on the OBSERVED debt/GDP path (reading EFFI, pb = FREV − FPEXP, and gNomQ = 100·Δlog(realGDP) + 100·Δlog(deflator) off the SAME glob columns the app reads), then NaN-s the COVID window so the COVID-as-missing Gibbs imputes those quarters (do NOT interpolate). It runs a closure check (< 1e-6pp) and aborts if the residual does not reproduce the observed debt path.
    4. Edits in place pipeline/country_rawdata/DataRaw_<CC>_glob.json — replacing the prior fiscal columns at their positions and appending the 4th (SFA) column with block:'fiscal' tags — and rewrites pipeline/fiscal_meta/<CC>.json so idx = {gdp, deflator, revenue, primaryExp, effRate, sfa} (NOT debtService), with d0 (last observed debt/GDP) and sfaDefault (non-COVID SFA mean).

    C2. Run it¶

    cd "<repo root>"
python3 pipeline/build_fiscal_<cc>_4var.py     # cc ∈ us ea jp uk ca cn ; edits glob + fiscal_meta in place

    (The US builder, build_fiscal_us_4var.py, overwrites only the SFA column and leaves FREV_GDP/FPEXP_GDP/EFFI byte-identical; the others — ca/cn/ea/jp/uk — additionally revert DSVC → EFFI and append SFA.)

    C3. Re-estimate, port, deploy¶

    The fiscal columns are part of the bloc's domestic/fiscal block, so they are estimated by the same GVAR-EM loop (or single-country exporter). Continue with §A2–A6 (re-estimate via export_bvar_gvar24_loop.m, port with port_gvar_exports.py, copy the refreshed pipeline/fiscal_meta/<CC>.json to app/public/data/bvar/<CC>/fiscal_meta.json, regenerate the coupling artifact, gate, deploy). The shipped fiscal_meta.json is what the frontend reads — porting the bloc JSON alone is not enough; the meta must be copied too.

    C4. MANDATORY RULE — regenerate fiscal_meta indices after ANY variable-order change¶

    RULE (must run after any re-estimation or port that can change a fiscal bloc's variable order or count): run node pipeline/fix_fiscal_meta_indices.mjs.

    fiscal_meta.json stores the fiscal-driver columns by POSITION (idx = {gdp, deflator, revenue, primaryExp, effRate, sfa}), and the debt recursion in fiscalAccounting.ts reads the level path at exactly those positional indices. Any change to a bloc's variable order or count silently re-points those indices at the WRONG columns. This is not hypothetical: the 33-bloc GVAR-EM refit inserted variables ahead of the fiscal block in every spec except US, which re-pointed (e.g.) EA's effRate onto the equity index (~col 537) and made the EA/UK debt-to-GDP path explode. The porter does not regenerate the meta, so the stale indices ship unless you fix them.

    pipeline/fix_fiscal_meta_indices.mjs re-resolves the fiscal indices by variable id (the canonical, order-independent source) from the deployed spec.json and rewrites fiscal_meta.json for all six fiscal blocs (US EA UK CA JP CN), preserving the model parameters d0/sfaDefault (verified — it only rewrites meta.idx). It is idempotent.

    cd "<repo root>"
node pipeline/fix_fiscal_meta_indices.mjs
# prints per bloc:  US: ok {…}   |   EA: FIXED {old} -> {new}   …   N fiscal_meta.json files corrected.

    Run it after every port that touches a fiscal bloc (it is cheap and idempotent — when nothing moved it prints ok for all six and changes nothing). The fiscal conditioning/back-cast canaries (fiscalAccounting.test.ts, fiscal-backcast.test.ts, fiscal-conditioning-integration.test.ts) are the backstop, but the explicit re-index is the fix — a green build with stale indices is still wrong.
