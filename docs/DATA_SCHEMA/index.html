    Data Contracts & JSON Schemas¶

    This page is the reference for every JSON contract the app reads from app/public/data/bvar/. It is the authority for what each file contains, the exact field names, the array shapes, and the conventions (variable ids, block tags, transformations, the 6-significant-figure compaction). It complements — and does not duplicate — the prose in the companion docs:

    • ARCHITECTURE_FOR_AI.md — how the compute engine uses these files (§4–§10).
    • README.md / USER_GUIDE.md — overview & UI.
    • fiscal-block-methodology.md — the math behind fiscal_meta.json.
    • ADDING_A_COUNTRY.md / REESTIMATING_MODELS.md — how the files are produced.

    If a fact here disagrees with the code, the code wins — these contracts are verified against the live files (app/public/data/bvar/US, …/EA, …/DE, …/CN, …/gvar) but the data is re-ported on every re-estimation.

    1. Where the files live¶

    Each economy is one directory app/public/data/bvar/<CC>/ keyed by ISO-2-ish country code (US, EA, JP, …; EA = euro-area aggregate). The cross-country GVAR coupling lives in app/public/data/bvar/gvar/.

    app/public/data/bvar/
  US/   EA/   JP/  …                 one directory per economy (~50)
    spec.json                        variable manifest (ids, blocks, transforms)
    companion_mean.json              posterior-mean VAR (betahat, sigmahat)
    companion_draws.json             flat posterior draws (lazy-loaded, large)
    baseline.json                    baseline fan in DISPLAY/GROWTH space
    baseline_levels.json             baseline fan in LEVEL space
    historical.json                  last ~40q in DISPLAY/GROWTH space
    historical_levels.json           last ~40q in LEVEL space
    transformations.json             per-variable 'log'|'lin' list
    fiscal_meta.json                 ONLY the 6 fiscal blocs (US/EA/JP/UK/CA/CN)
  gvar/
    weights_24bloc.json              33-bloc trade-weight matrix + topology source
    coupling_default.json.gz         gzipped precomputed (I−M)⁻¹ coupling artifact

    Loader (app/src/lib/data/loader.ts). loadBVARData(country) fetches spec/companion_mean/baseline/historical eagerly, then optionally baseline_levels/historical_levels/fiscal_meta (a 404 → null, which is normal for standalone economies that ship no fiscal_meta.json). companion_draws.json is loaded lazily by loadCompanionDraws(country) only when Compute is clicked (it is the largest file — tens of MB) and its flat arrays are converted once to Float64Array for fast indexing and zero-copy structured-clone into the band worker.

    2. Cross-cutting conventions¶

    2.1 The data space (log / lin)¶

    Every variable has a transformation:

    transformation stored as chart display
    "log" 100·log(level) q/q annualised growth = 4·Δ(100·log level)
    "lin" the level itself (rates, ratios, %-of-GDP, sentiment indices) the level as-is
    • *.json files in display/growth space (baseline.json, historical.json) carry the already-differenced 4·Δ(100·log level) for log vars and the level for lin vars.
    • *_levels.json files carry 100·log(level) for log vars and the level for lin vars (this is the "level space" that feeds the fiscal recursion, the GVAR solver, and the level-display fans).

    This is the single most bug-prone convention in the codebase — a stray ×4 or /100 between the two spaces diverges forecasts silently. The canary test scenario-scale-convention.test.ts pins it.

    2.2 The 6-significant-figure compaction¶

    The eight ported per-bloc files are rounded to 6 significant figures and serialised as compact JSON (separators=(",",":"), no whitespace) by pipeline/port_gvar_exports.py (sig6 → float(f"{x:.6g}"), recursing through lists/dicts via compact). The compacted set is exactly the FILES list in that script: spec.json, companion_mean.json, companion_draws.json, historical.json, historical_levels.json, baseline.json, baseline_levels.json, transformations.json. Integers, None, strings, booleans, and ±inf/NaN pass through (0/±inf/NaN map to 0/themselves). So in those files expect values like 8.04724, 1009.29, -1.45722 — never full double precision.

    coupling_default.json.gz is also 6-sigfig: its builder (app/scripts/precompute_gvar_coupling.mjs) rounds M/invImM/rhoSpectral with round6 = x => Number(x.toPrecision(6)) before gzip.

    Two files are NOT 6-sigfig — they are written full precision by their own builders and are not in port_gvar_exports.py's FILES:

    • fiscal_meta.json — emitted by pipeline/build_fiscal_<cc>_4var.py with json.dump(meta, indent=2); d0 is full double precision (e.g. 87.84509550831034), sfaDefault is round(sfa_mean, 4) (4 decimals).
    • weights_24bloc.json — only the US receiver row is 6-sigfig; the other 32 rows carry full double precision (e.g. 0.15734894312225214).

    Do not assume 6-sigfig when validating these two files.

    2.3 Variable ids, blocks, isFinancial¶

    Variable ids are stable string keys (e.g. GDPH, POLRATE, HICP_EA, GLB_OIL, ROWDEM_US). Suffixes encode role:

    • GLB_* — a global common (the 4 US-determined commons, ids GLB_OIL/GLB_USLR/GLB_USBAA/GLB_USEQ). These ids are shared across the GVAR blocs (US, EA, JP, …). The euro FSAP standalones carry the same four series in their block:'global' columns but under their US-exogenous ids WTI_US/FCM10_US/FBAA_US/EQ_US (not GLB_*) — resolve by block, not by id.
    • ROW{DEM,INF,FIN}_<CC> — a bloc's 3 trade-weighted stars.
    • _<CC> on a domestic var (GDP_EA, RATE_DE) — the country's own series.
    • _US on an exogenous regressor in a euro FSAP standalone (GDPH_US, FCM10_US).

    Each variable also carries a block tag that drives the GVAR/fiscal topology (read at runtime by configureGVARTopology / deriveBlocIdx in gvarSolve.ts — never hardcode indices):

    block meaning where it appears
    domestic the bloc's own endogenous macro/financial vars every model
    fiscal the 4 fiscal drivers {FREV_GDP, FPEXP_GDP, EFFI, SFA} the 6 fiscal blocs (US/EA/JP/UK/CA/CN)
    row the 3 trade-weighted stars ROW{DEM,INF,FIN}_<CC> GVAR-coupled blocs (incl. endogenous US)
    global the 4 US-determined commons (oil, US 10y, US Baa, US equity) GVAR blocs (as GLB_*) and euro FSAP standalones (as WTI_US/FCM10_US/FBAA_US/EQ_US)
    external EA-shared exogenous regressors (EA policy rate, EA corp spread, member rate) euro FSAP standalones only
    us US-linked exogenous regressors (GDPH_US, JGDP_US, FCM2_US, …) euro FSAP standalones only

    A bloc is GVAR-active iff its spec.json carries block:'row' variables and it has a row in weights_24bloc.json. The US is endogenous iff its spec carries the 3 ROW*_US stars and the weights file has a US row — this sets the module flag US_ENDOGENOUS. See ARCHITECTURE_FOR_AI.md §5/§5a.

    isFinancial (boolean) flags financial variables (rates, spreads, FX, equity, house prices, leverage, oil). It drives display units and the COVID treatment (financials are kept observed through 2020Q2–Q4 while macro vars are masked).

    3. spec.json — the variable manifest¶

    The authoritative list of variables in fixed column order. The engine reads n and the per-variable metadata; it is index-agnostic otherwise (resolves fiscal/GVAR columns by id, not position).

    Field Type Shape Meaning
    n int scalar number of endogenous variables (= length of variables)
    variables array [n] one entry per variable, in column order
    variables[i].id string stable variable key (column identity)
    variables[i].name string human-readable label (for the UI dropdown)
    variables[i].transformation "log"|"lin" data-space (§2.1)
    variables[i].isFinancial bool financial-var flag (§2.3)
    variables[i].prior "RW"|"WN" prior-mean family on the own first lag (§3.1): "RW" = random walk (own-lag prior mean 1, the default); "WN" = stationary white-noise (own-lag prior mean 0 → mean-reverting). Only the three fiscal GDP-ratios (FREV_GDP,FPEXP_GDP,SFA) carry "WN"; everything else is "RW"
    variables[i].block string topology tag (§2.3)

    Example (US, n=41, abridged — first/fiscal/global/star columns):

    {
  "n": 41,
  "variables": [
    {"id":"GDPH","name":"Real Gross Domestic Product","transformation":"log","isFinancial":false,"prior":"RW","block":"domestic"},
    {"id":"LR","name":"Civilian Unemployment Rate: 16 yr +","transformation":"lin","isFinancial":false,"prior":"RW","block":"domestic"},
    {"id":"POLRATE","name":"Effective Federal Funds Rate (policy rate)","transformation":"lin","isFinancial":true,"prior":"RW","block":"domestic"},
    {"id":"GLB_USLR","name":"10-Year Treasury Note Yield","transformation":"lin","isFinancial":true,"prior":"RW","block":"global"},
    {"id":"FREV_GDP","name":"Federal Receipts (% of GDP)","transformation":"lin","isFinancial":false,"prior":"WN","block":"fiscal"},
    {"id":"FPEXP_GDP","name":"Federal Primary Expenditure (% of GDP)","transformation":"lin","isFinancial":false,"prior":"WN","block":"fiscal"},
    {"id":"EFFI","name":"Federal Effective Interest Rate (% p.a.)","transformation":"lin","isFinancial":true,"prior":"RW","block":"fiscal"},
    {"id":"SFA","name":"Stock-Flow Adjustment (% of GDP)","transformation":"lin","isFinancial":false,"prior":"WN","block":"fiscal"},
    {"id":"GLB_OIL","name":"WTI crude oil ($/bbl)","transformation":"log","isFinancial":true,"prior":"RW","block":"global"},
    {"id":"ROWDEM_US","name":"US RoW Demand","transformation":"lin","isFinancial":true,"prior":"RW","block":"row"},
    {"id":"ROWINF_US","name":"US RoW Inflation","transformation":"lin","isFinancial":true,"prior":"RW","block":"row"},
    {"id":"ROWFIN_US","name":"US RoW Fin","transformation":"lin","isFinancial":true,"prior":"RW","block":"row"}
  ]
}

    The US carries the 4 globals as its own endogenous vars (cols where block:'global') plus its 3 ROW*_US stars (cols 38–40, block:'row') — this is what makes the US endogenous. A RoW bloc (EA, n=24) instead carries the globals as weakly-exogenous commons that the solver pins to the US-determined path. A euro FSAP standalone (DE, n=23) carries no row block — its foreign side is external/us/global exogenous regressors.

    3.1 The prior field — "RW" (default) vs "WN" (stationary fiscal ratios)¶

    variables[i].prior selects the prior mean on each variable's own first lag in the GLP/Minnesota prior. It takes exactly two values:

    prior own-first-lag prior mean implied behaviour who carries it
    "RW" 1 (random walk — the historical default) the series can drift at the long end (a unit-root prior centre) every variable except the three fiscal GDP-ratios
    "WN" 0 (white noise — stationary) the series mean-reverts to its in-sample mean only FREV_GDP, FPEXP_GDP, SFA (in all 6 fiscal blocs)

    Why this exists. Under a pure "RW" prior the fiscal GDP-ratios (revenue, primary expenditure, stock-flow adjustment) inherited a unit-root centre and drifted over the 20-quarter projection — e.g. US federal receipts stuck at ~19% of GDP instead of reverting to their ~17.4% in-sample mean. Switching them to "WN" centres the own-lag prior at 0 so the ratios mean-revert to their sample means at the long horizon, which is the economically correct stationary behaviour for a bounded share-of-GDP. EFFI (the effective interest rate) stays "RW" — it is a slow-moving level, not a bounded ratio. This change is committed (6f0161a/22250be/e38673a) and is a no-op (byte-identical export) for the 44 non-fiscal blocs, which carry no "WN" variable.

    The mechanism (verified end-to-end in the estimator).

    1. Spec → pos. The MATLAB estimators read the per-variable prior string out of spec.json and build the index vector of stationary columns:
    2. pipeline/export_bvar_gvar24_loop.m (statCols=find(Spec.Prior(keep)=="WN")', on the kept domestic block; stars are appended after and are always "RW").
    3. pipeline/export_bvar_euro_fsap.m (statCols = find(Prior=="WN")'). Both resolve via a robust priorsfrom(...) helper (handles MATLAB's jsondecode dropping a prior field that is present on only some variables; absent → all-"RW").
    4. pos → cgl_bvarV5. statCols is threaded as the 3rd positional argument (stationary) of BVAR/MFBVAR/cgl_bvarV5.m, where pos=stationary (line 19) and passed on as 'pos',pos to bvarGLPmf.
    5. pos → prior mean = 0. In BVAR/MFBVAR/GLPreplicationWeb/bvarGLPmf.m the own-first-lag prior block is built from diagb: 
      diagb=ones(n,1);
diagb(pos)=0;            % zero the own-first-lag prior mean for the WN columns
b(2:n+1,:)=diag(diagb);  % default columns keep prior mean 1 (random walk)
      So a "WN" column gets own-lag prior mean 0 (stationary); every other column keeps 1 (the random-walk centre). For the 44 non-fiscal blocs pos=[], so diagb=ones(...) unchanged → identical to the historical default.
    6. Round-trip. On export each estimator re-stamps the per-variable prior back into spec.json from statCols (if any(statCols==i), v.prior='WN'; else v.prior='RW'), so the shipped spec always reflects the prior actually used.

    Deployed prior assignment (verified in the live specs — all 6 fiscal blocs identical):

    Variable block prior rationale
    FREV_GDP (revenue % GDP) fiscal "WN" bounded ratio → mean-revert
    FPEXP_GDP (primary expenditure % GDP) fiscal "WN" bounded ratio → mean-revert
    SFA (stock-flow adjustment % GDP) fiscal "WN" bounded ratio → mean-revert
    EFFI (effective interest rate % p.a.) fiscal "RW" slow-moving level, not a bounded share
    everything else (macro, financial, stars, globals) any "RW" random-walk default

    3.2 The isFinancial flag — coherence audit¶

    isFinancial (§2.3) gates two things: the chart display units and the COVID treatment — financial variables are kept observed through the 2020Q2–Q4 data-augmentation window, while macro variables (and the two macro stars ROWDEM/ROWINF) are masked and DK-completed. A miscoded flag therefore changes both the picture and the estimate.

    Two coherence fixes are baked into the deployed specs (verified across all blocs):

    • GLB_USEQ (US equity) and GLB_OIL (oil) are isFinancial:true everywhere. Both are market-priced and continued trading through 2020 — keeping them observed (not imputed) through the COVID window is the correct treatment, and it matches their euro-FSAP aliases EQ_US/WTI_US, which are also isFinancial:true. (The earlier GLB_OIL:isFinancial=false was incoherent and has been corrected.)
    • Canada's IP_CA is a log-LEVEL. It is now transformation:"log", isFinancial:false, block:"domestic" — rebuilt by chaining the discontinued FRED level index forward with growth to 2025Q4 so it reads as a proper 100·log(level) series (its q/q annualised growth is the 4·Δ display form), not a level-vs-growth mismatch.

    All 1048 variables across the deployed blocs were audited for transformation · prior · isFinancial coherence; a per-economy variable table now lives in the user manual.

    4. companion_mean.json — posterior-mean VAR¶

    The posterior-mean reduced-form VAR(lags) coefficients and residual covariance. The constant (intercept) row is first — this matches kalman.ts's companion construction. Do not transpose.

    Field Type Shape Meaning
    betahat number[][] [k][n], k = n·lags + 1 coefficient matrix; row 0 = intercept, then lags blocks of n own/cross lags; column j = equation for variable j
    sigmahat number[][] [n][n] residual covariance (symmetric, PD)
    n int scalar number of variables (matches spec.n)
    lags int scalar VAR lag order

    k = n·lags + 1 (US: 41·3 + 1 = 124; EA: 24·3 + 1 = 73).

    Example (US shapes & first values):

    {
  "betahat": [
    [6.12187, -0.742157, -2.92772, "… 41 cols"],     // row 0 = intercept
    [0.760955, 0.0994946, -0.260185, "… 41 cols"],  // row 1 = lag-1 of var 0, etc.
    "… 124 rows total (1 + 41·3) …"
  ],
  "sigmahat": [ ["… 41×41 …"] ],
  "n": 41,
  "lags": 3
}

    5. companion_draws.json — flat posterior draws¶

    The full posterior sample, stored as flat row-major arrays (boxed number[] in JSON; converted to Float64Array on load). Lazy-loaded; the US file is ~34 MB.

    Field Type Shape Meaning
    n_draws int scalar number of posterior draws (currently 500)
    n int scalar variables
    lags int scalar lag order
    beta_shape int[3] [k, n, n_draws] shape of the flattened beta (k = n·lags + 1)
    beta number[] k·n·n_draws flattened coefficient draws
    sigma_shape int[3] [n, n, n_draws] shape of the flattened sigma
    sigma number[] n·n·n_draws flattened covariance draws

    Flat indexing (column-major within the 3-D shape, matching ARCHITECTURE_FOR_AI.md §6):

    beta[ draw*k*n  + j*k + i ]     // draw, equation column j, coef row i
sigma[ draw*n*n + j*n + i ]     // draw, (i,j) of the n×n covariance

    Example (US):

    {
  "n_draws": 500,
  "n": 41,
  "lags": 3,
  "beta_shape": [124, 41, 500],
  "beta": [8.04724, 0.751567, 0.174671, "… 2 542 000 doubles …"],
  "sigma_shape": [41, 41, 500],
  "sigma": [0.104776, 0.0609497, 0.143155, "… 840 500 doubles …"]
}

    6. baseline.json / baseline_levels.json — the baseline fan¶

    The unconditional baseline forecast and its predictive bands, over the projection horizon H (currently 20 quarters). Two files, identical shape, different space:

    • baseline.json — display/growth space (§2.1).
    • baseline_levels.json — level space (feeds the fiscal block, the GVAR solver, and the level-display fans).
    Field Type Shape Meaning
    mean number[][] [H][n] central path
    q05,q16,q84,q95 number[][] [H][n] predictive-band quantiles (5/16/84/95%)
    q50 number[][] [H][n] optional median; present in baseline_levels.json only, and only for the 16 euro FSAP standalones (AT, BE, DE, EE, ES, FI, FR, GR, IE, IT, LT, LU, NL, PT, SI, SK). No baseline.json (display-space) carries q50, and the GVAR blocs (US/EA/JP/…) carry it in neither file — read defensively
    horizon int scalar H (= number of forecast quarters, e.g. 20)

    Example (US, H=20, n=41):

    {
  "mean": [ [1.91073, 1.87189, -1.45722, "… 41 cols"], "… 20 rows" ],
  "q16":  [ ["…"], "…" ],
  "q84":  [ ["…"], "…" ],
  "q05":  [ ["…"], "…" ],
  "q95":  [ ["…"], "…" ],
  "horizon": 20
}

    baseline_levels.json has the same keys but in level space — e.g. mean[0] = [1009.29, 972.576, 680.448, …] (100·log levels for log vars).

    7. historical.json / historical_levels.json — recent actuals¶

    The last ~40 quarters of realised data, used to anchor the charts. Two files, same shape, different space (display/growth vs level).

    Field Type Shape Meaning
    data number[][] [n_periods][n] realised values, oldest → newest
    n_periods int scalar number of historical quarters (currently 40)

    Example (US):

    {
  "data": [
    [2.31139, 3.00848, 14.3557, "… 41 cols"],   // oldest quarter
    "… 40 rows …",
    [0.481089, 1.9089, -4.6351, "… 41 cols"]     // newest quarter
  ],
  "n_periods": 40
}

    historical.json growth is already the 4·Δ(100·log level) form for log vars (no further differencing in the chart). historical_levels.json is the matching level path.

    8. transformations.json — per-variable space¶

    A redundant, fast lookup of each variable's data space, aligned to spec.variables column order. (The same information is in spec.variables[i].transformation.)

    Field Type Shape Meaning
    transformations string[] [n] "log" or "lin" per variable column

    Example (US, abridged):

    {"transformations":["log","log","log","log","log","log","log","log","log","log","log","log","log","log","lin","log","lin","log","log","lin","lin","lin","lin","lin","lin","lin","log","log","log","log","lin","log","log","lin","lin","lin","lin","log","lin","lin","lin"]}

    9. fiscal_meta.json — fiscal-block parameters (6 blocs only)¶

    Shipped only by the 6 fiscal economies (US, EA, JP, UK, CA, CN); a 404 for any other economy is expected and yields null. It supplies the debt recursion's initial stock, default SFA, and the column indices the recursion reads from the level path (computeFiscalBlock in fiscalAccounting.ts). See fiscal-block-methodology.md and ARCHITECTURE_FOR_AI.md §7 for the math.

    Field Type Meaning
    d0 number initial debt-to-GDP ratio (%) at the projection seam
    sfaDefault number default stock-flow adjustment (% of GDP) — the historical mean of the estimated SFA
    idx object column indices into the level path (spec/*_levels column order)
    idx.gdp int real-GDP column
    idx.deflator int GDP-deflator column
    idx.revenue int FREV_GDP (revenue % GDP) column
    idx.primaryExp int FPEXP_GDP (primary-expenditure % GDP) column
    idx.effRate int EFFI (effective interest rate, % p.a.) column
    idx.sfa int SFA (stock-flow adjustment, % GDP) column

    The legacy idx.debtService key was removed; idx.sfa (the estimated SFA column) replaced it. With idx.sfa set and idx.debtService unset, computeFiscalBlock takes the effective-rate interest form and reads the per-quarter SFA from idx.sfa. Do not re-introduce debtService.

    Deployed values (read from the live files; NEVER hardcode indices — they are positional and DRIFT, see the gotcha below). d0 is shown rounded for readability — the file stores it full precision; sfaDefault is round(.,4):

    Bloc n d0 (file) sfaDefault idx {gdp,deflator,revenue,primaryExp,effRate,sfa}
    US 41 122.56815 1.7518 {0, 7, 33, 34, 35, 36}
    EA 24 87.84509550831034 0.325 {0, 6, 17, 18, 19, 20}
    JP 23 192.86944931521631 2.233 {0, 7, 16, 17, 18, 19}
    UK 25 96.24124260944826 3.1487 {0, 8, 18, 19, 20, 21}
    CA 25 100.1611274546207 3.191 {0, 8, 18, 19, 20, 21}
    CN 20 94.33130080334892 4.017 {0, 1, 13, 14, 15, 16}

    🔴 STALE-POSITIONAL-INDEX gotcha — the porter does NOT regenerate fiscal_meta.json. These six idx values are positional offsets into the level path, but pipeline/port_gvar_exports.py ports fiscal_meta.json verbatim. So any change to a bloc's variable ORDER or COUNT (e.g. the 33-bloc GVAR-EM refit, which inserted variables ahead of the fiscal block in every spec except US) silently re-points the indices at the wrong columns. Concretely: the pre-refit EA {0,6,14,15,16,17} pointed effRate at the equity index (~537), and d·(1+effRate/400) exploded the EA debt path. A porter (or anyone re-ordering variables) MUST run node pipeline/fix_fiscal_meta_indices.mjs after porting — it re-resolves all six indices BY VARIABLE ID (FREV_GDP/FPEXP_GDP/EFFI/SFA, gdp=0, deflator=first HICP|CPI|JGDP|PGDP|DEFL) from the deployed spec.json and rewrites the file (preserving d0/sfaDefault; idempotent). The fiscal-meta-indices.test.ts canary blocks a deploy whose indices are stale (asserts every bloc's derived debt path stays bounded).

    Example (US):

    {"d0":122.56815,"sfaDefault":1.7518,"idx":{"gdp":0,"deflator":7,"revenue":33,"primaryExp":34,"effRate":35,"sfa":36}}

    CN's deflator index is 1 (not 7). Since the 2026-06 surgical CN-only re-estimate, idx.gdp (GDP_CN) is real (spec.json var 0 = "CN Real GDP …", fiscal_meta.json idx.gdpIsNominal = false), and CN has been removed from NOMINAL_GDP_BLOCS, so the GVAR DEM contribution now reads gdp directly. The fiscal recursion gQ = Δ(idx.gdp) + Δ(idx.deflator) is now the correct real-GDP + deflator nominal growth, matching US/EA/JP/UK/CA — the prior nominal-GDP_CN double-count is resolved. The fix was CN-only; the other 32 blocs are byte-identical. The fiscal-block column counts above (n) reflect the live specs; treat them as data, not constants.

    10. weights_24bloc.json — GVAR trade-weight matrix & topology source¶

    The single source of GVAR topology: the bloc list and the row-normalised bilateral trade-weight matrix that builds each bloc's trade-weighted stars and the cross-country coupling. The filename is stale — it holds 33 blocs, not 24 (US first, with a US row that activates the endogenous-US coupling). Imported directly by gvarSolve.ts (import weights24 from '…/weights_24bloc.json') and consumed by configureGVARTopology.

    Field Type Shape Meaning
    source string provenance (IMF IMTS bilateral trade, SDMX 2.1)
    indicators string[] [2] trade indicators used (XG_FOB_USD, MG_CIF_USD)
    euro_area_code string aggregation code for the EA bloc ("G163")
    window string averaging window ("2021-2023 average")
    blocs string[] [33] the candidate bloc list, US first (= GVAR_BLOC_CANDIDATES)
    bloc_codes object 50 keys code → metadata map (display)
    note string human note (records that US is now endogenous)
    weights object 33 keys the matrix: weights[receiver][partner] = w, zero diagonal, each receiver row sums to 1 over its partners

    The weights matrix is keyed by receiver bloc (33 rows, including a US row); each row maps partner code → weight and is row-normalised (zero diagonal, sums to 1). At config time configureGVARTopology renormalises each row over the blocs that are actually deployed/active.

    11. coupling_default.json.gz — precomputed GVAR coupling¶

    The gzipped precomputed cross-country coupling: the linear-system matrix M and its BLAS-computed inverse (I − M)⁻¹ for the empty/baseline pin structure. Building/inverting M at this scale is many minutes in-browser (the in-JS O(dim³) inverse at dim≈2060 is ~46 min), so shipping this artifact is mandatory for interactivity, not optional. Decompressed in-browser via the native DecompressionStream (gvarLoader.ts); a plain .json fallback is tried for environments without it.

    Field Type Shape Meaning
    modelSig string model signature keying the artifact to the deployed data (see below)
    pinStructure string the pin-structure signature this artifact was built for — "" (empty/baseline) for the shipped one
    dim int scalar augmented-state dimension (currently 2060 under endogenous US)
    rhoSpectral number scalar spectral radius ρ(M) (the live coupling_default.json.gz reads 0.924041 < 1; it moves slightly with each refit — what matters is < 1)
    M number[][] [dim][dim] the coupling matrix (6-sigfig)
    invImM number[][] [dim][dim] (I − M)⁻¹, the BLAS-computed inverse (6-sigfig)

    modelSig is built by gvarSignature(raw) (gvarLoader.ts): for each qualified bloc c, "{c}:{n}_{lags}_{horizon}_{b0}_{bM}_{s0}" where b0 = betahat[0][0], bM = betahat[n·lags][n-1], s0 = sigmahat[0][0] (each in toExponential(6) form), joined with "|". So a change to any bloc's data changes modelSig and invalidates the artifact — it must be regenerated. At solve time (runGvarSolve.ts) the hydrated inverse is used only if art.modelSig === gvarSignature(raw) and the request's pin-structure signature matches art.pinStructure; otherwise the solver rebuilds.

    Header example (values are real; matrices abridged):

    {
  "modelSig": "US:41_3_20_6.121870e+0_-8.370660e-2_8.919980e-2|EA:24_3_20_2.551660e+0_-1.208040e-1_1.509770e-1|…",
  "pinStructure": "",
  "dim": 2060,
  "rhoSpectral": 0.924041,
  "M": [ ["… 2060 cols …"], "… 2060 rows …" ],
  "invImM": [ ["… 2060 cols …"], "… 2060 rows …" ]
}

    Regenerate (mandatory after any data change) from app/:

    npx vite-node scripts/precompute_gvar_coupling.mjs

    This builds M for the empty pin structure, inverts it via MATLAB BLAS (invert_coupling.m), and writes the gzip. gvar-coupling-artifact.test.ts asserts the shipped .gz hydrates and matches a fresh build (a canary that blocks deploy on drift).

    Spectral radius by panel size (context, not a contract): 6-bloc pilot ρ≈0.59; 24-bloc frozen-anchor ρ≈0.726 (dim≈1920); 33-bloc endogenous-US ρ=0.924041 (dim=2060, the deployed coupling_default.json.gz). The exact value moves slightly with each refit (earlier artifacts read ~0.89); what is load-bearing is that it stays comfortably < 1. The frozen-anchor figures no longer describe the live artifact.

    12. Validating a freshly ported bloc¶

    A quick sanity pass when adding or re-porting a bloc (paths absolute under app/public/data/bvar/<CC>/):

    1. spec.n === companion_mean.n === companion_draws.n, and length(spec.variables) === n.
    2. companion_mean.betahat is [n·lags+1][n]; sigmahat is [n][n].
    3. companion_draws.beta_shape === [n·lags+1, n, n_draws]; sigma_shape === [n, n, n_draws]; length(beta) === product(beta_shape); length(sigma) === product(sigma_shape).
    4. baseline.mean and baseline_levels.mean are [horizon][n] with all of q05/q16/q84/q95 present (q50 optional).
    5. historical.data and historical_levels.data are [n_periods][n].
    6. transformations.transformations has length n and matches spec.variables[i].transformation.
    7. A fiscal bloc additionally ships fiscal_meta.json with all six idx keys in range [0, n) AND each idx resolving to the correct variable id (idx.revenue → FREV_GDP, etc.). After any re-port, run node pipeline/fix_fiscal_meta_indices.mjs — the porter does not regen these positional indices (the STALE-POSITIONAL-INDEX gotcha, §9).
    8. After porting, regenerate coupling_default.json.gz (§11) so modelSig matches.

    Then run the green gate (npx tsc --noEmit && npm test && npm run test:heavy) — the canary tests (gvar-coupling-artifact, gvar-convergence, gvar-derived-topology, scenario-scale-convention, fiscalAccounting) encode the contracts above.

    13. Out-of-sample evaluation contracts (the em-bayesian-gvar-eval harness)¶

    Scope note. The JSON contracts in §§1–12 are read by the deployed app. The three contracts below are read/written by the separate empirical-evaluation repository em-bayesian-gvar-eval (out-of-sample forecast horse-race + endogenous-US spillover analysis). They live in em-bayesian-gvar-eval/results/ and are not shipped in app/public/data/. The full design is em-bayesian-gvar-eval/docs/EMPIRICAL_PROTOCOL.md (§§0–2). Results are preliminary/forthcoming; the contracts here are stable.

    13.1 weights_asof_<startY>_<endY>.json — rolling as-of trade weights cache¶

    The horse-race is a recursive expanding-window scheme over origins t0 = 2010Q1 … 2024Q4 (2020 COVID origins dropped). A hard look-ahead invariant (EMPIRICAL_PROTOCOL.md §0 invariant #2, decision D11) forbids using the shipped 2021–2023 weights_24bloc.json/weights_full.json at a pre-2020 origin — that would inject future trade structure. Instead the driver resolves, per origin, a leak-free as-of weights file whose IMTS averaging window ends ≤ year(t0), built by matlab/weights/rolling_asof_weights.m (a thin wrapper on the main repo's pipeline/build_trade_weights_full.py — same IMTS pull, EA=G163, mirror fallback for embargoed reporters, <0.05% partner drop + renormalization). Files are cached under results/weights_cache/.

    Two modes (env OOS_WEIGHTS_MODE):

    • fixed (default) — a single pre-sample window (e.g. 2005-2009) used at every origin: provably leak-free (biased toward the past, never the future).
    • rolling — a 3-yr window ending the calendar year before year(t0) (strictly < year(t0), so even Q1 origins use no same-year trade); one IMTS pull per origin-year, cached. This is the rigor robustness variant.

    Schema — identical top-level shape to weights_24bloc.json (§10) but on the headline 6-bloc panel and with a window-specific window string:

    Field Type Shape Meaning
    source string provenance (IMF IMTS, SDMX 2.1 — same string as §10)
    indicators string[] [2] ["XG_FOB_USD","MG_CIF_USD"]
    euro_area_code string "G163"
    window string the as-of window for this file, e.g. "2016-2018 average" — distinguishes cache entries and is copied into each forecast JSON's meta.weightsWindow
    blocs string[] [6] headline panel ["US","EA","JP","UK","CA","CN"] (US first)
    bloc_codes object 6 keys code → metadata map
    note string records the row-normalization, US-as-closed-anchor, <0.05% drop, mirror fallback
    weights object 5 keys (fixed/late files) or 6 keys weights[receiver][partner] = w, zero diagonal, each row sums to 1 over its partners

    Receiver-row count is window-dependent — read defensively. The 2005-2009 (and other early/fixed) files carry 6 receiver rows including a US row (US endogenous, has its own RoW block); some windows (e.g. the 2021-2023 cache entry) carry 5 rows (US is the closed anchor / partner-only at that vintage, no US receiver row). Do not assume a US receiver row exists — resolve by iterating Object.keys(weights), exactly as the GVAR topology code does for weights_24bloc.json. The cached files are 6-sigfig per the IMTS builder; rows sum to 1 up to rounding (e.g. 0.999999).

    Example (weights_asof_2016_2018.json, abridged — note the 5-bloc weights map under a 6-bloc blocs list, because US is partner-only at this window):

  "source": "IMF IMTS (International Trade in Goods, by partner country) — api.imf.org SDMX 2.1, flow IMF.STA:IMTS(1.0.0)",

  "euro_area_code": "G163",
  "window": "2016-2018 average",
  "blocs": ["US", "EA", "JP", "UK", "CA", "CN"],
  "bloc_codes": { "US": {"…": "…"}, "EA": {"…": "…"} },
  "note": "Row-normalized over the other blocs (zero diagonal). US is the closed anchor (no RoW block) but is a partner for everyone. Negligible partners (<0.05%) dropped and the row renormalized. Mirror fallback for embargoed reporters (RU, TW).",
  "weights": {
    "EA": {"US":0.324999, "JP":0.066389, "UK":0.293980, "CA":0.030158, "CN":0.284473},
    "JP": {"US":"…", "EA":"…", "UK":"…", "CA":"…", "CN":"…"},
    "UK": {"…": "…"}, "CA": {"…": "…"}, "CN": {"…": "…"}
  }

    13.2 forecasts/<model>/<origin>.json — per-origin predictive draw clouds¶

    For each (model, origin) the driver (matlab/oos_driver.m → matlab/harvest_forecasts.m) writes one file holding the full predictive draw cloud (not just the mean) for the headline targets, sliced from the estimator's native Dforecast(H, n, nDraws−nBurn). Models are one directory each: emgvar_endo, emgvar_frozen, rw, rw_clim (climatology), arp, sbvar (single-country BVAR, ablation iii), cgvar_a (textbook OLS GVAR, iv-a), cgvar_b (single-pass Bayesian GVAR, iv-b), dfm.

    Field Type Shape Meaning
    model string one of the model tags above
    origin string origin quarter, "YYYY-Qn" (e.g. "2018-Q3")
    blocs object 6 keys per-bloc forecasts (headline panel US/EA/JP/UK/CA/CN)
    blocs[c] object 3 keys one entry per target: GROWTH, INFL, RATE
    blocs[c][t].draws number[][] [m][nh] the predictive cloud: m draws × nh requested horizons, in scored space (4·Δ for growth/inflation, level for rate). m is model-dependent: the Bayesian DK-smoother models (emgvar_endo/emgvar_frozen/cgvar_b/sbvar) keep the post-burn cloud m = nDraws−nBurn (≈100 at nDraws=200); the parametric/OLS/bootstrap benchmarks (rw/rw_clim/arp/cgvar_a/dfm) emit the full m = nDraws (200). Read m = draws.length, never assume it
    blocs[c][t].mean number[] [nh] predictive mean per horizon
    meta object run metadata (see below)

    meta fields (verified in the harvested files):

    meta key Type Meaning
    nDraws int requested NDRAWS (e.g. 200; the kept cloud size m is model-dependent — see the draws row above)
    lags int VAR lag order (3)
    emIters int EM star-loop iterations to convergence at this origin (1 for the MAXIT=1 single-pass ablations; 0 for the non-iterative benchmarks rw/arp/cgvar_a)
    covidInSample bool whether 2020Q2–Q4 ≤ t0 (so the COVID-as-missing mask applied in-sample)
    weightsWindow string the as-of weights window used (copied from §13.1's window)
    seed number per-origin RNG seed (reproducibility; cgl_bvarV5 calls rng default then offsets)
    usEndogenous bool whether the US joined the coupled loop with its own ROW*_US block at this origin (present on the EM-GVAR-family files; may be absent on the simple benchmarks)
    tol number star-convergence tolerance (0.05 sweep / 0.03 final)
    horizon int the estimator's full H (20)
    horizons int[] the scored horizons, in column order, e.g. [1,2,4,8]

    The draws/mean columns are packed DENSELY by REQUESTED horizon, not by a dense 1:H layout. Column j of every draws[k]/mean is horizon meta.horizons[j] — so column 0 = h=1, column 1 = h=2, column 2 = h=4, column 3 = h=8 (for horizons=[1,2,4,8]). Always index horizons through meta.horizons; never assume column j ⇒ h = j+1. (harvest_forecasts.m records this mapping precisely for that reason.)

    Example (forecasts/emgvar_endo/2018Q3.json, abridged — real meta, draws/mean shapes shown for one target):

    {
  "model": "emgvar_endo",
  "origin": "2018-Q3",
  "blocs": {
    "US": {
      "GROWTH": {
        "draws": [ [1.91025, 2.44701, 1.69526, -0.78120], "… 100 draws × 4 horizons" ],
        "mean":  [3.07970, 2.39260, 1.39190, 0.79140]
      },
      "INFL": { "draws": [ ["…"], "…" ], "mean": ["…"] },
      "RATE": { "draws": [ ["…"], "…" ], "mean": ["…"] }
    },
    "EA": { "…": "…" }, "JP": {}, "UK": {}, "CA": {}, "CN": {}
  },
  "meta": {
    "nDraws": 200, "lags": 3, "emIters": 2, "covidInSample": false,
    "weightsWindow": "2015-2017 average", "seed": 1631573281,
    "usEndogenous": false, "tol": 0.05, "horizon": 20, "horizons": [1, 2, 4, 8]
  }
}

    The scoring harness (eval/score.py, eval/tables.py) ingests these clouds plus realizations.json (§13.3) and emits RMSE/MAE, CRPS (primary density score), parametric log-score, PIT/coverage, and the Giacomini–White / Clark–West / Diebold–Mariano differentials (with Romano–Wolf + Model Confidence Set for multiplicity), per EMPIRICAL_PROTOCOL.md §1.7.

    13.3 realizations.json — the truth panel for scoring¶

    The realized target values read from the full (untruncated) 2025Q4 vintage, written by matlab/build_realizations.m and indexed by [bloc][target][date] → scalar (the realized value in the same scored space as the forecasts: 4·Δ growth/inflation, level rate). The harness reads realizations[c][t][t0+h] for each scored (origin, horizon).

    Field Type Shape Meaning
    (top level) object 6 keys US/EA/JP/UK/CA/CN
    [c] object 3 keys GROWTH, INFL, RATE
    [c][t] object date → number "YYYY-Qn" → realized scalar in scored space

    Example (abridged):

    {
  "US": {
    "GROWTH": { "…": "…", "2023-Q4": 3.362060, "2024-Q1": "…" },
    "INFL":   { "2023-Q4": "…" },
    "RATE":   { "2023-Q4": "…" }
  },
  "EA": { "…": "…" }
}

    Scored-space consistency is load-bearing. Realizations are computed the same way as the forecast draws (4·Δ(100·log level) for growth/inflation from the gdp/ price columns resolved by the estimator's findcol order; the rate level as-is). Since the 2026-06 CN-only re-estimate, CN's GDP is real (no longer deflated by subtraction); growth is scored as 4·Δ(100·log GDP_CN) directly, the same convention as the other blocs, on both the forecast and realization side (EMPIRICAL_PROTOCOL.md §0).

    External-forecast import + full-session files (client-side, not shipped data)¶

    Two user-facing JSON contracts (not part of public/data/), consumed and produced entirely in the browser. Documented here so a third party can generate them programmatically.

    External-forecast import (ControlPanel → External forecasts → Import forecasts)¶

    One source, one bloc. Values are in the chart's display units (q/q-annualised % for log growth/inflation vars, the level in % for lin rates, the index level for level-displayed vars). Parsed by app/src/lib/compute/externalForecast.ts::parseExternalForecastFile; each point becomes a soft condition whose measurement variance is set by the per-source trust slider (predVar·(1−t)/t; trust 1 = exact pin). Quarters are YYYYQn and must fall in the forecast horizon.

    {
  "source": "WEO",
  "vintage": "2026Q1",
  "bloc": "US",
  "forecasts": {
    "GDPH":    { "2026Q1": 2.4, "2026Q2": 2.2 },
    "POLRATE": { "2026Q1": 3.6 }
  }
}

    A value may also be { "value": 2.4 }; non-finite/null points are dropped. A cell the user has pinned by hand always overrides an imported one. Template: public/data/sample_external_forecast.json.

    Full-session file (Export session / Import session)¶

    schema: "mfm-session/1". One portable bundle of the entire view — all countries' conditioning, GVAR cross-country pins, long-run anchors + solve mode, imported external forecasts + trust, and the display settings, with provenance (exportedAt). Produced/consumed by useStore.exportSession / importSession. Unlike the per-scenario Excel export, nothing is silently dropped.

    {
  "schema": "mfm-session/1",
  "exportedAt": "2026-07-06T…Z",
  "country": "US",
  "scenarios": [ … ],
  "activeScenarioId": "s1",
  "gvarPins": { "s1": { "US": { "conditionedVars": [ … ], "conditioningPaths": { … } } } },
  "longRunAnchorsByCountry": { "US": [ { "variableId": "GDPH", "targetValue": 2.0, … } ] },
  "longRunAnchorMode": "perVariable",
  "externalForecasts": { "US": { "GDPH": { "2026Q1": { "value": 2.4, "source": "WEO" } } } },
  "externalTrust": { "WEO": 0.7 },
  "externalSources": [ { "source": "WEO", "bloc": "US", "vintage": "2026Q1", "count": 6 } ],
  "display": { "growthDisplay": "qoq", "vsBaselineMode": "level", "inputMode": "levels" }
}

    Shipped WEO forecasts (Load WEO button)¶

    public/data/weo_forecasts.json, built by pipeline/build_weo_forecasts.py from the IMF DataMapper API (NGDP_RPCH real GDP growth, PCPIPCH avg CPI inflation). Real published WEO numbers, tagged with the retrieval date. WEO is annual; each year's figure is replicated to that year's four quarters (the app applies them as soft pins and lets the model shape the intra-year path). WEO has no policy rates and nothing quarterly, so only each bloc's growth + inflation variables are populated.

    The Excel template (Export template) uses the same variable-id-keyed layout as the import: a Forecasts sheet with quarter labels in column 1 and variable ids across the header, prefilled with the bloc's baseline, plus a Notes sheet naming the units. The importer accepts that XLSX or the JSON.
