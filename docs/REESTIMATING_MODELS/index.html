    How to update / re-estimate the models¶

    This is the operational runbook for re-estimating the GVAR macro blocs and rebuilding the fiscal block, then porting, verifying, and deploying. It documents the current state of the platform: an endogenous-US, 33-bloc GVAR estimated by an iterative GVAR-EM loop, plus a 4-variable fiscal block (with an exact-residual SFA) for the six fiscal economies.

    Scope. This covers re-estimation and data refresh of the existing blocs. To add a brand-new country or RoW bloc, see ADDING_A_COUNTRY.md (this runbook reuses the same port / verify / deploy gate). For the live methodology, see ARCHITECTURE_FOR_AI.md and fiscal-block-methodology.md.

    The deployed tool is live at the /bvar-scenario path of the academic website, hosted on GitHub Pages (the matyasfarkas/starter-hugo-academic Hugo Academic site).

    Golden rules. Exact MATLAB replication when porting (no shortcuts). Use real observed data, never synthetic placeholders. Never deploy without a green gate. Do not hand-edit the GVAR topology (IDX/W/ROW_COUNTRIES in gvarSolve.ts) — it self-configures from the deployed specs + weights.

    Bloc set is data-driven from app/public/data/bvar/gvar/weights_24bloc.json (blocs field, US first; the 24 in the filename is historical — it currently lists 33 blocs). The US is the first entry and has a weights.US row, which is what makes it endogenous.

    1. Re-estimate the GVAR macro blocs (GVAR-EM loop)¶

    1.1 What the loop does¶

    pipeline/export_bvar_gvar24_loop.m is the canonical GVAR-EM estimator:

    1. For each non-US bloc, build its 3 RoW "star" vars (ROWDEM/ROWINF/ROWFIN) = trade-weighted averages of partners' real-GDP growth / inflation / short rate, renormalised over the partners present each quarter (unbalanced panel). COVID + no-partner quarters start NaN.
    2. Estimate each bloc by cgl_bvarV5([domestic | 3 stars]) — the COVID-as-missing Gibbs sampler; the DK smoother fills the missing star + COVID quarters → a completed matrix per bloc.
    3. Loop (Jacobi): recompute every bloc's stars from partners' completed data, re-estimate all blocs in parallel (parfor), until the foreign components converge (max |Δstar| < TOL).

    Key facts encoded in the script (do not change without re-validating):

    • US_ENDOGENOUS=1 (env). With a weights.US row present, the US is folded into the convergence loop and re-estimated with its own 3 RoW stars (n=41: domestic + 3 ROW*_US). It drives RoW and receives RoW→US feedback. A no-stars bootstrap still runs first only to seed partners' iter-1 stars (it is discarded, not exported).
    • COVID window is the argmin of ΔGDP restricted to 2019Q4–2021Q1 (a global argmin misfires, e.g. KR's 2008Q4 GFC drop), then ± quarters around it are NaN'd → DK-imputed.
    • MAXIT=12, TOL=0.03 (MC noise floor ≈0.13). MAXIT=1 ⇒ single-country (separated): one-shot historical stars, no coupling iteration.
    • Scale traps (already correct in-script, listed so you don't "fix" them): stars are 4·Δ(100·log) not 400·; the completed panel is mean(Xcond,3); CN GDP is real (since the 2026-06 CN-only re-estimate; NOMINAL_GDP_BLOCS empty) → read directly, not deflated by subtraction; leading star NaN is back-filled; the _US column is dropped (no-op on _glob inputs); the short rate is optional (CH/NO have none → excluded from partners' ROWFIN); AR hyperinflation excluded from partners' INF/FIN (kept in DEM).

    1.2 Run it¶

    From the repo root (MATLAB with the Parallel Computing Toolbox; the loop uses parfor):

    # Full 33-bloc endogenous-US run (the production estimate). Hours, not minutes.
US_ENDOGENOUS=1 matlab -batch "run('pipeline/export_bvar_gvar24_loop.m')"

    Useful env overrides (for fast subset validation before committing the multi-hour full run):

    # Fast smoke: 3 blocs, few draws, 1 iteration (single-country, no coupling)
BVAR_GVAR_BLOCS="US EA JP" NDRAWS=200 MAXIT=1 US_ENDOGENOUS=1 \
  matlab -batch "run('pipeline/export_bvar_gvar24_loop.m')"
    Env Default Meaning
    US_ENDOGENOUS unset (anchor) 1 ⇒ US is a coupled bloc (requires weights.US). Set it for production.
    BVAR_GVAR_BLOCS all blocs in weights file space-separated subset, US first
    NDRAWS 1000 (burn = 500) MCMC draws per bloc
    MAXIT 12 max convergence sweeps (1 = separated)
    TOL 0.03 convergence tol on max |Δstar|
    GVAR_WEIGHTS_FILE weights_full.json else weights_24bloc.json which weights/bloc list to use

    Outputs. Each bloc's full deployed JSON contract to /tmp/gvar24_export/<CC>/ (spec, companion_mean, companion_draws, historical, historical_levels, baseline, baseline_levels, transformations). The full estimation state is also saved to /tmp/gvar24_workspace.mat before export, so an export-only bug is a cheap re-export, not a re-estimation.

    Watch the console. Each iteration prints max star change vs tol; convergence prints CONVERGED. Each export prints n, draws, and the 1-quarter GDP growth (GDP1q) — confirm it is a sane number, not NaN or a collapsed flat line.

    If a bloc returns a singular / NaN SIGMA, the data is the cause (a leading NaN in a discontinued series, or near-collinear national-accounts identities). Fix the raw DataRaw_<CC>_glob.json, not the estimator. With COVID imputed the empirical-Bayes lambda optimum is well-behaved (~0.24–0.56); a fixed lamfix is not needed (and pinning one with noc/sur both fixed makes lambda degenerate to NaN).

    2. Rebuild the fiscal block (4-variable, exact-residual SFA)¶

    The fiscal block carries 4 estimated BVAR variables for US, EA, JP, UK, CA, CN:

    ID Meaning Units
    FREV_GDP revenue / GDP × 100 % of GDP (annual)
    FPEXP_GDP primary expenditure / GDP × 100 % of GDP (annual)
    EFFI effective interest rate = net interest / lagged gross debt × 100, annualised % p.a.
    SFA stock-flow adjustment % of GDP (annual)

    SFA is the exact residual of the app's debt recursion. The build script rearranges the exact recursion in app/src/lib/compute/fiscalAccounting.ts (computeFiscalBlock, ~line 169) onto the observed gross-debt/GDP path so that the BVAR's SFA column closes the recursion to machine precision (it captures the federal-flows-vs-gross-debt perimeter seam / intragovernmental holdings — the deficit–debt discrepancy):

    d_t = d_{t-1}·(1 + EFFI/400)/(1 + gNomQ/100) − pb/4 + sfa/4        (% of GDP)
pb     = FREV_GDP − FPEXP_GDP
gNomQ  = 100·log(realGDP_t/realGDP_{t-1}) + 100·log(deflator_t/deflator_{t-1})   (% per quarter)
⇒ SFA_t = 4·(d_obs_t − d_{t-1} − snowball_t + pb_t/4)

    The crucial subtlety: gNomQ is read off the same glob columns the app reads (idx.gdp/idx.deflator, stored as 100·log), as a log growth — not a simple nominal SAAR growth. Building it the app's way is what drives the closure error to < 1e-6 (an interpolated COVID value can never equal the residual, so COVID quarters are set NaN for the Gibbs to impute).

    fiscal_meta/<CC>.json carries idx = {gdp, deflator, revenue, primaryExp, effRate, sfa} (note: there is no debtService — DSVC was reverted to EFFI), d0, and sfaDefault. The debt/GDP projection now carries the forecast SFA forward (idx.sfa set ⇒ the recursion reads the estimated SFA level path with its own dynamics + bands, instead of defaulting SFA to 0).

    2.1 Run the builds¶

    One script per fiscal economy. They are data-build only (no estimation, no network): each edits only the SFA column of DataRaw_<CC>_glob.json in place (everything else byte-identical), sets the COVID window to NaN, runs the closure check, and writes sfaDefault (the non-COVID, ex-t0 SFA mean) into fiscal_meta/<CC>.json.

    cd "<repo root>/pipeline"
python3 build_fiscal_us_4var.py
python3 build_fiscal_ea_4var.py
python3 build_fiscal_jp_4var.py
python3 build_fiscal_uk_4var.py
python3 build_fiscal_ca_4var.py
python3 build_fiscal_cn_4var.py

    The gate is automatic and hard: each script asserts the one-step closure |d_obs_t − recon_t| < 1e-6 for every non-COVID quarter t≥1 and aborts on failure. It prints backcastMaxErrPP (must be ~1e-13), the SFA range, and the chosen sfaDefault.

    Current sfaDefault (≈ non-COVID SFA mean, % of GDP) — sanity benchmarks after a rebuild:

    US EA JP UK CA CN
    sfaDefault 1.75 0.33 2.23 3.15 3.19 4.02

    Order: rebuild the fiscal SFA before re-estimating the GVAR if the fiscal data changed — the SFA column lives in the same DataRaw_<CC>_glob.json the GVAR-EM loop reads. If only the macro data changed, the fiscal SFA still must be rebuilt so the residual stays exact against the new gdp/deflator levels.

    2.2 (Optional) refresh the human-checkable walkthrough¶

    A reviewer-facing Excel walkthrough of the US/UK fiscal accounting lives at docs/fiscal_accounting_walkthrough_US_UK.xlsx (the user's source is docs/Fiscal_Accounting_US_UK_NSA.xlsx). Regenerate it after a fiscal rebuild:

    python3 pipeline/build_fiscal_walkthrough_xlsx.py

    3. Port the exports into the app (33 blocs, 6 sig-figs)¶

    # Ports EVERY bloc present in the export dir, compacting to 6 significant figures + compact JSON.
python3 pipeline/port_gvar_exports.py

# Restrict to specific blocs, or dry-run to report sizes:
GVAR_PORT_BLOCS="US EA JP" python3 pipeline/port_gvar_exports.py
python3 pipeline/port_gvar_exports.py --dry-run

    This copies /tmp/gvar24_export/<CC>/*.json → app/public/data/bvar/<CC>/*.json at the deployed 6-significant-figure precision.

    3.1 Refresh the app fiscal_meta¶

    The app reads fiscal_meta per country at app/public/data/bvar/<CC>/fiscal_meta.json. Copy the freshly built pipeline metas over:

    for cc in US EA JP UK CA CN; do
  cp "pipeline/fiscal_meta/$cc.json" "app/public/data/bvar/$cc/fiscal_meta.json"
done

    RULE — re-index the fiscal meta after porting (MANDATORY on any variable-order change). Run node pipeline/fix_fiscal_meta_indices.mjs. fiscal_meta.json indexes the fiscal drivers by column POSITION; any change to a bloc's variable order/count (the GVAR-EM refit inserts variables ahead of the fiscal block in every spec except US) silently re-points those indices at the WRONG columns — the cause of the recent EA/UK debt explosion (EA's effRate landed on the equity index ~col 537). The porter does not regenerate the meta. fix_fiscal_meta_indices.mjs re-resolves the indices by variable id from the deployed spec.json and rewrites all six fiscal metas, preserving d0/sfaDefault; it is idempotent (prints ok when nothing moved). Run it after every port that touches a fiscal bloc — a stale index can pass tsc/build and still ship a wrong debt path.

    node pipeline/fix_fiscal_meta_indices.mjs   # re-resolves idx{gdp,deflator,revenue,primaryExp,effRate,sfa} by id

    3.2 Regenerate the GVAR coupling artifact (MANDATORY after any bloc data change)¶

    The browser solves the cross-country fixed point x = (I−M)⁻¹ d from a pre-shipped coupling artifact. At 33 blocs the in-browser inverse is ~19 min — pre-shipping is mandatory, not an optimisation. With the endogenous US the artifact is the augmented (stars ++ US-global) operator: dim = 2060, ρ(M) ≈ 0.924 (< 1, stationary/solvable; verified rhoSpectral = 0.924041 in the shipped coupling_default.json.gz), keyed to the data by modelSig.

    From app/ (the script builds M in JS, then inverts (I−M) via MATLAB BLAS — the invert_coupling.py NumPy path is the no-MATLAB fallback):

    cd app
npx vite-node scripts/precompute_gvar_coupling.mjs

    It writes the gzipped app/public/data/bvar/gvar/coupling_default.json.gz directly (the plain .json is removed — at this bloc count it exceeds GitHub's 100 MB per-file limit; gzip ≈ 27–32 MB). The loader decompresses it via the browser's native DecompressionStream. The artifact is keyed to the data by modelSig, so it MUST be regenerated whenever any deployed bloc changes — a stale artifact (modelSig mismatch) is rejected and the app falls back to the slow path.

    4. Verify (the green gate — do not skip)¶

    4.1 Fast GVAR coupling / fixed-point gate¶

    cd app
NODE_OPTIONS=--max-old-space-size=8192 npx tsx scripts/verify_gvar24.mjs

    Uses the shipped artifact (no live rebuild). Checks: (1) artifact modelSig matches the deployed blocs (else STALE — re-run §3.2); (2) the shipped inverse really is the inverse (max|(I−M)·invImM − I| small); (3) ρ(M) < 1 and matches the stored value; (4) Gauss-Seidel ≡ artifact-seeded closed form for representative shocks.

    4.2 Type-check, tests, build¶

    cd app
npx tsc --noEmit
npx vitest run          # ALL must pass — watch the canaries
BASE_PATH=/bvar-scenario npm run build

    Canary / regression tests to watch (in app/src/__tests__/compute/):

    • Fiscal: fiscalAccounting.test.ts, fiscal-backcast.test.ts, fiscal-conditioning-integration.test.ts — the debt back-cast and the SFA-forward projection.
    • GVAR: gvar-us-endogenous.test.ts, gvar-coupling-artifact.test.ts (artifact ≡ live), gvar-closed-form-routing.test.ts, gvar-derived-topology.test.ts, gvar-direct-coupling-strip-equivalence.test.ts, gvar-convergence.test.ts, gvar-band-solve.test.ts.
    • Scale canary: scenario-scale-convention.test.ts — if n changed, its fixture may be pinned to the old layout; regenerate via pipeline/_verify/gt.m (resolves columns by variable id, so it is layout-robust) and re-run.

    4.3 Back-cast / ground-truth canaries (MATLAB)¶

    pipeline/_verify/gt.m recomputes the canonical conditional-forecast ground truth directly from the deployed app/public/data/bvar/US/*.json via conforekf.m; compare against the app's conditional output to confirm the scale conventions still hold after re-estimation.

    matlab -batch "run('pipeline/_verify/gt.m')"

    Done = tsc clean, all vitest green, build succeeds, verify_gvar24 all PASS.

    5. Deploy¶

    The deploy target is the bvarscenario remote (matyasfarkas/BVARScenarioTool). A push to its main triggers a GitHub Action (.github/workflows/deploy-to-website.yml, on push to main touching app/**) that runs npm ci && npm run build (with BASE_PATH=/bvar-scenario), then copies app/out/* into the matyasfarkas/starter-hugo-academic website repo's static/bvar-scenario/. That repo is a Hugo Academic site published to GitHub Pages (note the netlify.toml trigger), which rebuilds and serves it at the /bvar-scenario path. (This workflow has no Hugo step itself; it only copies app/out.)

    Do not push without explicit confirmation. Pushing to bvarscenario main is a deploy. Confirm the gate is green first.

    cd "<repo root>"
git add app/public/data/bvar pipeline/fiscal_meta pipeline/country_rawdata \
        app/public/data/bvar/gvar/coupling_default.json.gz
git commit -m "Re-estimate GVAR + fiscal block (data through <quarter>)"
git push bvarscenario main          # Action copies app/out → starter-hugo-academic → Hugo rebuilds & serves on GitHub Pages

    Hugo is pinned to 0.122.0 in the starter-hugo-academic repo (0.123+ removed getCSV and broke the theme — do not bump it); that pin lives in the website repo, not in this workflow.

    Verify live. Open the /bvar-scenario path of the academic site (GitHub Pages) and confirm a US baseline, a quick conditioning scenario, and the fiscal debt/GDP projection all look sane.

    6. End-to-end (copy-paste skeleton)¶

    # from <repo root>
# 1. GVAR-EM re-estimation (hours) — validate on a subset first
US_ENDOGENOUS=1 matlab -batch "run('pipeline/export_bvar_gvar24_loop.m')"

# 2. Fiscal SFA = exact residual (auto closure gate < 1e-6 each)
cd pipeline
for cc in us ea jp uk ca cn; do python3 "build_fiscal_${cc}_4var.py"; done
cd ..

# 3. Port (6 sig-figs) + refresh fiscal_meta + RE-INDEX fiscal meta + regen coupling
python3 pipeline/port_gvar_exports.py
for cc in US EA JP UK CA CN; do cp "pipeline/fiscal_meta/$cc.json" "app/public/data/bvar/$cc/fiscal_meta.json"; done
node pipeline/fix_fiscal_meta_indices.mjs   # MANDATORY: re-resolve fiscal idx by id (stale-index guard)
cd app && npx vite-node scripts/precompute_gvar_coupling.mjs

# 4. Gate
NODE_OPTIONS=--max-old-space-size=8192 npx tsx scripts/verify_gvar24.mjs
npx tsc --noEmit && npx vitest run && BASE_PATH=/bvar-scenario npm run build
cd ..

# 5. Deploy (ONLY after the gate is green and confirmed)
git add app/public/data/bvar pipeline/fiscal_meta pipeline/country_rawdata
git commit -m "Re-estimate GVAR + fiscal block (data through <quarter>)"
git push bvarscenario main
