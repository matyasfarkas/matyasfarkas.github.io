    Maintainer's Guide — Manually Updating a Country Model¶

    This is the core human runbook for taking one country model end-to-end: change its data, re-estimate it, check the MATLAB output, store the JSONs, regenerate the coupling artifact, and run the green gate. It is written to be copy-paste followable with exact paths and commands.

    Read this with, not instead of: ADDING_A_COUNTRY.md (full data/spec/registry mechanics, the worked US example, the fiscal-block builder), REESTIMATING_MODELS.md (the re-estimation/port/coupling/gate sequence with the env-knob tables), and ARCHITECTURE_FOR_AI.md (the compute engine, data-space conventions, topology). This page does not re-derive those — it links to them and adds the per-country maintenance angle: the decision tree, the file-by-file edit map, the verification checks, and a troubleshooting table.

    The platform today: ~50 economies = 33 GVAR-coupled blocs (a fully endogenous US dominant unit + an EA aggregate + 31 Rest-of-World blocs) + 17 euro-area FSAP standalones. Data is harmonized to a common 2025Q4 vintage from FRED / Eurostat / BIS / ECB. Live at the /bvar-scenario path of the academic website, hosted on GitHub Pages (the matyasfarkas/starter-hugo-academic Hugo Academic site).

    Repo root in every command below is the absolute path "/home/farkas/github/BVAR Policy Calibration Tool" (quote it — it has spaces).

    0. Decision tree — what kind of update is this?¶

    Pick the smallest job that covers your change. The rough times assume the production hardware (MATLAB with the Parallel Computing Toolbox).

    You changed… Job What you run Re-estimate? Coupling regen? Rough time
    A fiscal driver only (revenue/expenditure/debt grid) for a fiscal bloc Fiscal-only refresh build_fiscal_<cc>_4var.py → re-estimate that bloc → port → coupling yes (the bloc) yes minutes (build) + the bloc's estimation
    Macro data of one euro-area FSAP standalone (the 17 euro members: AT BE DE EE ES FI FR GR IE IT LT LU LV NL PT SI SK) Single-country re-estimate the country builder → export_bvar_euro_fsap.m (FSAP two-country, writes /tmp/euro_fsap_export/) → GVAR_EXPORT_DIR=/tmp/euro_fsap_export port → gate yes (that bloc only) no (not in the coupled GVAR — no weights row) ~minutes
    Macro data of one coupled bloc (US / EA / any RoW) Full GVAR re-estimate build_* → export_bvar_gvar24_loop.m (US_ENDOGENOUS=1) → port → coupling → gate yes (all 33 — stars change for everyone) yes hours
    A variable's PRIOR (RW↔WN stationary) on one bloc Prior-change re-estimate tag the prior in the raw _glob.json (tag_fiscal_wn.py for the 6 fiscal blocs, else edit by hand) → re-estimate that bloc → port → (coupling if coupled) → gate (§B5) yes (the bloc) only if coupled the bloc's estimation
    A GLB_* / variable isFinancial flag (COVID-mask membership) Coherence re-estimate fix the flag in the builder + raw _glob.json → re-estimate the affected bloc(s) → port → coupling → gate (§B6) yes yes (coupled) the bloc(s)
    Add a brand-new country Add-country follow ADDING_A_COUNTRY.md §B; if GVAR, add a weights row → full re-estimate yes yes (if GVAR) hours
    App/compute code only (no data) Code change edit → tsc + vitest + build no no ~minutes

    Why a coupled-bloc refresh re-estimates everyone. Each bloc's RoW "stars" are trade-weighted averages of partners' DK-completed series. Touch one bloc's data and every partner's star moves, so export_bvar_gvar24_loop.m re-estimates the whole 33-bloc set to the cross-country fixed point. There is no "just re-estimate the US" path for a coupled bloc. (Verified in pipeline/export_bvar_gvar24_loop.m: with US_ENDOGENOUS, rowB = blocs — the full set iterates.)

    When is a single-bloc re-estimate legitimate? Three cases, and only three: 1. A euro-area FSAP standalone (AT BE DE EE ES FI FR GR IE IT LT LU LV NL PT SI SK) — it has no weights row, no partner reads its series, so re-estimating it alone is exact. Use export_bvar_euro_fsap.m; no coupling regen. 2. A change that does not move the stars — e.g. flipping a domestic variable's prior (RW↔WN) or a non-star isFinancial flag on a coupled bloc. The bloc's own betahat changes, but the stars it exports to partners are unaffected only if the changed column is neither a target (gdp/price/rate, which feed ROWDEM/ROWINF/ROWFIN) nor the COVID-mask membership of a target. In practice the safe, audited path is still the full loop — see the caveat in §B5. A genuine one-bloc shortcut is the BVAR_GVAR_BLOCS="US <CC>" + MAXIT=1 separated run (§B7), which freezes every other bloc's stars at their last value; it is exact only when no partner's star depends on the changed bloc (or you accept frozen partner stars for a quick check, then do the full run before deploy). 3. App/compute code — no estimation at all.

    The honest rule of thumb: if the bloc is coupled and the change can move any of its three target columns, do the full loop. The single-bloc path is for euro standalones, code, and quick local checks — not for a production deploy of a coupled bloc.

    A. Change the DATASET¶

    A1. Pick the right builder¶

    Country class Builder (pipeline/) Output
    FRED majors (EA/UK/JP/CA …) build_country_data.py raw matrix from FRED
    US (macro + fiscal drivers) build_us_auto.py US glob + fiscal drivers
    OECD EU/EFTA (Eurostat) build_oecd_eurostat.py / build_oecd_batch.py OECD economies
    GVAR globals/stars assembly build_gvar_2025q4.py, build_globals_new_blocs.py DataRaw_<CC>_glob.json (domestic + 4 GLB_*)
    Euro policy-rate + house prices build_euro_polrate_hpi.py rate + HPI columns
    Euro shared financial block build_euro_financial_block.py EA corp/rates
    Fiscal block (4-var) build_fiscal_<cc>_4var.py (cc ∈ us/ea/jp/uk/ca/cn) edits glob + fiscal_meta

    The FRED key is now a hardcoded fallback in the scripts, but always pass your own to be safe: FRED_API_KEY=<your_key> python3 pipeline/<builder>.py. Never commit a key (app/.env.local is git-ignored).

    A2. Where the raw data lives, and the precedence¶

    The estimator reads one raw file per bloc, in this precedence (first that exists wins — verified in pipeline/export_bvar_gvar24_loop.m):

    pipeline/country_rawdata/DataRaw_<CC>_glob.json   # 1st — globals-augmented (domestic + 4 GLB_* commons)
pipeline/country_rawdata/DataRaw_<CC>_2025q4.json # 2nd — the dated vintage
pipeline/country_rawdata/DataRaw_<CC>.json        # 3rd — legacy

    Precedence trap. If a *_glob.json exists, an edit to *_2025q4.json or the bare file is invisible to the estimator. For a GVAR bloc, always edit/rebuild the _glob.json — it is the file with the 4 GLB_* commons the loop needs.

    A3. Add / swap a series¶

    A series is a tuple in the builder's COUNTRIES block (mirror an existing entry, e.g. EA in build_country_data.py):

    ("FRED_ID", "GDP_XX", "XX Real GDP", "log", False),
# (fred_id, internal_id, display_name, transformation, is_financial)
    • transformation: 'log' = stored as 100·log(level), displayed as 4·Δ (q/q annualized growth) — real GDP, prices, indices, FX, equity. 'lin' = pass-through — rates, ratios, unemployment.
    • is_financial: groups under the "financial" header and is load-bearing for COVID masking — financial columns stay observed through COVID (only macro/isFinancial==false columns are NaN'd; see §B3).
    • After editing, run the builder (A1) to repull/rebuild the raw matrix.

    A4. Reflect it in the spec block tags¶

    The spec.json block tag is what the engine routes on (verified in app/src/lib/compute/gvarSolve.ts deriveBlocIdx/configureGVARTopology and exportbloc in the loop):

    block Meaning Where it comes from
    domestic the bloc's own variables from the _glob build (GVAR) / the bare DataRaw (euro FSAP)
    global the 4 shared commons GLB_OIL/GLB_USLR/GLB_USBAA/GLB_USEQ (identical across blocs, US-determined) the _glob build (GVAR) / the US-block commons WTI_US/FCM10_US/FBAA_US/EQ_US (euro FSAP)
    row the 3 trade-weighted stars ROW{DEM,INF,FIN}_<CC> (GVAR blocs only) appended at export time by exportbloc — do not hand-add
    fiscal the 4 fiscal drivers FREV_GDP/FPEXP_GDP/EFFI/SFA (6 fiscal blocs) the fiscal builder
    us / external the exogenous US block of a euro FSAP standalone (jointly estimated, hidden from charts, non-conditionable) export_bvar_euro_fsap.m

    There is no plain non-GVAR OECD/EM standalone in the deployed set — every non-euro bloc is a GVAR bloc. The 17 euro members are FSAP two-country standalones whose foreign block is us/external (plus the 4 global commons), not all-domestic. block:'row' is what marks a bloc GVAR-active (plus a weights row), so a new GVAR bloc needs a row in app/public/data/bvar/gvar/weights_24bloc.json (the filename is historical — it now holds 33 blocs, US first; verified: blocs: 33, has US weights row: True).

    The variable list for spec.json is reflected in pipeline/country_specs.ts; the dropdown registry is app/src/lib/data/countries.ts. (Full mechanics: ADDING_A_COUNTRY.md §B2–B3.)

    B. ESTIMATION SETTINGS¶

    The estimator is pipeline/export_bvar_gvar24_loop.m (GVAR-EM loop). Knobs are env vars (no code edit). All values below are verified against the script.

    B1. Env knobs¶

    The script hardcodes its own repo root and the absolute country_rawdata/export paths, so it can be launched from anywhere — only the run(...) argument is a path. Use the absolute script path:

    # Production full run (33 blocs, US endogenous). HOURS.
US_ENDOGENOUS=1 matlab -batch \
  "run('/home/farkas/github/BVAR Policy Calibration Tool/pipeline/export_bvar_gvar24_loop.m')"

# Fast smoke (validate BEFORE the multi-hour run): 3 blocs, few draws, 1 sweep (separated).
BVAR_GVAR_BLOCS="US EA JP" NDRAWS=200 MAXIT=1 US_ENDOGENOUS=1 matlab -batch \
  "run('/home/farkas/github/BVAR Policy Calibration Tool/pipeline/export_bvar_gvar24_loop.m')"
    Env Default (in-script) Meaning
    US_ENDOGENOUS unset (anchor) 1 ⇒ US is a coupled bloc with its own ROW*_US stars (needs a US weights row). Set for production.
    BVAR_GVAR_BLOCS all blocs in weights file space-separated subset, US first
    NDRAWS nDraws=1000 (nBurn=500) total MCMC draws; burn auto-set to NDRAWS/2. Exported n_draws = post-burn = NDRAWS − nBurn (500 by default). The inner GLP sampler's 'Ndraws',1000 keyword is a separate, fixed knob in the cgl_bvarV5 call — NDRAWS does not change it.
    MAXIT 12 convergence sweeps; MAXIT=1 ⇒ single-country / separated (no coupling)
    TOL 0.03 convergence tol on max|Δstar| (MC noise floor ≈ 0.13)
    GVAR_WEIGHTS_FILE weights_full.json else weights_24bloc.json which weights / bloc list

    Fixed in-script (don't "fix" without re-validating): lags=3, H=20, MCMCconst=1.

    B2. Priors (lambda / mu) and the BVAR call¶

    Each bloc is estimated by cgl_bvarV5([domestic | 3 stars]) — the COVID-as-missing Gibbs sampler. The call (verified, both the US bootstrap and the parfor body):

    cgl_bvarV5(x, lags, [], nDraws, nBurn, MCMCconst, H, ...
  'mcmc',1,'MNpsi',0,'sur',0,'noc',1,'Ndraws',1000,'Ndrawsdiscard',500);
    • lambda/mu are empirical-Bayes (MH on the completed-data marginal likelihood) — not pinned. With COVID imputed the optimum is well-behaved (estimated ~0.24–0.56).
    • sur=0, noc=1, MNpsi=0 — mirror these exactly when porting any other estimator.
    • Do not pin lamfix. If you must, keep noc=1 or sur=1 free — pinning lambda with both noc and sur fixed degenerates lambda to NaN.

    Model-selection guidance for lambda (the empirical-Bayes prior tightness) by bloc size:

    Bloc size n Typical lambda Note
    small (n≈14–20: most RoW — e.g. CR 14, RU 15, TR/BR 18, CN/MX/IN 20) looser, ~0.3–0.5 fewer params, less shrinkage needed
    medium (n≈23–25: JP 23, EA 24, UK/CA 25) ~0.25–0.4
    large (US n=41) tighter, ~0.24–0.3 many params (124×41 betahat) ⇒ more shrinkage

    These are observed empirical-Bayes optima, not knobs you set; they are a sanity range — a post-run lambda far outside it flags a data problem.

    B3. The COVID window + macro-only masking¶

    Verified in the loop: the COVID window is fixed 2020Q2–2020Q4 (3 quarters) — find(ismember(dts, ["2020-Q2","2020-Q3","2020-Q4"])). Over that window only macro columns (isFinancial==false) are NaN'd: domestic macro + ROWDEM + ROWINF. The financial domestic columns and the foreign-rate star ROWFIN stay observed, so the DK smoother imputes the macro holes conditioned on the observed financials. This is why is_financial (A3) is load-bearing.

    An older path auto-detected the GDP-trough window; it could misfire on other crises (KR's 2008Q4 GFC drop). The current script uses the fixed 3-quarter window. Leave it.

    B4. Exogeneity / the external block¶

    A bloc's foreign variables are weakly exogenous regressors: the 3 stars (block:'row') and the 4 globals (block:'global'). The stars are trade-weighted over partners present each quarter (unbalanced panel, renormalized), built by buildstars in the loop. Special cases baked in (mirror the engine — see ARCHITECTURE_FOR_AI.md §5):

    • CN GDP is real (since the 2026-06 surgical CN-only re-estimate; NOMINAL_GDP_BLOCS is now empty) → CN's DEM reads gdp directly, no longer the deflated X_g − X_p. (Historically GDP_CN was nominal and deflated by subtraction.)
    • Rate-less blocs (CH/NO) have no own short rate → excluded from partners' ROWFIN.
    • AR hyperinflation → excluded from partners' ROWINF/ROWFIN (kept in ROWDEM).
    • Leading-NaN star columns are back-filled with the first valid value (a NaN-prefix gives a singular SIGMA).

    B5. Change a variable's PRIOR (RW ↔ WN stationary) and re-estimate¶

    This is the per-variable Minnesota/GLP prior-mean knob: it sets the own-first-lag prior mean to 1 (random walk, the historical default) or 0 (white noise / stationary). It is not an env var — it travels on the raw _glob.json as a per-variable prior field, and the estimators read it.

    B5.1 What WN does, mechanically (verified)¶

    The GLP prior in BVAR/MFBVAR/GLPreplicationWeb/bvarGLPmf.m builds the prior coefficient block as a diagonal of ones, then zeros the rows named in pos:

    % bvarGLPmf.m (verified, lines ~193–195)
diagb=ones(n,1);
diagb(pos)=0;          % own-first-lag prior mean -> 0 for the WN (stationary) columns
b(2:n+1,:)=diag(diagb);

    pos is the 3rd positional argument of cgl_bvarV5.m (pos=stationary;, verified line 19) — the COVID-as-missing Gibbs sampler threads it straight into bvarGLPmf and into every logMLMFVAR/BVAR call. So:

    prior tag pos membership own-lag prior mean Behaviour at the long end
    RW (default) not in pos 1 random walk — the variable drifts, no mean reversion
    WN (stationary) in pos 0 white noise — the variable mean-reverts to its in-sample mean

    The estimators build pos from the spec's per-variable prior:

    • export_bvar_gvar24_loop.m: statCols = find(Spec.Prior(keep)=="WN")' (verified ~line 60) — indices into the kept domestic block; stars are appended after and are always RW.
    • export_bvar_euro_fsap.m: statCols = find(Prior=="WN")' (verified ~line 110).
    • Both pass statCols as the 3rd positional arg of cgl_bvarV5 and write the real prior back into spec.json (v.prior='WN'|'RW'), so the deployed spec is self-describing.

    No-op for non-WN blocs (byte-identical). When no variable is tagged WN, statCols is empty and pos=zeros(1,0), which indexes identically to the historical pos=[] in diagb(pos)=0. The 44 non-fiscal blocs are therefore bit-for-bit unchanged by the prior machinery (verified: commit 6f0161a notes "byte-identical for non-fiscal blocs"; confirmed in code by the empty-index path).

    B5.2 Current production state (verified in the deployed specs)¶

    The fiscal GDP-ratios mean-revert; the effective rate drifts. All 6 fiscal blocs (US EA JP UK CA CN) ship with:

    Variable prior Why
    FREV_GDP (revenue/GDP) WN a ratio — must mean-revert to its in-sample level, not drift
    FPEXP_GDP (primary exp./GDP) WN a ratio — same
    SFA (stock-flow adjustment) WN a residual flow — mean-reverts to sfaDefault
    EFFI (effective interest rate) RW a rate level — stays RW (drifts with the rate environment)

    Verified across all six (app/public/data/bvar/<CC>/spec.json): every fiscal bloc shows FREV_GDP/FPEXP_GDP/SFA → WN, EFFI → RW. Effect: US revenue now mean-reverts toward its in-sample mean (≈17.4% over the full sample; the deployed 40-quarter baseline ends ≈17.9% vs the last observed ≈18.8% and the old RW path stuck near ≈19%), instead of drifting at the long horizon. Committed 6f0161a / 22250be / e38673a.

    B5.3 How to change a prior — copy-paste¶

    Step 1 — tag the raw _glob.json. For the 6 fiscal blocs, use the idempotent helper (pipeline/tag_fiscal_wn.py); it writes prior on every variable object (WN on FREV_GDP/FPEXP_GDP/SFA, RW on the rest incl. EFFI):

    cd "/home/farkas/github/BVAR Policy Calibration Tool"
python3 pipeline/tag_fiscal_wn.py     # idempotent; format-preserving; re-runnable after any rebuild

    For any other variable, edit the variable object in pipeline/country_rawdata/DataRaw_<CC>_glob.json to add "prior":"WN" (or "RW"). Critical: tag EVERY variable object in that file, not just the one you change — MATLAB's jsondecode returns J.variables as a struct array only when all objects share the same field set; a prior field on some objects makes it a cell array and breaks the estimators' arrayfun(@(v)v.id, …) calls. (The estimators' priorsfrom helper tolerates the cell-array shape, but the safe, audited convention is homogeneous tagging — exactly what tag_fiscal_wn.py does.)

    Step 2 — re-estimate the bloc.

    • Coupled bloc (US/EA/RoW): the spec change is local, but the audited deploy path is the full loop (§B1) so the stars stay at the cross-country fixed point: 
      US_ENDOGENOUS=1 matlab -batch \
  "run('/home/farkas/github/BVAR Policy Calibration Tool/pipeline/export_bvar_gvar24_loop.m')"
      For a quick local check of just the changed bloc (frozen partner stars — not for deploy): 
      BVAR_GVAR_BLOCS="US <CC>" MAXIT=1 US_ENDOGENOUS=1 NDRAWS=400 matlab -batch \
  "run('/home/farkas/github/BVAR Policy Calibration Tool/pipeline/export_bvar_gvar24_loop.m')"
    • Euro FSAP standalone: re-estimate only that bloc (no coupling): 
      GVAR_EXPORT_DIR=/tmp/euro_fsap_export matlab -batch \
  "run('/home/farkas/github/BVAR Policy Calibration Tool/pipeline/export_bvar_euro_fsap.m')"

    Step 3 — confirm the prior landed in the export, then port + (coupling) + gate (C–E):

    python3 -c "import json,sys; cc=sys.argv[1]; \
 s=json.load(open(f'/tmp/gvar24_export/{cc}/spec.json')); \
 print(cc, [(v['id'],v['prior']) for v in s['variables'] if v.get('block')=='fiscal'])" US
# expect: US [('FREV_GDP','WN'),('FPEXP_GDP','WN'),('EFFI','RW'),('SFA','WN')]

    Step 4 — sanity-check the mean reversion. A WN ratio should bend back toward its in-sample mean over the horizon; an RW one drifts. After porting, the baseline's end-of-horizon value of a WN ratio should sit inside its history's range, not extrapolate past it. (For the fiscal closure invariant after a fiscal change, run C4 — the prior change does not touch the SFA-closure assertion, which is purely an accounting identity on the observed debt path.)

    Pitfall: WN on a true level. Only tag WN on series that genuinely mean-revert (ratios, spreads, gaps). Tagging a level (real GDP in 100·log, a price index, the effective rate) WN would pull its own-lag toward 0 and kill the unit root, forcing the level itself to revert — wrong. That is why EFFI stays RW. When in doubt, leave RW.

    B6. isFinancial / GLB_* and the COVID mask — coherence rules¶

    isFinancial is load-bearing, not cosmetic: it is the COVID-mask membership test (§B3).     B7.1 What BVAR_GVAR_BLOCS="US <CC>" actually does (verified)¶

    The loop loads only the listed blocs into D (for bi=1:nB over blocs, verified ~lines 46–81). buildstars then trade-weights only the partners present in D (parts = parts(ismember(parts, blocs)), verified ~line 178) and renormalizes over them. So:

    • A subset run does not freeze the absent partners at their full-run values — it drops them entirely and recomputes the listed blocs' stars from only the listed partners (US + <CC>).
    • The listed blocs' seed comp panels are built fresh from their own raw _glob.json (line 78), not loaded from /tmp/gvar24_workspace.mat. There is no "load the last full run and re-estimate one bloc" path in the script.

    G. The long-run-target overlay (δc drift sliders) — what the operator needs to know¶

    The control panel's "Long-run targets" section is a runtime, in-browser overlay. As a maintainer the one thing to internalise is: it is NOT part of the estimation / port / coupling / gate pipeline. There is nothing to rebuild, re-estimate, re-index, or commit when this feature changes the display — it computes everything from the already-shipped companion_mean.json of the bloc the user is viewing. A data refresh (§A–§F) ships it for free; a data refresh does not need any extra step for it.

    G1. What it is (operator summary)¶

    Each slider sets one variable's long-run (≈H-quarter) target in the chart's display units (4·Δlevel = q/q-annualised growth for log vars; the level for lin vars). Setting a target shifts that variable's drift — the companion constant of its own equation — by the single amount δc_j needed for its terminal forecast to hit the target, and the estimated model dynamics carry every other variable along. It is:

    • Deterministic and exact — δc_j = (target − currentTerminalForecast) / sens_j; the terminal forecast lands on the target by construction (the panel badge reads applied · exact). An unmoved slider ⇒ δc=0 ⇒ a byte-exact no-op (it defaults to the model's own terminal forecast).
    • Instant — no Compute, no re-estimation, no GVAR re-solve, no draw reweighting. The deviation is ΔY = S_h[:, j]·δc_j, where S_h (the partial companion-power sum Σ_{k<h} T^k) is built once per bloc in the browser from the shipped companion and cached; each slider change is then a cheap mat-vec add (computed in app/src/lib/compute/driftSetter.ts).
    • Band-coherent — ΔY is added identically to the central path and to every quantile (q05/q16/q50/q84/q95), so the fan translates rigidly: band widths are invariant and quantiles cannot cross. Safe to apply on top of the precomputed bands.

    This replaced the old entropic-tilt long-run-anchor mechanism (longRunTilt.ts), which reweighted posterior draws, needed a Compute, and degenerated (ESS→1) on a joint multi-target tilt. The drift overlay has none of those failure modes (see the superseded known-issue note below).

    G2. Interpreting / disabling a variable (the |sens|<0.3 guard)¶

    The slider's sensitivity sens_j is how far the variable's terminal display value moves per unit of drift (4·(S_H[j,j] − S_{H-1}[j,j]) for log; S_H[j,j] for lin). When |sens_j| < 0.3 (SENS_GUARD in driftSetter.ts) the response is near-singular — a tiny δc would imply an enormous, ill-posed move — so the variable is disabled: it shows in the dropdown as "(near-singular — unavailable)" and cannot be targeted. This is expected, not a fault. It typically hits a near-unit-root level (where the finite-horizon drift barely moves the terminal display). Operator guidance: target the growth/ratio form rather than the level, or pick another variable. Do not try to "fix" it by lowering the guard — the guard is what keeps the overlay well-posed.

    G3. Caveats to relay to users¶

    • It shifts the long-run drift, not persistence — label/treat it as a long-run target, not a path controller.
    • The cross-variable response (other variables moving) is a real estimated-model side effect, not a bug.
    • The real-growth target owns the debt direction (the (i−g) snowball) — it is not a structural debt fix; it changes the growth assumption the debt path inherits.
    • GVAR bloc → marginal. It shifts that bloc's own long-run forecast only; trade partners and the US-determined global commons are not re-solved, so the realised joint long-run value may differ from the target. The panel surfaces this amber caveat automatically for a GVAR bloc.

    G4. The policy rate (Taylor overlay removed)¶

    The policy rate is an ordinary lin variable of the VAR, so it carries its own long-run-target slider like every other series — set its terminal value directly through the drift setter. There is no separate Taylor-rule overlay: an earlier taylorOverlay toggle (policy-rate row, i = r* + π_fc + φ_π(π_fc − π*)) was removed, and no taylorOverlay / setTaylorOverlay symbol remains in ControlPanel.tsx or useStore.ts. (The surviving "Taylor" code in taylor-rule.ts is the unrelated DSGE policy-rule counterfactual, not a forecast overlay.)

    G6. Model-based vs. with-judgement, and the consensus-anchor preset¶

    Two operator-relevant framings sit on top of the drift setter:

    • Model-based vs. with-judgement (per variable). A variable with its slider at the model estimate is model-based (δc=0, exact no-op). Setting a target makes it with-judgement — the panel highlights the target, shows a per-row reset-to-estimate control, and the footer badge reads N targets · exact. The flag is per-country and per-variable and is cleared on a country switch (targets reference the old bloc's variable ids). This is a UI/state distinction only — no new data artifact and nothing to re-estimate.
    • Consensus-anchor preset (app/public/data/default_anchors.json). An opt-in, off-by-default one-click set of publicly-defensible long-run values per bloc ({ COUNTRY: { variableId: value } }), surfaced as Apply consensus anchors. It is keyed by country code then spec variable id (e.g. GDP_BE, HICP_BE, LR_BE), in display units. The panel auto-drops any id that is not in the bloc's targetable set or whose slider is near-singular (the |sens| < 0.3 guard) — so the perverse policy-rate sliders (BE/CA/CZ) are intentionally omitted from the JSON / never applied. To curate the values, edit default_anchors.json (the canonical source is pre-ship-smoke-test.md §2); it ships under app/public/data/ and is fetched at panel mount. No re-estimation, no green-gate impact — but keep the variable ids in sync with each bloc's spec when a model is re-ported (a stale id is silently skipped, not an error).

    Known issues / limitations (flagged, NOT yet fixed)¶

    These are open, honestly-documented defects in the live tool — flagged for fix, not yet fixed. State them as such; do not present the affected outputs as fully validated.

    1. CN fiscal debt — inflation double-count (RESOLVED 2026-06). For China, fiscal_meta.json idx.gdp used to point at a nominal GDP_CN, so the snowball gNomQ = Δlog(idx.gdp) + Δlog(idx.deflator) double-counted inflation. The surgical CN-only re-estimate replaced it with real GDP (CN/spec.json var 0 = "CN Real GDP (quarterly, chain-linked SA volume index, base 2011Q1=100)" and CN/fiscal_meta.json idx.gdpIsNominal = false), so the roll-forward now reads the correct real-GDP + deflator nominal growth, matching the other five fiscal blocs. The fix re-estimated CN only; the other 32 GVAR blocs are byte-identical to the prior working model.
    2. Long-run tilt — no non-finite filter on the weights (SUPERSEDED by the δc drift overlay; tilt no longer the live long-run mechanism). longRunTilt.ts solveEntropicTilt computes the per-draw weights as w_i = exp(γ'·(g_i − target)) over the long-run moments g_i. Unstable draws (unit-root / near-singular (I − ΣB_l)) return a NaN long-run moment (verified: the steady-state path returns NaN for the singular case), and the weight solve does not drop them — one NaN g_i poisons wsum and silently empties the anchor band. (The std-scaling loop and weightedQuantile do filter non-finite, but the dual/Newton weight solve does not.) The live "Long-run targets" panel no longer uses this tilt — it uses the deterministic δc drift overlay (§G), which has no draw reweighting and therefore none of this failure mode. longRunTilt.ts is now dormant/legacy; this bug only bites if it is re-wired in. Fix-if-revived: filter non-finite g_i before solveEntropicTilt.
    3. Closed-form GVAR solve — no ρ(M)<1 divergence guard. solveGVARClosedForm (gvarSolve.ts) solves x = (I−M)⁻¹d either via the cached inverse or a Neumann fixed-point (x ← d + Mx, capped at maxIt=2000). There is no check that ρ(M)<1: a diverged fixed point (or a Neumann iteration that hit the cap without converging) returns a confident-wrong forecast with no warning. The shipped artifact is stationary (ρ=0.924), but a future re-estimation that pushed ρ≥1 (e.g. disaggregating the euro members) would not be caught here. Fix: assert ρ<1 / Neumann convergence and surface a divergence warning.
    4. Conditional-forecast band uses the FILTERED (not DK-smoothed) covariance. The conditional band variance is taken from filteredP (kalman.ts: "Extract forecast variances from filtered covariances", variances.push(submatrix(filteredP[t], …))), not the smoother covariance. A future pin tightens the smoothed uncertainty of the quarters preceding it, but the filtered covariance does not see that future observation — so the band is OVERSTATED for quarters before a future pin.
    5. The porter does not regenerate fiscal_meta. Positional fiscal indices go stale on any variable-order change — see the RULE in D2b (fix_fiscal_meta_indices.mjs). This was the cause of the recent EA/UK debt explosion. The cause is documented as a limitation; the fix is the mandatory re-index step.

    Empirical evaluation — where it lives, and what it does NOT touch¶

    The model's out-of-sample evaluation is a separate repository, /home/farkas/github/em-bayesian-gvar-eval (full design + runbook in its docs/EMPIRICAL_PROTOCOL.md — read that before touching the eval). It is read-only with respect to this repo: it consumes this repo's estimator (cgl_bvarV5.m, export_bvar_gvar24_loop.m) and the raw DataRaw_<CC>_glob.json panels, but changes nothing here and is not part of the green gate or the deploy. Nothing in §A–§F depends on it.

    What it implements (design final; results now produced — read them off results/tables.{txt,json} + results/tierA/us_rate_100bp.json, do not re-key the numbers from memory):

    • An OOS forecast horse-race, 9 models. Proposed = the EM-Bayesian GVAR (endogenous-US emgvar_endo
    • frozen-anchor emgvar_frozen). Benchmarks = Random Walk (rw) + climatology (rw_clim), AR(p) (arp), single-country BVAR (sbvar, strip the 3 ROW* columns), the textbook OLS Pesaran–Schuermann–Weiner / Dees GVAR (cgvar_a), a single-pass Bayesian GVAR (cgvar_b, MAXIT=1), and a Bayesian DFM (dfm). Recursive expanding window, 56 origins 2010Q1–2024Q4 (2020 COVID origins dropped from the headline), rolling as-of-t0 trade weights (leak-free — the shipped 2021–2023 weights_full.json is forbidden in the OOS sweep), targets {real GDP growth, inflation, short rate} for US/EA/JP/UK/CA/CN, horizons h∈{1,2,4,8}.
    • Scoring. CRPS is the primary density score + a 90% Model Confidence Set (CRPS-broken parametric log-score is a secondary column — see the caveat). PIT/coverage; significance via Giacomini–White (headline, valid under nesting), Clark–West (nested point), Diebold–Mariano (non-nested), with Romano–Wolf + the MCS for multiplicity.
    • An endogenous-US spillover analysis off the US_ENDOGENOUS=1 model (Tier-A).

    Headline (read the exact numbers off results/tables.json). AR(p)/DFM have the lowest point CRPS overall, but the 90% MCS shows emgvar inside the INFLATION MCS at every horizon (h=1,2,4,8) and inside the SHORT-RATE MCS at h=1,2,4 — statistically indistinguishable from the best on the nominal block; it is out of the GDP-GROWTH MCS at h=1 and h=8. emgvar_endo ⪰ emgvar_frozen in the MCS (the endogenous-US design earns its keep — e.g. at RATE h=1 endo is in the set and frozen is not). It decisively beats the textbook OLS classical GVAR (cgvar_a is removed from most MCS sets) — the EM-Bayesian treatment is the demonstrable value-add. Best INFL h=1 is emgvar_endo (relRMSE 0.799).

    Honest caveats (must state). (1) The draw-based densities are OVER-CONFIDENT — PIT cov90 ≈ 0.73–0.85 for emgvar vs the 0.90 target. (2) The parametric log-score column is numerically broken on the under-covered clouds (it returns ~1e12–1e16); CRPS + the MCS are the robust primaries — do not cite the LPS column. (3) Tier-A spillover (eval/tier_a_scenario.mjs): a US +100bp multi-period conditional projection shows non-trivial endogenous-only cross-country feedback the frozen anchor structurally omits (its own note: gap = endogenous − frozen = the RoW→US feedback the frozen anchor cannot represent), but the magnitudes are large and sign-flipping (e.g. US growth peak +1.28 endo vs a −4.37 frozen-path spike) and it is a conditional projection, NOT a causal IRF/GIRF — the magnitudes need a validation pass before being headlined. Do not overclaim a causal interpretation.

    If you re-estimate or re-prior a bloc here, the eval must be re-run on its side to stay consistent — but that is its repo's concern, not a step in this runbook.
