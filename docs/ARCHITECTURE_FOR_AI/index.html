    Codebase Guide (for AI agents and new developers)¶

    This document is a dense, accurate map of the Monetary-Financial Model (a global projection and scenario tool, built on an EM-Bayesian Global VAR engine) for an automated agent or a developer who needs to make a correct change quickly. It explains where things are, why they are shaped the way they are, and the conventions you must not break. Read it before editing compute or pipeline code.

    Companion docs: README.md (overview), USER_GUIDE.md (UI), fiscal-block-methodology.md (fiscal math), ADDING_A_COUNTRY.md (human runbook for data/model updates).

    1. What the app is¶

    A static, client-side web app: a Next.js (App Router) frontend that does all econometrics in the browser in TypeScript. There is no required backend for the core features — the pre-estimated models are static JSON under app/public/data/ and every Kalman filter / smoother / matrix op runs on the user's machine. A FastAPI backend exists only for computing DSGE IRFs from custom policy rules and is not needed for normal use.

    The economic content is a three-layer methodology:

    1. Conditioning — impose user paths on any BVAR variable; solve the conditional forecast with a Durbin–Koopman (DK) simulation smoother on the BVAR companion form.
    2. Counterfactual — overlay a DSGE-model policy counterfactual (MMB IRFs + a policy rule, or the Barnichon–Mesters optimal-policy QP).
    3. Uncertainty — posterior-draw fan charts.

    On top of these sit three extensions: a 6-country fiscal block (US/EA/JP/UK/CA/CN — derived debt/GDP), a 33-bloc GVAR layer with a fully endogenous US (cross-country spillovers across an endogenous US bloc + an EA aggregate + 31 Rest-of-World blocs, ~90% of global demand), and a long-run anchor (entropic tilt of the predictive density toward a user-set long-run target — longRunTilt.ts).

    🔴 TOPOLOGY CHANGE — the US is now an ENDOGENOUS coupled bloc, NOT the frozen anchor. The deployed US/spec.json carries n=41: its domestic + fiscal vars plus 3 ROW*_US trade-weighted stars (cols 38–40, block:'row') plus the 4 global commons it determines. With a US row in weights_24bloc.json, configureGVARTopology sets US_ENDOGENOUS = true: the US now drives RoW AND receives RoW→US feedback through its own stars. The frozen-anchor code paths still exist (gated on !US_ENDOGENOUS) and stay byte-identical for legacy/US-less-stars data, but the live deployment runs the endogenous path. The closed form is solved on an augmented state (stars ++ US-global deltas, dim=2060, ρ(M)=0.924041 in the live artifact). See §5/§5a.

    The app ships ~50 economies in one registry (app/src/lib/data/countries.ts): 6 core majors, 34 OECD members, 10 non-OECD EM. 33 of them are GVAR-coupled — the endogenous US bloc + an EA aggregate bloc + 31 RoW blocs listed in weights_24bloc.json (the filename is stale; it now holds 33 blocs, US first, with a US weights row that activates the endogenous coupling). Everything else is a standalone single-country model. In particular, the 17 euro-area members (DE FR IT ES NL AT BE FI GR IE LU PT SI SK EE LT LV) are standalone (domestic + 4 global commons + trade-weighted foreign stars, estimated in separation), not in the coupled GVAR and with no weights row. The reason: the fully-disaggregated euro GVAR was non-convergent (ρ(M)=1.089>1); aggregating the euro area into one EA bloc restores stationarity (ρ(M)=0.726) — this is why EA is a single bloc and the members are standalone.

    3. The render/compute flow (one pass)¶

    ChartGrid is the hub. Per render it computes:

    • condForecast = useConditionalForecast(...) — single-country conditional forecast (mean + level deviation + scenario levels) via the DK smoother.
    • gvarFc = useGvarForecast(...) — for any active GVAR bloc (the endogenous US + EA + the 31 RoW blocs), the cross-country propagated forecast (solved in a Web Worker). null for a non-GVAR (standalone) country. Under the endogenous-US topology it returns a two-tier result: a fast mode:'direct' first-order preview on every pin edit, and the complete mode:'full' fixed point on the explicit "Solve full GVAR" button (solveFullToken) — see §5a/§8.
    • effective = (isGvar && gvarFc && !hasFiscalTarget) ? {...condForecast, ...gvarFc fields} : condForecast — the path actually drawn. This is the single source of truth for "the displayed scenario line".
    • fiscalBlock (the 6 fiscal countries US/EA/JP/UK/CA/CN) — derived PB / debt-service / debt-to-GDP from effective.scenarioLevels by the accounting identity.

    Refs gvarBandRef and fiscalMeanRef snapshot effective and fiscalBlock so the band-computation effect (which fires only on Compute, deps [uncertaintyComputing]) reads fresh values without re-subscribing.

    3a. Per-chart render memoization (so one edit repaints ≈1 of ~35 panels)¶

    A single conditioning edit changes scenarios, re-running the monolithic charts useMemo → every chart object gets a fresh identity. Absent memoization, Plotly would repaint all ~35 panels for one dragged point. Two levers (in ChartGrid.tsx) collapse that to ≈1 panel:

    • reconcileCharts(next, prev, onRebuilt?) (ChartGrid.tsx:205) reuses the PRIOR render's chart object reference for any panel whose freshly-built content is deep-equal (chartDeepEqual, :173), keyed by chartVar.bvarIdx. It returns the prior array outright when every element kept its reference (stable outer identity → no [charts] effect churn). The data is byte-identical — this only preserves reference identity where the content did not move.
    • MemoDraggableChart = memo(DraggableChart) (:2523) skips the re-render entirely when a panel's chart reference and the grid's useCallback'd handler props are all ===.

    So a conditioning edit (or a GVAR spillover) repaints only the conditioned panel + any partner panels that actually moved. Pinned by chartgrid-render-memo.test.ts via the exported chartRebuildCounter / draggableChartRenderCounter instrumentation (test-only, ~zero production cost). This is the render optimization; the compute optimization (keeping the heavy GVAR solve off the drag path) is §5a/§8b.

    5. The GVAR engine (app/src/lib/compute/ + lib/data/gvarLoader.ts)¶

    33-bloc GVAR (~90% of global demand). The US is a fully endogenous coupled bloc (n=41): it carries its domestic + fiscal vars, the 4 global commons as its OWN endogenous variables (the US determines oil / US10Y / USBAA / USeq), AND 3 ROW*_US trade-weighted stars (cols 38–40, block:'row') from a US trade-weights row — so the US drives RoW through the globals/its linked series AND receives RoW→US feedback through its own stars, iterating in the cross-country fixed point exactly like a RoW bloc. The other 32 blocs are an EA aggregate plus 31 Rest-of-World blocs (JP, UK, CA, CN, CH, KR, MX, AU, TR, SE, PL, NO, DK, IN, BR, RU, ID, SA, AR, ZA, TH, MY + the 9 newer non-euro economies CL, CO, CR, CZ, HU, IL, IS, NZ, TW). EA and all 31 RoW blocs each carry two kinds of foreign variable:

    • 3 trade-weighted stars (block:'row'): ROWDEM/ROWINF/ROWFIN = trade-weighted partner real-GDP growth / inflation / short rate, renormalized over partners present each quarter.
    • 4 global commons (block:'global', the SAME 4 ids in every bloc, US-determined): GLB_OIL (WTI), GLB_USLR (US 10Y), GLB_USBAA (US BAA → credit spread), GLB_USEQ (US equity). All 4 globals are isFinancial:true in every bloc (verified across US/EA/JP/CN/…), so the COVID-as-missing window keeps them OBSERVED (not macro-masked / imputed) — a coherence fix that applies in particular to GLB_OIL and GLB_USEQ, which were previously mis-tagged: oil and US equity are market data and must condition the COVID imputation, not be holes in it.

    How the globals propagate (applyGlobalPins in gvarSolve.ts). Each RoW bloc PINS its 4 global columns to its own baseline + the US scenario DEVIATION of the matching global (matched by id, 1:1): pin = cBaseLvl[col] + (usScenLvl[usCol] − usBaseLvl[usCol]). So a shock to US oil / US financial conditions moves the SAME common delta into every RoW bloc, and each bloc's domestic vars respond via its own estimated coefficients. deriveBlocIdx exposes the global columns as IDX[c].glob (id → column); a bloc with no globals gets glob = {} and applyGlobalPins is a no-op (the legacy globals-free path stays byte-identical).

    Frozen-anchor vs endogenous-US treatment of the US global delta. In the frozen-anchor path (!US_ENDOGENOUS) the US is solved FIRST and frozen, so the US scenario deviation is a constant: the global pin is a US→RoW exogenous forcing that enters the per-bloc forcing/baseline d, NOT the RoW↔RoW coupling M (the globals are held fixed across the per-period M probes). Under the endogenous-US path (the live deployment), the US scenario deviation of each global is itself part of the fixed point (the US responds to RoW feedback), so the US-global delta becomes an augmented state variable and its coupling enters M via the buildLinearSystemEndo channels (2)+(3) — see §5a. Indirect effects (globals move a bloc's macro → its stars → other blocs) always flow through the existing star coupling M and are not duplicated.

    🔴 DESIGN DECISION — pin the US DELTA, not the US LEVEL (load-bearing). Each RoW bloc estimated its OWN baseline path for the globals (weakly-exogenous regressors), and those baseline LEVELS differ from the US's by tens of 100·log units on oil. Pinning the US level would inject that entire baseline gap as a spurious delta even at ZERO scenario (every country would jump the moment it was selected). Pinning the US delta on each bloc's own baseline gives exactly zero spillover at zero scenario AND an identical common shock under conditioning — mirroring how the star pins use baseRoW + partnerDelta, not the partner level. The first (wrong) implementation pinned the level; it was caught by the CN-deflation and zero-scenario tests. Verified on the real GVAR data: a US GLB_OIL pin yields an identical oil delta across all RoW blocs to ≤1e-9, and a zero scenario yields zero RoW spillover.

    The scenario deviation is a linear fixed point x = M·x + d (baseline is data-driven, computed once). The engine solves it in closed form as x = (I − M)⁻¹·d. Under the live endogenous-US topology the state is augmented (RoW+US stars ++ the US-global deltas), so dim = 2060 and ρ(M) = 0.924041 < 1 in the deployed artifact (see §5a for the channel construction; the exact value moves slightly with each refit, what is load-bearing is < 1). M and its inverse come from the precomputed coupling artifact (bvar/gvar/coupling_default.json.gz, GZIPPED and decompressed in-browser via the native DecompressionStream; a plain .json fallback is tried for environments without it) — building/inverting M at this scale is many minutes in-browser, so the artifact is mandatory for interactivity, not optional. The shipped artifact carries modelSig (33-bloc, US:41), pinStructure: '' (the empty/baseline structure), dim, rhoSpectral, M, AND the BLAS-computed invImM. Gauss–Seidel (solveGVAR, which now iterates the US too when endogenous) is kept as a cross-check; closed-form ≡ iterative to ~1e-6 over the full conditioning battery. (Historical note: the pre-endogenous frozen-anchor coupling had dim≈1920, ρ(M)≈0.726 — those figures no longer describe the deployed artifact.)

    Topology is self-configured, not hardcoded. configureGVARTopology(specs) (in gvarSolve.ts) reads each deployed spec.json + weights_24bloc.json (the filename is stale — it now lists all 33 blocs) and, for every candidate bloc, derives its gdp/price/rate columns (deriveBlocIdx, mirroring the estimator's findcol prefix order), its RoW-star indices, and its global columns (IDX[c].glob). A RoW bloc is active iff its spec carries block:'row' vars AND it has a weights row. The US is endogenous iff its OWN spec carries the 3 ROW*_US stars AND weights_24bloc.json has a US row — this sets the module flag US_ENDOGENOUS. Two derived sets are mutated in place: - ROW_COUNTRIES = every bloc except the US (the weight-row + global-pin recipients; the US is the global SOURCE, never a global RECEIVER). - COUPLED_COUNTRIES = the blocs that iterate in the fixed point: ROW_COUNTRIES when the US is frozen, or ['US', ...ROW_COUNTRIES] (US first) when US_ENDOGENOUS.

    IDX, W, ROW_COUNTRIES, and COUPLED_COUNTRIES are mutated in place at load (importers hold live references); W is keyed for every COUPLED bloc (the US too, when endogenous), renormalised over the active panel. gvarLoader re-configures from whatever blocs are actually deployed. Never hardcode bloc indices or a bloc subset — read IDX/ROW_COUNTRIES/COUPLED_COUNTRIES at runtime.

    File Role
    gvarSolve.ts The engine. configureGVARTopology/deriveBlocIdx (self-config from spec+weights; sets US_ENDOGENOUS), IDX/W/ROW_COUNTRIES/COUPLED_COUNTRIES (live, mutated in place), solveGVARClosedForm (the production entry — routes to the endogenous augmented closed form when US_ENDOGENOUS, else the frozen-anchor (I−M)⁻¹d, both with the amortized coupling cache), solveGVARClosedFormEndo/buildLinearSystemEndo/blocKernels (the augmented-state endogenous path, §5a), solveGVARDirect (fast first-order preview from prewarmed kernels), prewarmGVARKernels/prewarmGVARKernelsChunk (off-critical-path kernel-cache warm), solveGVAR (Gauss–Seidel cross-check; iterates the US when endogenous), buildGVARCouplingMEndo/buildGVARCouplingEndo (export-time M / coupling builds), seedGVARCouplingCache/hydrateGVARCoupling (consume the precomputed artifact), solveViewedScenLvl (viewed-bloc-only; US short-circuits the sweep ONLY when frozen).

Two coherence fixes shipped with the WN-prior refit: the 4 globals are now isFinancial:true everywhere (above), and IP_CA (Canada industrial production) was rebuilt as a log LEVEL — chained from the discontinued FRED level index plus growth through 2025Q4 — rather than a growth/diff series, so it lives in the same 100·log level space as the other log variables (verified: IP_CA transformation:'log' in CA/spec.json).

    5·EM. How the coupled blocs are estimated — the GVAR-EM loop (offline)¶

    The coupling above operates on blocs that are already estimated. They come out of the offline GVAR-EM iterative estimator pipeline/export_bvar_gvar24_loop.m (the M-step/E-step loop; the runbook to run it is §12a). The fixed point is in the completed star variables, not in the scenario deltas — distinct from the solve-time x=Mx+d of §5. Algorithm (verified against the script):

    GVAR-EM (export_bvar_gvar24_loop.m), env knobs MAXIT (default 12), TOL (default 0.03):

  load every bloc's raw panel  DataRaw_<CC>_glob.json   (≤ vintage; 100·log / lin via transformData)
  mask COVID:  NaN the macro columns over 2020Q2..Q4 (financials + ROWFIN stay OBSERVED)   # COVID-as-missing
  initialise each bloc's completed panel D.<CC>.comp (seed interpolation)
  repeat up to MAXIT times:
    # --- E-step: rebuild the trade-weighted stars from the current completed partner panels ---
    for each COUPLED bloc c:
      ROWDEM/ROWINF/ROWFIN_c  =  Σ_j  w_{c,j} · partner_j's (deflated) gdp-growth / inflation / rate
                                  read from D.<j>.comp, renormalised over partners present each quarter
                                  (CN gdp deflated by gdp−price; CH/NO dropped from FIN; AR dropped from INF/FIN)
    # --- M-step: re-estimate every bloc in PARALLEL on its [domestic | 3 stars] panel ---
    parfor each bloc c:                                                  # parpool('local')
      pos = find(prior=='WN')                                           # §6a — WN cols (fiscal ratios), [] otherwise
      [Xcond, BETA, SU, Dforecast] = cgl_bvarV5([dom | stars], lags=3, pos, nDraws, nBurn, MCMCconst, H,
                                                'mcmc',1,'MNpsi',0,'sur',0,'noc',1, 'Ndraws',1000,'Ndrawsdiscard',500)
      #  cgl_bvarV5 = GLP/Minnesota csminwel mode (bvarGLPmf) + Durbin–Koopman simulation smoother:
      #  the DK smoother fills the masked COVID + missing-star cells given the OBSERVED columns (per-cell, not per-row)
      D.<c>.comp <- Xcond                                              # the DK-completed panel
    until  max | Δ(stars) | < TOL   or   MAXIT reached                  # MC noise floor ~0.13 ⇒ a few iters suffice

  # MAXIT=1  ⇒  no star re-convergence = single-country / "separated" estimation (the (iv-b)/single-BVAR ablation)
  export per bloc:  spec.json, companion_mean.json, companion_draws.json, baseline(_levels).json, historical(_levels).json

    The "E-step" is the Durbin–Koopman completion of the foreign/star (and COVID-masked) cells; the "M-step" is the Bayesian re-estimation of each bloc; iterating to a fixed point in the completed stars is the EM analogy. Set US_ENDOGENOUS=1 to fold the US into the parfor set (it then builds its own ROW*_US stars from its partners and feeds the global commons — §5/§5a). This is the estimator the empirical horse-race (§14) drives at every out-of-sample origin.

    5a. Endogenous-US augmented closed form + the affine-kernel perf path¶

    When US_ENDOGENOUS, the US both (i) receives partner spillovers through its own ROW*_US stars and (ii) drives every RoW bloc's globals (which the RoW blocs pin to own-base + the US-global DELTA). With the US frozen that delta was a constant in d; now it is part of the fixed point, so the compact state is AUGMENTED (buildLinearSystemEndo):

    x = [ star pin-delta s_{c,v,t}  for every COUPLED bloc c (US included), v∈{DEM,INF,FIN}, t ]
    ++ [ US global-column delta gΔ_{p,t}  for each US global column p, t ]

    with three coupling families assembled into M from the constant per-period smoother kernels: 1. star ← star — receiver c's star v at t ← partner j's linked-col response from j's star pin (vp,tp); D[v] = identity (rate) or 4·Δ (gdp/price). The US is BOTH partner and receiver. 2. gΔ ← US star — the US global col p at t = the US free-var response to its OWN star pins (a definition row). 3. star ← gΔ — a RoW partner j's domestic vars respond to its global pin (Kg[j][p]), feeding the stars of the receivers that have j as a partner (weight w_rj; US excluded — it has no global pins).

    d then carries only the exogenous own-pin contributions (each bloc's own scenario pins moving its linked series at zero partner/global delta) plus the US's own-pin global delta; the US-as-partner term is in M (1), the US-global term in M (2)+(3). solveGVARClosedFormEndo solves x = (I−M)⁻¹d — using the hydrated artifact inverse when present, otherwise an inverse-free Neumann fixed-point x ← d + Mx (the in-JS O(dim³) inverse at dim≈2060 is ~46 min, NEVER built live; the Neumann path converges at ρ(M)≈0.924 in a few-hundred cheap matvecs) — then reconstructs each bloc's full level with one final conditionalForecast.

    Affine-kernel perf (blocKernels + affineBlocKernels). A bloc's response kernels — K[v][tp] (star-pin IRFs) and Kg[p][tp] (global-pin IRFs) — depend ONLY on that bloc's params + its OWN pin STRUCTURE (which cells are pinned), not on partner data, the global topology, or pin VALUES. So: - All (3 stars + #globals)·H per-period kernels for a bloc come from one filter+smoother pass on the probe baseline plus cheap per-cell matvec recursions (affineBlocKernels in kalman.ts) instead of one full conditionalForecast per (var, tp) — numerically identical (<1e-9), ~10–20× faster on the n=41/lags=3/H=20 US bloc. - The kernels are cached (_blocKernelCache, keyed by modelSig + bloc + own-pin-cells). In the dominant flow (the user pins ONLY the US) all 32 RoW blocs keep an empty own-pin structure ⇒ their kernels are reused ⇒ only the US re-probes. prewarmGVARKernels/prewarmGVARKernelsChunk warm the empty-structure kernels off the critical path (in the worker right after load); prewarmGVARKernels also caches the empty-structure assembled M so the first US-pin solve reuses it.

    Two-tier UX (solveGVARDirect + the "Solve full GVAR" button). The full fixed point pays smoother reconstructions and an (I−M) solve. To stay reactive on a drag, solveGVARDirect returns a first-order preview: x_direct = d (the US scenario's one-hop spillover, no RoW↔RoW or RoW→US feedback) mapped to each bloc's level delta straight from the prewarmed kernels (pure matvecs, no Kalman pass, sub-50 ms) — it UNDER-states the response (omits multiplier feedback). runGvarSolve.ts routes mode:'direct' → solveGVARDirect (live preview on pin edits) and mode:'full' → solveGVARClosedForm (the complete fixed point, on the explicit button). The slow path is a structure-change direct solve (placing/moving a US pin re-probes the US bloc, several seconds): the hook (useGvarForecast.ts) defers it to drag-end so it fires once per gesture, shows a "computing…" badge, and the worker strips the ~95 MB coupling artifact (withoutCoupling) from direct-mode posts (solveGVARDirect never uses it). Value-only re-drags (same structure ⇒ kernel-cache hits) and full solves dispatch live. ChartGrid.tsx shows the mode badge ('DIRECT preview' amber / 'full solve' blue) and the ▶ Solve full GVAR button whenever the active result is a direct preview.

    6. Data-space conventions (CRITICAL — most bugs live here)¶

    • A variable's transformation is 'log' or 'lin' (in spec.json).
    • log variables are stored as 100·log(level). The chart displays q/q annualized growth = 4·Δ(100·log level). Historical growth in historical.json is already this 4·Δ form.
    • lin variables (rates, ratios, unemployment) are passed through as-is (level = display).
    • Conditioning is WYSIWYG: a directly-pinned quarter shows exactly baseline + your deviation.
    • Posterior draws are flat arrays with shape beta_shape=[k, nv, nDraws] (k=n·lags+1, nv=n) and sigma_shape=[n, n, nDraws]. Index: beta[draw*k*nv + j*k + i], sigma[draw*n*n + j*n + i]. These are Float64Array after loadCompanionDraws (converted once for clone/index speed).
    • GVAR RoW pins exchange paths in 100·log level space (avoids a differencing bug): for a log RoW var the pin delta is 4·Δ(level-gap), for a lin rate it is the level-gap.

    When in doubt, the canary tests (§11) encode the exact conventions — run them.

    6a. The per-variable prior convention (RW vs WN) — prior in spec.json¶

    Every variable in spec.json carries a prior field — 'RW' (random walk, the default) or 'WN' (white noise / stationary). It selects the prior mean on that variable's OWN first lag in the GLP/Minnesota prior:

    prior Own-first-lag prior mean Behaviour Used for
    'RW' (default) 1 the variable is a priori a random walk (persistent, drifts at the long end) every macro/financial level, all RoW stars, EFFI
    'WN' (stationary) 0 the variable is a priori white noise → its forecast mean-reverts to the in-sample mean the fiscal GDP-ratios FREV_GDP, FPEXP_GDP, SFA

    The mechanism (verified — BVAR/MFBVAR/GLPreplicationWeb/bvarGLPmf.m). The Minnesota prior-mean matrix b is built with a diagonal of diagb over the own-first-lag block; WN columns get that diagonal entry zeroed:

    % bvarGLPmf.m, lines 192–195
b=zeros(k,n);
diagb=ones(n,1);
diagb(pos)=0;   % Set to zero the prior mean on the first own lag for variables selected in the vector pos
b(2:n+1,:)=diag(diagb);

    pos is the vector of WN column indices. It is the 3rd positional argument of the single-bloc estimator cgl_bvarV5.m (named stationary there; pos=stationary at line 19) and is threaded straight into the bvarGLPmf call ('pos', pos) and every downstream logMLVAR_* / BVAR call.

    Where it flows (spec.json → estimator pos → forecast). 1. The estimators read the per-variable prior from the spec and build the pos index set: - pipeline/export_bvar_gvar24_loop.m: statCols = find(Spec.Prior(keep)=="WN")' (on the kept domestic block — stars are appended AFTER and are always RW), passed as cgl_bvarV5(x, lags, Dc.statCols, …). - pipeline/export_bvar_euro_fsap.m: statCols = find(Prior=="WN")', passed as cgl_bvarV5(x, lags, statCols, …). - Both use a robust priorsfrom(vars) helper that tolerates MATLAB's jsondecode dropping a prior field that is present on only some variables (cell-array vs struct-array shapes); a missing/empty prior defaults to 'RW' (the historical behaviour) — so the WN prior is purely additive and a no-op (byte-identical) for the 44 non-fiscal blocs (their pos is empty). 2. The estimator bakes the prior into the exported coefficients — the WN mean-reversion lives in the posterior betahat/sigmahat (and the draws) written to companion_mean.json / companion_draws.json. 3. The TypeScript runtime never reads .prior (grep: no .prior access anywhere in app/src). The forecast uses the estimated coefficients only — the prior has already done its work at estimation time. prior in the deployed spec.json is therefore estimation provenance / documentation, not a runtime switch. (Consequence: changing a prior requires a re-estimation + re-export, not just a spec edit.)

    Result (verified on the US bloc). Under the old all-RW prior the fiscal ratios drifted at the long end (US revenue stuck near ~19%); with the WN prior FREV_GDP mean-reverts toward its in-sample mean — the deployed US/baseline_levels.json shows FREV_GDP ≈ 18.06 at h=0 → ≈ 17.4 by h=8 → ≈ 17.9 at the long horizon, instead of drifting up. EFFI (the effective interest rate) deliberately stays RW (it is persistent). Committed 6f0161a / 22250be / e38673a. The prior=='WN' assignment is consistent across all 6 fiscal blocs (verified: FREV_GDP/FPEXP_GDP/SFA = WN, EFFI = RW in US/EA/JP/UK/CA/CN spec.json).

    7. The fiscal block (fiscalAccounting.ts) — 6 countries (US/EA/JP/UK/CA/CN)¶

    Stock-flow-consistent, CBO/IMF-style. The BVAR estimates 4 fiscal drivers — FREV_GDP (revenue/GDP·100), FPEXP_GDP (primary-exp/GDP·100), EFFI (the effective interest rate = net interest / lagged gross debt ·100, annualised) and SFA (the stock-flow adjustment, % of GDP); reporting variables (debt service, debt/GDP, deficit, GFN, snowball) are derived. The debtService (DSVC) parametrisation was reverted — the endogenous interest variable is back to EFFI, and SFA is now the new estimated BVAR variable.

    Stationary (WN) prior on the GDP-ratios (see §6a). The three GDP-ratio drivers FREV_GDP, FPEXP_GDP, SFA are estimated with the prior:'WN' stationary mean (own-first-lag prior mean = 0) so they mean-revert to their in-sample means instead of drifting at the long end (the old all-RW prior left US revenue stuck near ~19%; it now reverts toward ~17.4%). EFFI stays prior:'RW' — the effective rate is genuinely persistent. This is purely an estimation-prior choice baked into the exported coefficients; fiscalAccounting.ts is unchanged by it.

    • pb = revenue − primaryExp (primary balance, + = surplus).
    • SFA is a NEW estimated BVAR variable = the EXACT residual of the app's debt recursion — the deficit–debt discrepancy (observed dDebt − (deficit + interest − growth)), driven by the federal-flows-vs-gross-debt perimeter seam / intragovernmental holdings. It is computed so the recursion closes to machine precision; COVID quarters are set NaN (the COVID-as-missing Gibbs imputes them). The debt/GDP projection now carries the FORECAST SFA forward (its own dynamics + bands) instead of defaulting to 0.
    • Recursion (per quarter): d_t = d_{t-1}·(1 + EFFI/400)/(1 + gNomQ/100) − pb/4 + sfa/4, with pb = FREV − FPEXP and gNomQ = 100·log(realGDP)-diff + 100·log(deflator)-diff (q/q nominal, read off the level path). Debt service (interest, % GDP) = EFFI·d_{t-1}/100. Debt-to-GDP is derived by rolling this budget identity forward from d0 (debt is not a BVAR variable).
    • fiscal_meta.json carries { d0, sfaDefault, idx:{ gdp, deflator, revenue, primaryExp, effRate, sfa } }. debtService is NO LONGER in the deployed idx — computeFiscalBlock therefore takes the effRate branch (interest = effRate·d_{t-1}/100) and reads the per-quarter SFA from idx.sfa (the estimated SFA column). Deployed indices (read from the live files, NEVER hardcode — these are positional and DRIFT on any variable-order change, see the gotcha below): US {0,7,33,34,35,36}, EA {0,6,17,18,19,20}, JP {0,7,16,17,18,19}, UK {0,8,18,19,20,21}, CA {0,8,18,19,20,21}, CN {0,1,13,14,15,16} for {gdp,deflator,revenue,primaryExp,effRate,sfa}. SFA means (sfaDefault, % GDP): US 1.75, EA 0.33, JP 2.23, UK 3.15, CA 3.19, CN 4.02. Variable counts n: US 41, EA 24, JP 23, UK 25, CA 25, CN 20.

    🔴 STALE-POSITIONAL-INDEX gotcha (flagged-for-process, not a code bug). The porter (pipeline/port_gvar_exports.py) does not regenerate fiscal_meta.json, and these indices are positional. The 33-bloc GVAR-EM refit inserted variables ahead of the fiscal block in every spec except US, so the old EA {0,6,14,15,16,17} / JP {0,7,14,15,16,17} / UK,CA {0,8,16,17,18,19} / CN {0,1,11,12,13,14} silently pointed at the WRONG columns — EA's effRate landed on the equity index (col ~537), so d·(1+effRate/400) compounded at ~134%/q and debt/GDP exploded. After any re-port you must run node pipeline/fix_fiscal_meta_indices.mjs (re-resolves the six indices by variable id from the deployed spec.json, preserves d0/sfaDefault, idempotent). The fiscal-meta-indices.test.ts canary asserts every deployed bloc's derived debt path stays bounded — it is what catches a stale port. - The fiscalAccounting.ts debtService/dsvc plumbing (runFiscalRecursion, computeFiscalBlock, requiredPrimaryBalance) is dormant but retained: it activates only when a model ships an idx.debtService, which the current deployment does not. With idx.sfa set and idx.debtService unset, computeFiscalBlock reads levelPaths[t][idx.sfa] for the per-quarter SFA and uses the effective-rate interest form. - There is a checkable Excel walkthrough docs/fiscal_accounting_walkthrough_US_UK.xlsx (and the user's source docs/Fiscal_Accounting_US_UK_NSA.xlsx).

    See fiscal-block-methodology.md for the full derivation.

    8. Uncertainty bands (two modes — know the difference)¶

    The band-computation effect lives in ChartGrid.tsx (deps [uncertaintyComputing], fires on Compute). Quick=50 draws, Full=200.

    • DEFAULT (model-specific density, fast, on by default). For every posterior draw, take the unconditional forecast (bvarForecastDraw, full forecast width) and parallel-shift it by the same mean scenario delta sourced from gvarBandRef.current.effective.forecasts (the GVAR-propagated path for any active GVAR bloc, else the single-country conditional). The band keeps full width (does not collapse when one point is pinned) and centres on the displayed line. The derived-fiscal charts (the 6 fiscal countries) get a matching parallel-shift band (per-draw unconditional computeFiscalBlock, shifted by mean scenario−baseline PB/debt via fiscalMeanRef).
    • EXPERIMENTAL (per-draw iterative GVAR density, opt-in). Gated behind the store flag experimentalFeatures (lower-left ExperimentalToggle). Re-solves the conditional cross-country system per draw (runGvarBandSolve). This is the conditional cloud — it pinches at pins (and collapses when lightly conditioned) and is slower. The request builder iterates the active GVAR bloc set (not a hardcoded subset), so every bloc's own-pins are carried; solveViewedScenLvl skips the cross-country sweep for a US-viewed bloc only when the US is frozen (!US_ENDOGENOUS) — under the endogenous-US topology the US depends on the fixed point like any bloc, so it goes through the full solve.

    Why two modes: the per-draw conditional re-solve over-determines the conditional mean when only a point or two is pinned, so it compresses too much; the default parallel-shift preserves the model's own forecast width while still centring on the scenario. The default is the right answer for almost everyone; the experimental mode is there for users who want the pinching behaviour.

    8a. Long-run anchor (PLR) — longRunTilt.ts¶

    A user can pin a long-run target for a variable or a spread h'·y. Instead of re-estimating, longRunTilt.ts entropically tilts the existing posterior draws (KL-min / exponential-tilting importance reweighting) so the weighted-mean long-run moment hits the target. The tilt now moves the UNCONDITIONAL central path (the weighted mean from reduceCloud), not just the fan — so both the baseline and the scenario fans are pulled toward the target. The long-run moment defaults to the long-horizon forecast E[y_{T+H}] at the ≈10-year (40-quarter) horizon (reuses the bvarForecastDraw companion iteration, no innovation), not the steady state — the level VARs are near-unit-root so (I − ΣB_l)⁻¹c diverges. This is the user-facing, after-estimation analogue of GLP (2019) "Prior for the Long Run".

    UI/Compute (ControlPanel.tsx): the Compute button is enabled whenever an enabled anchor exists, and applying the tilt auto-turns-on Quick uncertainty if it is off; the target input sits below the variable dropdown; the panel includes a "what it tilts toward" explanation and an ESS badge (low ESS ⇒ target far from the data — relax it or use one anchor). The panel is no longer gated to non-GVAR models: it now renders for GVAR blocs too, with a "GVAR bloc — marginal anchor" disclaimer — for a GVAR bloc it reweights only that bloc's own draws, anchoring the bloc's marginal long-run forecast, not the joint cross-country solution (partners + US-determined globals still pull the displayed path), so the realised long-run value may differ from the target. A fully joint re-anchoring is a planned extension.

    8b. Drag-perf UX — the heavy GVAR solve is OFF the live drag path (DEPLOYED)¶

    The dominant interaction is dragging a pin on a GVAR bloc. The full cross-country fixed point is multi-second (§5a), so the live drag path must do zero heavy compute. Four pieces make the drag feel instant while the correct answer reconciles on release:

    • Zero solves during a drag; one cheap solve on pointer-up. The drag handler signals beginGvarDrag() / endGvarDrag() through the gvarDragState.ts bus. While isGvarDragging(), useGvarForecast withholds every GVAR mean solve (value-only AND structure-change — each is ~4.5 s warm / ~13 s cold on the live 50-bloc panel, verified at useGvarForecast.ts:279) and just marks one as owed. On onGvarDragEnd (pointer-up) it dispatches exactly one direct solve, deferred ~100 ms (coalesces back-to-back gestures + waits for the final store write). The full fixed point + uncertainty bands run only on the explicit Compute / "Solve full GVAR" token (solveFullToken). The mousemove path is itself rAF-coalesced to one store write per frame (ChartGrid.tsx:2388), so a drag is ~one cheap condForecast per frame, not a solve.
    • The dragged bloc's line stays live via condForecast fallback. While dragging, ChartGrid's effective memo (subscribed to the drag flag via subscribeGvarDragging) swaps the viewed bloc's line from the frozen GVAR path to the single-country DK-smoother condForecast (~1–5 ms), then flips back to the propagated GVAR path on release. (gvar-drag-line-band.test.ts.)
    • Optimistic conditioned-point dot (conditionedDotY.ts). The pinned marker, the line at the pin, and the collapsed band there snap to the just-clicked value instantly — they cannot wait for the ~700 ms solve. At active+conditioned quarters ChartGrid overrides the scenario y with conditionedDotDisplayY(...) (ChartGrid.tsx:1656), reconstructed from the conditioning PATH the click wrote (the exact inverse of the display→stored map conditioningDeviationFromDisplayY used by both the drag and the right-click). So the dot round-trips to the cursor for every variable type (level-display oil/FX/equity/house via logToIndexLevel; growth-displayed log / lin direct). The smoothed line + bands reconcile after the solve. (conditioned-dot-optimistic.test.ts.)
    • Right-click plants the bullet at the cursor. A right-click on a forecast chart enables and pins the quarter at the cursor's display y, using the same conditioningDeviationFromDisplayY mapping (ChartGrid.tsx:2484, :1300) — so a planted bullet and a dragged one are identical. (right-click-plant-at-cursor.test.ts.)

    A "computing…" badge (driven by setGvarComputing / subscribeGvarComputing on the same bus, so only the badge re-renders) shows while the release solve is in flight; ChartGrid shows the mode badge ('DIRECT preview' amber / 'full solve' blue) and a ▶ Solve full GVAR button whenever the active result is a direct preview (§5a).

    9. State (store/useStore.ts, Zustand)¶

    • scenarios[] + activeScenarioId — each scenario holds its own conditioning paths, conditioned quarters, counterfactual settings, pbCond/debtCond fiscal targets, cachedBands.

    10. The per-country data contract (public/data/bvar/<CC>/)¶

    Each country directory provides (loaded by loader.ts / gvarLoader.ts):

    File Contents

The US spec.json now also carries block:'row' stars (ROWDEM_US/ROWINF_US/ROWFIN_US, cols 38–40) and the 4 globals — this is what makes the US endogenous (§5/§5a). Standalone (non-GVAR) economies carry only 'domestic' (+ 'fiscal' for a fiscal bloc).

    12a. Re-estimation + coupling-artifact runbook (the data refresh path)¶

    The models are estimated offline and ported into app/public/data/bvar. Verify each command against the actual scripts before running — do not invent flags.

The script re-resolves all six indices by variable id from the deployed spec.json (idempotent); fiscal-meta-indices.test.ts then catches any that are still stale.
    4. Regenerate the coupling artifact (MANDATORY after any data change — modelSig keys it to the data): from app/, npx vite-node scripts/precompute_gvar_coupling.mjs. It builds M in JS (buildGVARCouplingMEndo when US_ENDOGENOUS, else buildGVARCouplingM) for the empty pin structure, computes the inverse via MATLAB BLAS (invert_coupling.m, shelled matlab -batch — the in-JS O(dim³) inverse at dim≈2060 is ~46 min), and writes app/public/data/bvar/gvar/coupling_default.json.gz (gzip, 6-sigfig: {modelSig, pinStructure:'', dim, rhoSpectral, M, invImM}). gvar-coupling-artifact.test.ts asserts the artifact matches a fresh build.
    5. Gate + deploy — green gate (above), then the deploy path (below). Push to the bvarscenario remote → GitHub Action copies app/out into starter-hugo-academic → Hugo rebuilds & serves the tool at the /bvar-scenario path on GitHub Pages.

    13. Conventions to respect (a checklist before you commit)¶

    1. Don't hardcode variable indices. Read n from companion_mean.json, fiscal indices from fiscal_meta.json, GVAR indices from IDX.
    2. Respect the data space. A stray ×4 or /100 between level and growth space will diverge forecasts silently — the canary tests catch it.
    3. The GVAR bloc set is data-driven (33 blocs: endogenous US + EA + 31 RoW), not a hardcoded list. Never hardcode a subset like ['US','EA','JP'] — read ROW_COUNTRIES / COUPLED_COUNTRIES / IDX (live, set by configureGVARTopology) or iterate the active candidates from gvarLoader.ts. Note the US is now in the fixed point (COUPLED_COUNTRIES, gated by US_ENDOGENOUS): US pins are NOT free — they change the US smoother and thus M, so they enter the coupling-cache key and the pin-structure signature. A leftover pilot subset loop in the band request builder once silently dropped some blocs' own-pins (band on baseline while the line followed the propagated path); that class of regression is invisible to tsc and only shows on the dropped blocs.
    4. gvarPins persists across setCountry — by design.
    5. Exact MATLAB replication when porting an estimator — no shortcuts; flag and ask if unsure.
    6. The green gate is mandatory before deploy — and it is two halves (npm test + npm run test:heavy), not one vitest run. See §11.

    13a. Known issues & limitations (flagged-for-fix, NOT yet fixed)¶

    These are flagged-but-unfixed at the current deployment — document them honestly; do not treat the affected output as verified. The module-reference MODULES.md § Known issues carries the file:line detail; the short forms:

    1. CN fiscal debt — inflation double-count (RESOLVED 2026-06). The CN debt roll-forward used to read Δ(GDP) + Δ(deflator) against an already-NOMINAL GDP_CN, double-counting inflation. The surgical CN-only re-estimate replaced GDP_CN with real GDP (spec.json var 0 = "CN Real GDP …", fiscal_meta.json idx.gdpIsNominal = false), so the growth term is now the correct real-GDP + deflator nominal growth, matching US/EA/JP/UK/CA. The fix is CN-only; the other 32 blocs are byte-identical.
    2. Entropic long-run tilt does not filter non-finite moments. solveEntropicTilt feeds the raw per-draw long-run moments into the Newton dual; a single non-finite moment from an unstable draw NaN-poisons the softmax weights and silently empties the anchor band. Fix: drop non-finite draws before the iteration (as the band-step weightedQuantile already does). (longRunTilt.ts.)
    3. No ρ(M) < 1 divergence guard on the live closed-form solve. The endogenous Neumann fixed point iterates to maxIt/tolerance with no stationarity check: a refit yielding ρ(M) ≥ 1 would return a confident-wrong forecast with no warning. ρ is shipped in the artifact (currently 0.924) but not read as a live guard. (gvarSolve.ts:1296.)
    4. Conditional band uses the FILTERED, not DK-smoothed, covariance. conditionalForecast returns the smoothed MEAN but the FILTERED variance, so the band is overstated for quarters that precede a future pin (the smoother would tighten them). Mean correct; band conservative. (kalman.ts:405.)
    5. The porter does not regenerate fiscal_meta.json (STALE-POSITIONAL-INDEX). Positional indices go stale on any variable-order change (the EA/UK debt-explosion cause). Mandatory mitigation: run node pipeline/fix_fiscal_meta_indices.mjs after every re-port; fiscal-meta-indices.test.ts guards it. (§7, port_gvar_exports.py.)

    14. Empirical evaluation — lives in a SEPARATE repo (em-bayesian-gvar-eval)¶

    The forecast-evaluation / publication work is not in this repository. A dedicated harness repo, /home/farkas/github/em-bayesian-gvar-eval, implements the paper's two empirical cores. It references this repo, never copies it — its drivers addpath into $BVAR_TOOL_ROOT (default "/home/farkas/github/BVAR Policy Calibration Tool") and re-use cgl_bvarV5.m, export_bvar_gvar24_loop.m, the per-bloc raw panels, the trade weights, and gvarSolve.ts directly. There are NO copies of the estimator or solver in the eval repo, and there is NO eval / horse-race / scoring code in this repo (verified: nothing under this repo's git ls-files matches the harness). The full, code-grounded design is em-bayesian-gvar-eval/docs/EMPIRICAL_PROTOCOL.md — read it before touching the evaluation.

    ⚠️ The DESIGN is final and a first full scoring pass has now been produced (em-bayesian-gvar-eval/results/tables.{txt,json}, results/tierA/us_rate_100bp.json). The headline read below is from those files (56 origins 2010Q1–2024Q4 with 2020 dropped, 9 models). Treat the numbers as a first run with the honest caveats stated — CRPS + MCS are the robust primaries; the parametric log-score column is numerically broken and the draw-based densities are over-confident.

    The two cores:

    1. Out-of-sample forecast horse-race (Part 1). Recursive expanding-window, origins 2010Q1–2024Q4 (drop the 2020 COVID origins for the headline), leak-free ROLLING as-of-t0 trade weights (rebuilt from IMTS bilateral trade ≤ t0 — using the shipped 2021–2023 weights at an early origin would inject future trade structure), targets {real GDP growth, inflation, short rate} for US/EA/JP/UK/CA/CN, horizons h ∈ {1,2,4,8}. COVID-as-missing is re-applied in-sample at every origin whose window spans 2020Q2–Q4.

    2. Endogenous-US spillover analysis (Part 2). The endogenous-vs-frozen conditional-scenario contrast that this repo's gvarSolve.ts already computes is Tier A (the headline, zero new code: solveGVARClosedFormEndo vs the frozen path). GIRF / FEVD / Diebold–Yilmaz connectedness is Tier B — new code in the eval repo (constructible from the exported betahat/Σ + the coupling M), not a call into the shipped solver. The shipped solver computes conditional-mean scenario deltas, NOT GIRFs/FEVD — do not relabel a multi-period rate pin a "shock" or a "GIRF."

    Models in the race (config ids — em-bayesian-gvar-eval/config/eval_config.yaml):

    id Model What it is
    emgvar_endo Proposed, endogenous US the full §5·EM loop, US_ENDOGENOUS=1
    emgvar_frozen Proposed, frozen anchor US estimated once as a closed anchor; cheaper, and the Part-2 contrast
    rw / rw_clim Random walk / climatology driftless no-change (headline reference) + sample-mean anchor
    arp AR(p) per series p by AIC over {1..4}; iterated + direct
    sbvar Single-country BVAR strip the 3 ROW* stars, identical cgl_bvarV5 (value of coupling)
    cgvar_a Textbook OLS GVAR Pesaran–Schuermann–Weiner / Dees-et-al VARX*, fixed weights, no COVID-as-missing (the literature competitor)
    cgvar_b Single-pass Bayesian GVAR MAXIT=1, GLP + COVID-as-missing (the clean EM ablation)
    dfm Dynamic factor model 3–5 factors, Bayesian VAR on factors (non-GVAR multivariate benchmark)

    Scoring: CRPS (primary density score) + parametric log-score + PIT/coverage; significance via Giacomini–White (headline, valid under nesting + the expanding scheme), Clark–West (nested point), Diebold–Mariano (non-nested), with Romano–Wolf + Model Confidence Set for multiplicity.

    Headline result (first full pass — 56 origins, 9 models; from results/tables.{txt,json}). AR(p) and the DFM have the lowest point CRPS, BUT the 90% Model Confidence Set tells the publishable story: emgvar_endo is IN the INFLATION MCS at every horizon (h=1,2,4,8) and IN the SHORT-RATE MCS at h=1,2,4 — statistically indistinguishable from the best on the nominal block — while it is OUT of the GDP-GROWTH MCS at h=1 and h=8. emgvar_endo ⪰ emgvar_frozen across the MCS (the endogenous-US design earns its keep — e.g. at RATE h=1 and GROWTH h=4 the endogenous bloc is in the set and the frozen one is removed). The EM-Bayesian treatment decisively beats the textbook OLS classical GVAR: cgvar_a is removed from most MCS sets (GROWTH h=1/4/8, INFL h=4/8, RATE h=8) and carries CRPS/RW up to ~2.5. Best inflation h=1 is emgvar_endo (relRMSE 0.799, CRPS/RW 0.814).

    Honest caveats (must state, verified in the tables). - Draw-based densities are OVER-CONFIDENT. PIT cov90 for emgvar_endo runs ~0.73–0.83 (vs the 0.90 target); the frozen variant is similar. The bands are too tight at the draw counts used. - The parametric log-score column is numerically broken on the under-covered clouds (the LPS(G) entries blow up to ~1e14+). CRPS and the MCS are the robust primaries; treat LPS(G) as unusable.

    Tier-A spillover (eval/tier_a_scenario.mjs, results/tierA/us_rate_100bp.json). A US +100 bp conditional projection shows non-trivial endogenous-only cross-country feedback that the frozen anchor structurally omits (the file's own meta.note: the endo−frozen gap is the RoW→US feedback the frozen anchor cannot represent). BUT the magnitudes are large and sign-flipping across horizons (e.g. the frozen-anchor US/UK growth paths swing to −4.4 / −3.1 at h=20), and it is a multi-period conditional projection, NOT a causal IRF (the file labels itself exactly that). The magnitudes need a validation pass before being headlined — do not over-claim a causal interpretation.
