    Build, Test & Deploy¶

    How the EM-Bayesian Global VAR platform goes from a working tree to the live public site under the /bvar-scenario path of the academic website. This page documents the green gate (the local checks that must pass before any deploy), the test taxonomy (which suites block a deploy versus which guard a feature), and the deploy chain (the GitHub Action that builds the static export and republishes it into the starter-hugo-academic site repo, which publishes to GitHub Pages).

    For the higher-level architecture and the data-refresh / re-estimation runbook, see ARCHITECTURE_FOR_AI.md and REESTIMATING_MODELS.md. This page does not repeat those.

    The green gate¶

    Everything runs from the app/ directory. The commands below are the full pre-deploy contract — types clean, tests green (fast suite + the heavy solver canaries), static export succeeds — before any push to the deploy remote.

    cd app
npx tsc --noEmit                       # 1. types: must report 0 errors
npm test                               # 2. fast suite (excludes the heavy solver canaries)
npm run test:heavy                     # 3. heavy solver/perf canaries (serialised, 1 fork)
BASE_PATH=/bvar-scenario npm run build # 4. static export with the deploy base path
    Step Command What it proves
    Types npx tsc --noEmit TypeScript compiles with no errors (no emit; type-check only).
    Fast tests npm test The fast compute suite in app/src/__tests__/compute/, minus the heavy solver canaries (see the split below), including the cheap canaries.
    Heavy tests npm run test:heavy The expensive full-panel solver / affine-kernel / band-solve canaries, run serialised (--poolOptions.forks.maxForks=1) so the multi-second US-endogenous solves do not contend for cores.
    Build npm run build (next build) The Next.js static export (output: 'export') completes and writes app/out/.

    Green gate = tsc 0 errors and npm test green and npm run test:heavy green and the build succeeds. Only then is a deploy allowed. npx vitest run (= npm run test:all) runs everything in one pass — use it for a full sweep, but note it runs the heavy suites with default parallelism (slower, can over-subscribe).

    The fast/heavy split is real, and it is in package.json — read it, don't assume. npm test is not vitest run: it is vitest run with a list of --exclude globs. The excluded (heavy) patterns are gvar-us-endogenous, gvar-direct-channel, gvar-band-solve, gvar-closed-form-*, gvar-band-closedform-*, affine-blockernels-parity, gvar-kernel-affine-independence. npm run test:heavy runs exactly those (matched by vitest run <name…>), one fork at a time. A change that only npm test covers leaves the dim-2060 solver paths unverified — both halves are mandatory before deploy. (npm run test:watch is vitest — the interactive watcher, never a CI gate.)

    A few notes that are easy to get wrong:

    • The build is a static export: next.config.mjs sets output: 'export', images.unoptimized: true, and routes the deploy prefix through basePath/assetPrefix/NEXT_PUBLIC_BASE_PATH, all driven by the BASE_PATH env var (see BASE_PATH below).
    • This is not stock Next.js — see app/AGENTS.md. Read the bundled docs under node_modules/next/dist/docs/ before changing build config.

    Test taxonomy¶

    All compute tests live in app/src/__tests__/compute/. They fall into three roles:

    • CANARY — guards a load-bearing invariant or a deployed artifact. A red canary means the shipped model data, the scaling convention, or the cross-country solver identity is broken; it must block the deploy, not be skipped.
    • FEATURE — proves a specific user-facing capability (a Taylor rule, optimal policy, the fiscal debt recursion, US-endogenous coupling, the entropic long-run tilt, GVAR globals/topology, …).
    • REGRESSION — pins a previously-fixed bug so it cannot silently return (scaling differencing, WYSIWYG conditioning, counterfactual recovery, …).

    Fast (npm test) vs heavy (npm run test:heavy) — which is which¶

    npm test excludes seven heavy patterns; npm run test:heavy runs exactly those, serialised to one fork. The three deploy-blocking canaries (scenario-scale-convention, gvar-convergence, gvar-coupling-artifact) are all in the fast suite. The heavy suite is the multi-second full-panel solver and the affine-kernel/coupling perf-parity gates:

    Heavy suite (npm run test:heavy) Why it is heavy
    --exclude glob (from package.json) currently-present files
    ----------- --------
    gvar-us-endogenous gvar-us-endogenous.test.ts
    gvar-direct-channel gvar-direct-channel.test.ts
    gvar-band-solve gvar-band-solve.test.ts
    gvar-closed-form-* gvar-closed-form-routing.test.ts
    gvar-band-closedform-* (no file at present)
    affine-blockernels-parity affine-blockernels-parity.test.ts
    gvar-kernel-affine-independence gvar-kernel-affine-independence.test.ts

    The fast/heavy split is keyed on glob patterns, not a fixed file list: npm test --excludes the seven patterns above and npm run test:heavy selects gvar-us-endogenous gvar-direct-channel gvar-band-solve gvar-closed-form gvar-band-closedform affine-blockernels gvar-kernel-affine — so a new gvar-closed-form-* or gvar-band-closedform-* file is auto-routed to the heavy half. Everything else in app/src/__tests__/compute/ runs under npm test. Both halves gate the deploy — a heavy suite that only test:heavy runs still blocks a release if red.

    Canary suites (block deploy)¶

    Test file Guards
    scenario-scale-convention.test.ts The conditional-display scaling convention: response paths for log vars are shown as 4·(Δlevel_t − Δlevel_{t−1}) (q/q-annualized rate deviation), not 4·levelGap_t (cumulative price-level gap). Pins the corrected convention against the MATLAB conforekf ground truth for the canonical Baa +100bp / 8q shock; a break would re-introduce the ~23× far-horizon overstatement.
    gvar-convergence.test.ts Gauss-Seidel ≡ closed-form: for the same US scenario the iterative solveGVAR (Gauss-Seidel sweeps, US frozen) and the linear solveGVARLinear (build M by unit-RoW probes, then (I−M)⁻¹ d / Smith doubling) must agree to ~machine precision per RoW country. This is the proof that the N-bloc linear-solver generalization is correct; also reports empirical and power-iteration ρ(M).
    gvar-coupling-artifact.test.ts The pre-shipped coupling artifact (public/data/bvar/gvar/coupling_default.json.gz) is FRESH: its modelSig matches the deployed models, the augmented empty-pin-structure dim is self-consistent with the live US-endogenous build, ρ ∈ (0,1), and an artifact-hydrated solve equals the live-built closed-form solve (≤1e-3, the 6-sig-fig rounding bound). Also asserts the route REFUSES the empty-structure artifact for a non-empty (e.g. US-pinned) request. Skips loudly if the .gz is missing rather than failing.

    These three are exactly the suites that, if red, must stop a release — they cover the two things a static deploy cannot self-correct: the shipped artifacts (coupling_default.json.gz, the per-bloc model JSON) and the display/solver conventions the UI renders on top of them.

    Regression suites¶

    Test file Bug it pins
    conditioning-wysiwyg.test.ts A directly-conditioned variable must display exactly the dragged path (WYSIWYG), including sparse/partial per-quarter conditioning — the scaling-differencing fix must apply only to RESPONSE vars.
    counterfactual-irf.test.ts A counterfactual recovers the IRF when the rate adjustment equals the IRF.
    counterfactual-recovers-scenario.test.ts A counterfactual with a vanishing policy adjustment must recover the scenario (policy effect measured against the right baseline).
    fiscal-backcast.test.ts The budget-identity recursion in fiscalAccounting.ts reproduces the observed US federal debt/GDP back-cast from observed drivers (Step-1 gating).
    matrix.test.ts Core linear algebra (Cholesky/solve/inverse, ~1e-15 singularity threshold) in matrix.ts.
    affine-blockernels-parity.test.ts The affineBlocKernels perf path reproduces the old full filter+smoother probe loop to <1e-9 (baseline mean Sc and every per-period kernel) and is >=3× faster, on the real US-endogenous bloc — the optimization must stay a numerical no-op. (heavy)
    gvar-kernel-affine-independence.test.ts The affine-independence claim underlying the per-bloc kernel cache-key change (RANK-1 gate). (heavy)
    fiscal-meta-indices.test.ts Pins the STALE-POSITIONAL-INDEX bug (§ below): re-resolves every fiscal bloc's fiscal_meta.json indices by variable id from the deployed spec.json and asserts each bloc's derived debt path stays bounded — the regression that caught EA's effRate landing on the equity index (col ~537) so debt/GDP exploded. Also pins the level-display drag conversion (oil → $100 must not over-shoot to ~$190).

    The perf-fix parity suites (affine-blockernels-parity, gvar-kernel-affine-independence) are regression gates in spirit — they prove an optimization is numerically a no-op — but because they protect the production GVAR solve path, treat a red result there with the same severity as a canary.

    The STALE-POSITIONAL-INDEX gotcha (porter must regen). fiscal_meta.json stores the fiscal-driver column indices positionally, but the porter (pipeline/port_gvar_exports.py) does not regenerate them. Any variable-order change (the 33-bloc GVAR-EM refit inserted vars ahead of the fiscal block in every spec except US) silently re-points the indices at the wrong columns — the cause of the recent EA/UK/CA/JP/CN debt explosion. The fix is to run node pipeline/fix_fiscal_meta_indices.mjs (re-resolves the six indices by variable id from the deployed spec.json, preserves d0/sfaDefault, idempotent) after porting and before the gate; fiscal-meta-indices.test.ts is the canary-grade guard that blocks a deploy with stale indices. See REESTIMATING_MODELS.md and the DATA_SCHEMA fiscal-meta section.

    The file list above is enumerated from app/src/__tests__/compute/. When you add a test, slot it into the right row here and confirm it actually runs under vitest run.

    Deploy chain¶

    The source repo is matyasfarkas/BVAR-Policy-Calibration-Tool (the origin remote). The deploy target is the bvarscenario remote (matyasfarkas/BVARScenarioTool), whose GitHub Action builds the export and republishes it into the matyasfarkas/starter-hugo-academic website repo. That site is a Hugo Academic site published to GitHub Pages (note the netlify.toml trigger). This workflow contains no Hugo step: it only copies app/out/* into the website repo, and the Hugo build downstream then rebuilds and serves it.

    push -> bvarscenario main
   │   (paths: app/** or netlify.toml)
   ▼
.github/workflows/deploy-to-website.yml   (runner: ubuntu-latest, Node 22)
   │  npm ci
   │  BASE_PATH=/bvar-scenario npm run build      → app/out/
   │  checkout matyasfarkas/starter-hugo-academic
   │  rm -rf website/static/bvar-scenario/*
   │  cp -r app/out/* website/static/bvar-scenario/
   │  commit + push to the website repo
   ▼
Hugo rebuilds the academic (Hugo Academic) site from the website repo
   ▼
served at the /bvar-scenario path of the academic site (GitHub Pages)

    Workflow facts, verified against .github/workflows/deploy-to-website.yml:

    • Triggers: push to main with changed paths under app/** or netlify.toml, plus manual workflow_dispatch.
    • Runner / toolchain: ubuntu-latest, actions/setup-node@v4 with node-version: 22 and npm cache keyed on app/package-lock.json.
    • Build step runs in working-directory: app: npm ci then npm run build, with BASE_PATH: /bvar-scenario in the environment.
    • Publish step checks out matyasfarkas/starter-hugo-academic (via secrets.WEBSITE_DEPLOY_TOKEN), clears website/static/bvar-scenario/, copies app/out/* in, and commits/pushes (a no-op git diff --cached --quiet short-circuits when nothing changed).

    The Hugo build that assembles the final academic site is not in this workflow file — it runs as part of the website repo's downstream build that publishes to GitHub Pages, and any Hugo version pin lives in the starter-hugo-academic repo, not here (see below).

    BASE_PATH¶

    The site is served from the /bvar-scenario sub-path of the academic site (GitHub Pages), so the static export must be built with that prefix. next.config.mjs:

    const nextConfig = {
  output: 'export',
  basePath: process.env.BASE_PATH || '',
  assetPrefix: process.env.BASE_PATH || '',
  images: { unoptimized: true },
  env: { NEXT_PUBLIC_BASE_PATH: process.env.BASE_PATH || '' },
};
    Build BASE_PATH Asset/link prefix
    Local dev / preview unset → '' root-relative (/…)
    Deploy (CI) /bvar-scenario /bvar-scenario/…

    The CI step always sets BASE_PATH=/bvar-scenario. Reproduce the exact deploy artifact locally with BASE_PATH=/bvar-scenario npm run build. Client code that needs to fetch a shipped data file under the prefix reads process.env.NEXT_PUBLIC_BASE_PATH.

    Hugo version pin (0.122.0)¶

    The website repo's build is pinned to Hugo 0.122.0. The pin is enforced in the website pipeline (.github/workflows/updater-wip.yml). Rationale: Hugo 0.123+ removed the getCSV / getCSS data helpers that the wowchemy (academic) theme depends on; later versions break the site build. Do not bump the pin without first migrating the theme off those removed helpers. (Documented in README.md and ARCHITECTURE_FOR_AI.md.)

    What to commit (and what not to)¶

    • Commit only deploy-relevant files (app/**, docs, config).
    • Leave pipeline/ scratch — diagnostics, staging dirs, raw CSVs, _verify/ — untracked.
    • Never commit app/.env.local or any FRED_API_KEY.

    Quick reference¶

    # Full local green gate (run before every deploy):
cd app
npx tsc --noEmit
npm test                # fast suite (excludes the heavy solver canaries)
npm run test:heavy      # heavy solver / perf-parity canaries (1 fork)
BASE_PATH=/bvar-scenario npm run build

# Or sweep everything in one pass (slower, default parallelism):
npm run test:all        # = vitest run

# Run a single suite while iterating:
npx vitest run src/__tests__/compute/gvar-coupling-artifact.test.ts

# Regenerate the shipped coupling artifact (keeps the coupling canary green
# after a model refresh) — see REESTIMATING_MODELS.md:
npx vite-node scripts/precompute_gvar_coupling.mjs

    If the gvar-coupling-artifact canary goes red after a model refresh, the artifact is stale: regenerate coupling_default.json.gz with the precompute_gvar_coupling.mjs script (full data-refresh runbook in REESTIMATING_MODELS.md).
